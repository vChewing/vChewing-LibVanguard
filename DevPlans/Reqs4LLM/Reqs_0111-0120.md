## 手術範圍

本次手術同時牽涉到對 `vChewing-macOS`、`vChewing-OSX-Legacy` 兩個倉庫。

## 總體手術目的

截至 v4.5.4 版為止，唯音輸入法已經採用 LRU Map 的形式、讓單個 client 盡量重複利用 InputSession 副本。然而，在某些電腦上仍舊會有使用者在藉由 CpLk 切換中英文輸入法時遇到遲鈍現象。但其中有使用者給出了下述關鍵資訊：

> 奇摩輸入法沒有這個問題，且奇摩輸入法原始碼倉庫指出奇摩輸入法的建置過程並未啟用 ARC。

於是就有了至少下述要求：

- 所有由唯音輸入法的 Swift 部分直接曝露出去的 ObjC 元件，都應該被系統認為是需要 MRC 管理的東西（而非 ARC）、不能持有任何 Properties、不能親自處理任何 async task（GCD 等）。。這包括 AppDelegate 與「IMKInputController Subclass」。
- IMKTextInput Client 作為 NSConnection Remote Object 也不能被 Swift ARC 直接從 IMKInputController 讀取的方式持有。
- 整個 MainAssembly (Swift) 與 InputMethodKit 之間的 Facade 必須藉由記憶體位址翻譯的方式進行、且杜絕任何 ARC API 的參與。

這些要求構成唯音輸入法 4.5.5 版的總趣旨。

該 Phase 書寫完畢時，4.5.5 版已經完成開發且發版。但仍需書面報告備份，所以就需要在此留下手術報告、記載了 4.5.4 至 4.5.5 版為止對 InputSession、SessionCtl (IMKInputController Subclass)、AppDelegate、Xcode 專案配置、Swift Package 配置內容的全部變更。

## Phase 111 實作結果（2026-07-20, macOS Repo & Legacy Repo）

### 實作策略

本 Phase 的總體目標：徹底消除唯音輸入法與 IMK (InputMethodKit) 之間的所有 ObjC ARC 參與。核心手段有三：

1. **全專案停用 ObjC ARC 編譯標誌**（`-fno-objc-arc`），使所有 ObjC-visible NSObject 子類別在編譯層面被系統視為 MRC 管理。
2. **以原始記憶體位址（`UInt`）取代所有的 ARC-managed reference**，徹底繞過 retain/release。
3. **讓關鍵物件（AppDelegate、SessionCtl）不擁有任何 stored property**，使其 alloc / dealloc 全程無 ARC 參與。

由於 `vChewing-OSX-Legacy` 使用 Xcode 15 工具鏈（無 SIL ownership verifier 問題）且 `SessionCtl` 沒有 `@MainActor`，部分實作細節因應兩倉差異有所取捨（詳見下文各小節）。

#### 一、Session 快取改用 NSMutex 保護

**受影響之原始碼**: `InputSession.swift`（雙倉）

將 `sessionsByClient` 字典與其 `sessionsLock`（`NSLock`）合併為單一 `NSMutex<[Int: InputSession]>()`，使資料與鎖結合成一個不可分割的單位，杜絕鎖外存取的可能性。呼叫端改用 `withLock { sessionClientMap in ... }` 閉包式存取，閉包參數命名為 `sessionClientMap` 以避免遮蔽靜態屬性。

```swift
// Before
private static var sessionsByClient = [Int: InputSession]()
private static let sessionsLock = NSLock()

// After
private static var sessionsByClient = NSMutex([Int: InputSession]())
```

`NSMutex` 為專案內既有的 `NSLock` 包裝型別（定義於 `SwiftFoundationImpl.swift`），提供 `withLock(_:)` / `withLockRead(_:)` 與 `.value` computed property，具備 Rust `Mutex<T>` 式語義。雙倉同步。

#### 二、全專案停用 ObjC ARC

**受影響之檔案**: `vChewing_IMKUtils/Package.swift`、`vChewing_MainAssembly4Darwin/Package.swift`、`vChewing.xcodeproj/project.pbxproj`、root `Package.swift`（macOS repo）；Legacy 的 Xcode project（雙倉）

在三個層級施加 `-fno-objc-arc`：

| 層級 | 檔案 | 目的 |
|------|------|------|
| ObjC bridging target | `IMKUtils/Package.swift` → `IMKSwift` target | 無條件啟用（純 ObjC target 不受 Swift compiler bug 影響） |
| Swift library target | `MainAssembly4Darwin/Package.swift` | 條件式啟用：僅 `#if compiler(>=6.4)` |
| Root executable target | root `Package.swift` → `vChewing` target | 條件式啟用：僅 `#if compiler(>=6.4)` |

**條件式啟用的原因**：Xcode 26.x (Swift 6.3) 的 SIL ownership verifier 會因 `-Xcc -fno-objc-arc` 影響 Clang importer 對 `NSObject.init()` 的 ownership 判定而觸發 assertion failure。Swift 6.4 (Xcode 27) 已修正。由於 legacy repo 使用 Xcode 15 且無此 compiler bug，該處保持無條件啟用。

`vChewing.xcodeproj` 改 `CLANG_ENABLE_OBJC_ARC = NO`（Debug + Release）。

#### 三、以 UInt 記憶體位址取代 ClientObj 的閉包傳遞

**受影響之原始碼**: `InputSession.swift`、`SessionCtl.swift`（雙倉）

將 `InputSession` 持有的 client 閉包從 `() -> ClientObj?` 改為 `() -> UInt?`（儲存 client 物件記憶體位址），避免閉包持有 `ClientObj`。新增 `theClient()` 解析方法：

```swift
// Before
public var theClient: () -> ClientObj?
init(..., client inputClient: @escaping (() -> ClientObj?))
reassign(to:, clientProvider: @escaping () -> ClientObj?)

// After
public var theClientAddr: () -> UInt?
public func theClient() -> ClientObj? { ... }  // takeUnretainedValue()
init(..., clientAddr inputClientAddr: @escaping (() -> UInt?))
reassign(to:, clientAddrProvider: @escaping () -> UInt?)
```

`SessionCtl.getClientAddrProvider()` 同步改為回傳 `UInt?`（用 `Unmanaged.passUnretained(clientObj).toOpaque()` 編碼）。`callCoreAtLeastOnce` 內建立新 Session 的 client closure 也改為 UInt 編碼。

#### 四、以 static map 取代 SessionCtl._core weak var

**受影響之原始碼**: `InputSession.swift`、`SessionCtl.swift`（雙倉）

`SessionCtl` 原本持有 `@MainActor private weak var _core: InputSession?` 以查詢對應的 InputSession。該 weak var 被完全移除，改以 `InputSession` 的靜態字典 `sessionAddrByControllerAddr: NSMutex<[Int: Int]>()` 記錄 controller 位址 → session 位址的對照。

新增三支 `nonisolated static` 輔助方法：

| 方法 | 用途 |
|------|------|
| `sessionAddr(for:)` | 由 SessionCtl 位址查詢 InputSession（`takeUnretainedValue()` 解析） |
| `registerSessionAddr(_:for:)` | 登記 controller → session 對照（init、reassign 時呼叫） |
| `unregisterSessionAddr(for:)` | 移除對照（SessionCtl.deinit 呼叫，連帶將 session 的 `inputControllerAssignedAddr` 設為 nil） |

`SessionCtl.core` 改為 computed property：getter 走 `InputSession.sessionAddr(for: self)`，setter 走 `InputSession.registerSessionAddr(session, for: self)`。`SessionCtl.deinit` 呼叫 `InputSession.unregisterSessionAddr(for: self)`。

自此 `SessionCtl` 的 alloc / dealloc 完全無 ARC 參與——沒有任何 stored property，只有 computed `core` 與一個 integer cleanup 呼叫。

#### 五、以 UInt 取代 InputSession 的 weak var inputControllerAssigned

**受影響之原始碼**: `InputSession.swift`（雙倉）

`InputSession` 原本持有 `public weak var inputControllerAssigned: SessionCtl?`。該 weak var 被 `public var inputControllerAssignedAddr: UInt?`（兩倉均標記 `nonisolated(unsafe)`）取代。`inputController` computed property 改為 resolve 模式：

```swift
// Before
public weak var inputControllerAssigned: SessionCtl?
public var inputController: SessionCtl? { inputControllerAssigned }

// After
public var inputControllerAssignedAddr: UInt?
public var inputController: SessionCtl? {
    guard let addr = inputControllerAssignedAddr,
          let opaque = UnsafeRawPointer(bitPattern: addr) else { return nil }
    return Unmanaged<SessionCtl>.fromOpaque(opaque).takeUnretainedValue()
}
```

Orphan 檢查（`cachedSession` 與 `evictLRUIfNeeded`）同步改為 `session.inputControllerAssignedAddr == nil`。`unregisterSessionAddr(for:)` 會在清理 controller 對照時連帶將 session 的 `inputControllerAssignedAddr` 設為 nil，確保 orphan 偵測正確。

自此 `InputSession` 與 `SessionCtl` 之間的雙向參照均改為純 UInt 位址 lookup + `takeUnretainedValue()`。

#### 六、SessionCtl 閉包內 [weak self] 以位址技巧取代

**受影響之原始碼**: `SessionCtl.swift`（雙倉）

`getClientAddrProvider()` 與 `callCoreAtLeastOnce` asyncOnMain 內的 `[weak self]` 兩倉均改為 UInt 位址 capture：

```swift
// Before
{ [weak self] in ... self?.client() ... }
asyncOnMain { [weak self] in guard let this = self else { return } ... }

// After
let thisAddr = UInt(bitPattern: Unmanaged.passUnretained(self).toOpaque())
return { guard let opaque = UnsafeRawPointer(bitPattern: thisAddr) else { return nil }
         let this = Unmanaged<SessionCtl>.fromOpaque(opaque).takeUnretainedValue()
         ... this.client() ... }
```

另 `SessionCtl.hidePalettes()`、`inputControllerWillClose()` 與 `selectionRange()` 在兩倉均改為直接呼叫 `core?.xxx()`，asyncOnMain / mainSync 包裝移至 `InputSession` 內部（`InputSession.hidePalettes()` 與 `InputSession.inputControllerWillClose()` 內的新增 asyncOnMain 閉包使用 `[weak self]` 作為 InputSession 層的保護）。

#### 七、AppDelegate 不持有任何 stored property

**受影響之原始碼**: `AppDelegate.swift`（雙倉）

將唯一的 instance stored property `private var folderMonitor: FolderMonitor` 改為 static `NSMutex<FolderMonitor>`：

```swift
// Before
private var folderMonitor = FolderMonitor(url: ...)

// After
private static var folderMonitor = NSMutex<FolderMonitor>(FolderMonitor(url: ...))
```

連帶將 `reloadOnFolderChangeHappens(forced:)` 改為 `static` 方法（因其內部只存取 global singleton）。所有 `[weak self]` / `[weak this]` 閉包改為直接呼叫 `Self.reloadOnFolderChangeHappens()` 或 `Self.folderMonitor.withLock { ... }`。`suppressUserDataMonitor(for:)` 與 `updateDirectoryMonitorPath()` 同步更新為 `Self.folderMonitor.value.*` / `Self.folderMonitor.withLock { ... }`。

自此 `AppDelegate` 沒有 instance stored property──`shared = AppDelegate()` 的 init 只有 `super.init()`（NSObject），零 ARC 參與。

#### 八、Sendable / nonisolated / Actor Isolation 調整

- `sessionAddrByControllerAddr`（`NSMutex<[Int: Int]>`）兩倉均標記 `nonisolated(unsafe)`，使其可在 `nonisolated static unregisterSessionAddr` 與 `SessionCtl.deinit`（nonisolated）中安全存取。
- `inputControllerAssignedAddr` 兩倉均標記 `nonisolated(unsafe)`，原因同上。
- 三支 helper (`sessionAddr` / `registerSessionAddr` / `unregisterSessionAddr`) 標記 `nonisolated static`，使其可從 nonisolated context（如 deinit）呼叫。

### 已知權衡

- **Xcode 26.x compatibility**：由於 Xcode 26.x (Swift 6.3) 的 SIL ownership verifier bug，`-fno-objc-arc` 在 Swift target 上採條件式啟用（`#if compiler(>=6.4)`）。這意味著在 Swift 6.3 工具鏈上，僅 `IMKSwift`（純 ObjC target）的 ARC 停用生效；Swift target 本身（`MainAssembly4Darwin`）的 `-fno-objc-arc` 會跳過。實際影響極小，因為 `SessionCtl` 本身已無 stored property 且所有 cross-reference 均為 UInt 位址。
- **`takeUnretainedValue()` 安全性依賴 IMK 呼叫生命週期**：所有 `takeUnretainedValue()` 的呼叫均發生在 IMK 同步事件處理期間（或 deinit 序列內），此時參照對象保證存活。asyncOnMain 路徑已排除此模式。

## 手術範圍

本次手術同時牽涉到對 `vChewing-macOS`、`vChewing-OSX-Legacy` 兩個倉庫。

## 總體手術目的

唯音輸入法 4.5.5 版發版之後，為了提升記憶體利用效率、減少 dangling pointer 作祟之可能，故決定動手術引入至少下述設計：

- 將 `IMKInputController` 的 Subclass 過程從 Swift Codebase 完全驅逐、僅保留 ObjC 的 subclass `IMKInputSessionController`。在 IMKSwift ObjC 層面引入諸如 assignable lambda properties 等特性、使得 barebone controller class 整個生命週期管理都不沾 Swift。（`IMKTextInput` Client 的生命週期則在 MRC 的前提下已經被系統自動妥善管理。）
- 追蹤 `IMKInputSessionController` 的生命週期，在輸入法選單內得以隨時注意到當前存活的 `IMKInputSessionController` 副本數量。
- 輸入法選單由專門的 Sputnik 管理。

## 手術報告

### 一、核心架構變更：從 `SessionCtl` 到 `IMKInputSessionController` + Block Dispatch

**問題**：Swift `SessionCtl`（`IMKInputController` 子類別）的整個生命週期都沾染 Swift ARC，與 IMK 內部 MRC 機制交織，導致在 ARC 啟用時「CapsLock 頻繁切換時出現 retain/release 卡頓」且在 ARC 停用時「記憶體利用效率變得不夠高」，以及 dangling pointer crash（見 Phase 111 的 v4.5.5 停用 ARC 的後續分析）。

**方案**：將所有 `IMKInputController` protocol 實作從 Swift 層剝離，移入 ObjC MRC target 的 `IMKInputSessionController`，透過 assignable block properties 讓 Swift 端僅在 init 階段賦值閉包，後續所有 IMK 回呼均由 ObjC 端直接 invoke block，完全不經 Swift ARC。

**具體變更**：

| 檔案 | 變更說明 |
|------|----------|
| `IMKSwift.h` / `IMKSwift.m` | 新增 12 個 block properties（`onActivatingServer`、`onDeactivatingServer`、`onDealloc`、`onShowingPreferences`、`onHidingPallettes`、`onInputControllerWillClose`、`onProvidingSelectionRange`、`onProvidingIMEMenu`、`onProvidingComposedString`、`onAutoCommittingComposition`、`onProvidingRecognizedEvents`、`onHandlingGivenNullableEvent`、`onSettingObjCValue`），全部以 MRC macro 實作 getter/setter。所有 `IMKInputController` override 均改為呼叫對應 block，傳入 raw `uintptr_t` 而非物件參照。 |
| `SessionControllerSputnik.swift` | 取代原 `SessionCtl`。包含 `assignBlocks()`（為所有 block properties 賦值）、`callCoreAtLeastOnce()`（建立或複用 InputSession）、`_installConstructorHook`（透過 `class_addMethod` + `NSInvocation` 在 controller init 完畢後同步完成 track + assignBlocks + InputSession 建立）。引入 `clientServerPairAction<T>` 泛型工廠：所有需 client 的 block 均透過 `ClientControllerAddrPair.unwrapped` 做 tracker 安全核驗後才解讀記憶體位址。 |
| `ClientAddrUnwrapper.swift`（macOS） / 內聯（Legacy） | 泛型輔助結構，封裝 `(givenClientAddr, thisAddr)` 的安全解讀模式。 |
| `IMEMenuSputnik.swift` | 取代 `SessionCtl_Menu.swift`。使用 `class_addMethod` + `imp_implementationWithBlock` 動態註冊 synthetic selectors（`IMEMenuAction_N:`），以 closure IMP 直接綁定選單命令。IMK 透過 Carbon `didCommandBySelector:client:` dispatch menu → selector → block invoke。每次 `build()` 時計數器歸零，避免 stale selector 累積。 |
| `SessionCtl.swift` | 刪除。所有邏輯移入 Sputnik。 |
| `SessionCtl_Menu.swift` | 刪除。選單邏輯移入 `IMEMenuSputnik`。 |

### 二、記憶體安全：`ObjCMemoryLeakTracker` + `ClientControllerAddrPair`

**問題**：`Unmanaged.takeUnretainedValue()` 對 dangling pointer 無防護。Controller 被 dealloc 之後，任何對其記憶體位址的解讀都會導致 EXC_BAD_ACCESS。

**方案**：
1. **`ObjCMemoryLeakTracker`**：以 `NSMapTable`（zeroing weak）追蹤所有 `IMKInputSessionController` 實例。每次 init 時 `track()`，dealloc 時透過 `DeallocSentinel` 自動 `untrack()`。提供 `isTracked(addr:)` 查詢介面。
2. **`ClientControllerAddrPair`**：包裝 `(clientAddr, controllerAddr)`。`unwrapped` 屬性在解讀前先透過 `IMKControllerLifetimeTracker.shared().isAddressAlive(addr:)` 核驗 controller 是否仍存活。
3. **`_installConstructorHook` init 路徑**：使用 constructor 傳入的 `givenClient`（非 `client()`），因為 macOS 10.9 下 `client()` 在建構期間為 nil。同步完成 track + assignBlocks + InputSession 建立，確保 `activateServer:` 觸發前一切就緒。
4. **`onDealloc` block**：在 ObjC `-[IMKInputSessionController dealloc]` 內呼叫，執行 `InputSession.unregisterSessionAddr(forControllerAddr:)` 清理對照表。

### 三、Session 生命週期：LRU 快取 + Deferred Dealloc

**快取優化**：
- `sessionsByClient`（`NSMutex<UInt, InputSession>`）：以 client 記憶體位址為鍵的快取字典。
- `sessionAddrByControllerAddr`（`NSMutex<UInt, UInt>`）：controller → session 位址對照表，不保留強引用。
- **LRU 淘汰**：快取容量超過 `max(2, 當前存活 controller 數)` 時淘汰最舊的非活躍 session。
- **LRU eviction dangling fix**：淘汰時同步清理 `sessionAddrByControllerAddr` 中對應條目，避免後續 `session(for:)` 取出已釋放的位址導致 crash。

**Deferred Dealloc**：
- `IMKInputSessionController` 在 `deactivateServer:` 後排程 `performSelector:afterDelay:3.0`。
- `activateServer:` 時取消排程。
- 若 3 秒內未重新啟用，`IMKSwift_delayedDealloc` 觸發：釋放全部 13 個 block ivars 並呼叫 `_onDealloc` 清理 Swift 端對照表。若 timer 為 controller 的最後一個外部 retain，則 controller 隨後 dealloc。
- `activateServer:` 時若發現 blocks 已被釋放（`!_onActivatingServer`），則透過 NSInvocation 重新觸發 init hook，完整重建 Swift 端狀態。

### 四、逆向工程結論

- **IMK（10.9 / 10.15）**：IMKInputController 與 client DO proxy 為 1:1 共生關係，controller 從不被跨 client 複用。LRU 的理論基礎成立。
- **Menu dispatch**：IMKServer 內部使用 `didCommandBySelector:client:`，無視 `NSMenuItem.target`/`action`。`NSMenuActionTrampoline`（associated object 方案）對 IMK menu 無效，僅能用 `class_addMethod`。
- **`uniqueClientIdentifierString`**：XPC async RPC，初期可能為空字串 → 快取以 raw proxy addr 為 fallback key。
- **CpLk 切換時的 Controller 孤兒化**：每次 CpLk toggle 時，IMKServer 建立的 DO/XPC proxy 物件記憶體位址均不同。`_controllers` dictionary 以 `[NSNumber numberWithUnsignedLong:proxyAddr]` 為 key，故 toggle 後舊 controller 永遠無法被查詢命中，`sessionFinished:` 也永遠不會為舊 proxy 觸發——造成 controller 實例在 `_controllers` 內永久孤兒化。

### 五、本次善後修正（Phase 112 vs Phase 111）

| 問題 | 根因 | 修復 |
|------|------|------|
| `activateServer:` 時 EXC_BAD_ACCESS crash | LRU eviction 只清 `sessionsByClient` 未清 `sessionAddrByControllerAddr`，dangling addr 在下一次 `session(for:)` 時被 `takeUnretainedValue()` 解讀為已釋放的 InputSession | 淘汰時同步清理 controller→session 對照表 |

### 七、後續修正（2026-07-24）

Phase 112 發版後，透過 Instruments 記憶體追蹤（`MemoryLeakRecords.trace`）、macOS 27.0 實機 crash log（`.ips`）及對 IMK 10.9/10.15 的逆向工程，定位並修復了三項記憶體相關缺陷。

#### 7.1 `asyncOnMain` dangling pointer crash

**問題**：`callCoreAtLeastOnce` 內以 `asyncOnMain` 非同步重指派 `InputSession` 的 clientAddrProvider。該 closure 捕獲 `controllerAddr`（UInt），在 async dispatch 期間 controller 可能已於 macOS 27.0 下被 IMKServer 釋放，導致 `takeUnretainedValue()` 解讀已釋放記憶體 + Swift ARC 對參數插入 `objc_retain` → `EXC_BAD_ACCESS`。

**修復**（`SessionControllerSputnik.swift`，雙倉）：在 `asyncOnMain` closure 內 `takeUnretainedValue()` 之前加入 `IMKControllerLifetimeTracker.shared().isAddressAlive(addr:)` guard。controller dealloc 時 `DeallocSentinel` 自動 untrack，async closure 即可安全 early-return。

```swift
// Before:
asyncOnMain {
    guard let opaque = UnsafeRawPointer(bitPattern: controllerAddr) else { return }
    let controllerUnwrapped = Unmanaged<...>.fromOpaque(opaque).takeUnretainedValue()
    ...
}

// After:
asyncOnMain {
    guard IMKControllerLifetimeTracker.shared().isAddressAlive(addr: controllerAddr),
          let opaque = UnsafeRawPointer(bitPattern: controllerAddr) else { return }
    ...
}
```

#### 7.2 Deferred Dealloc 主動釋放 Blocks

**問題**：Phase 112 的 `IMKSwift_delayedDealloc` 為空方法，僅作為 timer target。controller 被 CpLk deactivate 後，雖 timer 觸發，但 blocks 永遠不被釋放（因 IMKServer 仍持有 controller）。

**修復**（`IMKSwift.m`，雙倉）：`IMKSwift_delayedDealloc` 改為主動 release 全部 13 個 block ivars 並呼叫 `_onDealloc`：

```objc
- (void)IMKSwift_delayedDealloc {
    if (_onDealloc) _onDealloc((uintptr_t)self);
    [_onDealloc release]; _onDealloc = nil;
    [_onActivatingServer release]; _onActivatingServer = nil;
    // ... 13 blocks 全數 release
}
```

`activateServer:` 則在偵測到 blocks 已被清空時，透過 NSInvocation 重新觸發 init hook 重建 Swift 端狀態。

#### 7.3 Orphan Controller 主動掃除

**問題**：CpLk 每次 toggle 都建立新的 DO/XPC proxy → `_controllers` 內 key（proxy 位址）每次不同 → 舊 controller 永遠留在 `_controllers` 內 → `sessionFinished:` 永遠不會為舊 proxy 觸發 → controller 實例永久孤兒化。

**修復**（`IMKSwift.m`，雙倉）：

1. 新增 `static uint64_t _IMKSwift_controllerGeneration` 計數器，以 `objc_setAssociatedObject` 在每個 controller 上 stamp generation number
2. 新增 `+IMKSwift_pruneStaleControllersOnServer:excludingSelf:` class method：
   - 當 `_controllers.count > 3` 時觸發
   - 掃描 `_controllers.allValues`，排除 `_currentController` 與 `self`
   - 找出 generation 最舊的 controller
   - 反查 dictionary key 後 `removeObjectForKey:`
3. `-initWithServer:delegate:client:` 在 super init 成功後、hook 觸發前，先 stamp generation 再呼叫 prune

```objc
- (instancetype)initWithServer:(IMKServer *)server ... {
    self = [super initWithServer:server ...];
    if (self) {
        NSNumber *gen = [NSNumber numberWithUnsignedLongLong:++_IMKSwift_controllerGeneration];
        objc_setAssociatedObject(self, &kIMKSwiftGenerationKey, gen, OBJC_ASSOCIATION_RETAIN_NONATOMIC);
        [IMKInputSessionController IMKSwift_pruneStaleControllersOnServer:server excludingSelf:self];
        // ... hook dispatch ...
    }
    return self;
}
```

**清理鏈**（純 ObjC MRC，無 Swift ARC 參與）：

```
delayedDealloc (3s timer):
  → _onDealloc(addr) → Session 對照清理
  → terminateForClient*(clientProxy) → 從 IMK 全域 wrapper 快取移除
     → wrapper dealloc → NSXPCConnection 釋放

prune (下次 init 時):
  → removeObjectForKey: → [controller release] → retain count → 0
     → dealloc → [super dealloc] → associated objects cleanup
          → DeallocSentinel dealloc → ObjCMemoryLeakTracker.unregister(addr)
```

Client wrapper terminate 移至 `IMKSwift_delayedDealloc`——與 controller 自身的 soft-cleanup 同週期，消除 prune 內的 T0→T1 時間差疑慮。逆向 `IPMDServerClientWrapper` 確認 `terminateForClientDOProxy:` / `terminateForClientXPCConn:` 會從 `_gInputClientWrapperDictionary` 移除 wrapper 條目。

**編譯器警告修復**（2026-07-24）：因 `+terminateForClientXPCConn:` / `+terminateForClientDOProxy:` / `+terminateForClient:` 為 IMK private class methods，無公開 header 宣告，Clang 對 `[wrapperClass terminateForClientXPCConn:...]` 發出 `+terminateForClientXPCConn: not found` 警告。修復方案為在 `IMKSwift.m` 頂部新增 forward-declared category：

```objc
@interface NSObject (IPMDServerClientWrapperTermination)
+ (void)terminateForClientXPCConn:(id)client;
+ (void)terminateForClientDOProxy:(id)client;
+ (void)terminateForClient:(id)client;
@end
```

此 category 僅提供 compile-time selector 簽名資訊——`respondsToSelector:` 在 runtime 仍以實際 class 為對象檢查方法是否存在。

**macOS 15 Sequoia ~ 27 相容性修復**（2026-07-24）：逆向 macOS 15 Sequoia ~ 27 GoldenGate 的 `InputMethodKit.framework`，發現 `IPMDServerClientWrapper` 類別已被拆分為 `_IPMDServerClientWrapperModern` 與 `_IPMDServerClientWrapperLegacy`——`NSClassFromString(@"IPMDServerClientWrapper")` 在 macOS 15 Sequoia 起將回傳 `nil`。同時，`IMKServer`、`IMKServerPrivate` 也做了同樣的 Modern/Legacy 雙軌拆分，但 `_controllers` ivar 仍同時存在於 `_IMKServerPrivateModern` 與 `_IMKServerPrivateLegacy`（KVC prune 路徑不受影響）。`+terminateForClient:` 無後綴版本已自 macOS 15 Sequoia 起移除。

修復方案：`IMKSwift_delayedDealloc` 內的 `NSClassFromString` 改為依序嘗試 `_IPMDServerClientWrapperModern` → `_IPMDServerClientWrapperLegacy` → `IPMDServerClientWrapper`（fallback 供 ≤10.15）：

```objc
Class wrapperClass = nil;
for (NSString *name in @[
    @"_IPMDServerClientWrapperModern",
    @"_IPMDServerClientWrapperLegacy",
    @"IPMDServerClientWrapper"
]) {
    wrapperClass = NSClassFromString(name);
    if (wrapperClass) break;
}
```

| 檔案 | 變更內容 |
|------|---------|
| `IMKSwift.m`（雙倉） | 新增 prune 機制；prune 時連帶 terminate client wrapper；`initWithServer:` 內 stamp generation + 呼叫 prune；`IMKSwift_delayedDealloc` 改為主動 release blocks；`terminateForClient*` class lookup 改為 Modern → Legacy → fallback 三階嘗試以相容 macOS 15 Sequoia 起 |

## 原始想法

我在尋思 InputSession LRU Table 容量上限 與 IMKInputSessionController 清理閾值 或許還是適合設為 2。

原因：不可能會有超過兩個 session 同時工作。其中一個 session 負責當前打字，另一個 session 負責 deactivateServer 階段的收尾工作、且這個工作往往發生在新的 session 開始執行 activation 之後。

## 可行性論證（by Deepseek）

該論證是對的。

### 時間線分析

IMK 所有事件都在 main thread 序列化執行：

```
T1: app A activateServer  → session_A 活躍
T2: app A deactivateServer → delayedDealloc 排程 (3s)
T3: app B activateServer  → session_B 活躍
                              session_A 仍在「冷卻中」(timer 未觸發)
                              此時共存: 2 個 session

T4: (3s 後) delayedDealloc_A 觸發 → blocks 釋放 → session_A 變成孤兒
                              此時共存: 1 active + 0 cooling = 1

T5: app C activateServer  → session_C 活躍
                              session_B 進入冷卻 (timer 剛排程)
                              此時共存: session_C + session_B = 2

-- 極端情況：3 秒內切 3 個 apps --
T6: app D activateServer  → 此時 session_B, session_C 都還在冷卻
                              共存: 3 個 session
                              但這極罕見 (< 3 秒內連切 3 apps 且都要打字)
```

### 為什麼 2 足夠

- **活躍 session**：永遠只有 1 個（唯一正在處理按鍵的）
- **冷卻 session**：最多 1 個（上一個 deactivate 後 3 秒內）
- 冷卻 session 的用途：處理 `performServerDeactivation` 的收尾（但實際上 `guard Self.current?.id == id` 會讓它 no-op，因為 current 已切到新 session）

超過 2 的第三個 session 必然是孤兒（其 controller 已被 prune 或 blocks 已被 `delayedDealloc` 釋放），留著無意義。

### 可同時改兩處

| 位置 | 現行 | 改為 |
|------|------|------|
| `IMKSwift_pruneStaleControllersOnServer` | `count <= 3`（保留 3 controllers） | `count <= 2`（保留 2） |
| `evictLRUIfNeeded` | `max(2, trackedCount)` | `2` |

維護 `_controllers` 內 3 個 controller 是多餘的——第三個必定是前兩次 CpLk toggle 之前的 stale controller。同樣，LRU cache 保留 2 個 session 已覆蓋「當前打字 + 冷卻收尾」的全部場景。

## 手術範圍

本次手術同時牽涉到對 `vChewing-macOS`、`vChewing-OSX-Legacy` 兩個倉庫。

## 總體手術目的

我有一個辦法徹底減少讓 InputSession 重建的必要性：
準備兩個 static InputSession instances，彼此極性互轉。
IMKSwift 端給 IMKInputSessionController 曝露其「極性」參數，該「極性」參數可以由自身的 generation number 的奇偶狀態來確定。根據這個奇偶狀態令其對接不同的 InputSession singleton。

## 可行性調查結果

### 極性雙緩衝（Parity-based Double Buffering）機制

Phase 112 §7.3 已引入 `_IMKSwift_controllerGeneration`（`static uint64_t`，每次 `initWithServer:` 遞增）。本方案的對映邏輯極簡：

```
Controller gen 0 (偶) → Session A
Controller gen 1 (奇) → Session B
Controller gen 2 (偶) → Session A（複用）
Controller gen 3 (奇) → Session B（複用）
```

每次 CpLk toggle 必定產生一個新 controller、generation 遞增、極性翻轉，新 controller 自動對接到**另一個** session。上一個 session 保持原樣，等待其所屬的舊 controller 在 `delayedDealloc`（3 秒 timer）後自然善後。

### 可消除的既有機制

| 既有機制 | 消除原因 |
|----------|----------|
| `sessionsByClient` LRU dictionary | 無需動態建立／淘汰 session。兩個 static singleton 永久存活 |
| `evictLRUIfNeeded()` | 無 LRU → 無淘汰邏輯 |
| `hasOrphan` 偵測（Phase 113） | session 永不變成孤兒——它始終保留，等待下一個同極性 controller 接手 |
| `max(2, trackedCount)` 閾值計算 | 無動態 pool → 無數量管理 |
| `InputSession` 的動態 `init()` / dealloc | 僅在輸入法啟動時配置一次，之後不新建也不釋放 |

### InputSession 的行為變更

`InputSession` 不再以「一個 controller → 一個 session」模式新建／銷毀。改為以 `reassign` 為核心生命週期操作：

當新 controller（與當前 session 同極性）透過 `_installConstructorHook` 觸發 `callCoreAtLeastOnce` 時：

1. **清理組字狀態**：`composer`、`inputHandler`、`IMEState` 重設為 `.ofEmpty`
2. **更新 `theClientAddr` provider**：指向新 controller 的 client proxy（cpLk toggle 後位址已變）
3. **更新 `inputControllerAssignedAddr`**：指向新 controller
4. **更新 `sessionAddrByControllerAddr`**：移除舊 controller 位址 → 寫入新 controller 位址
5. **設定 `Self.current`**：指向自身

### 舊 Controller 延遲善後的安全性

CpLk 兩次快速切換（A→B→A）的場景：session A 被新 controller（gen 2，偶）重新接手，而舊 controller（gen 0，偶）的 `delayedDealloc` 還在 3 秒倒數中。

`delayedDealloc` 最終會呼叫 `_onDealloc(addr)` → `InputSession.unregisterSessionAddr(for:)`。該方法會執行：

```swift
sessionAddrByControllerAddr.withLock { $0[controllerAddr] = nil }
```

由於舊 controller 位址 ≠ 新 controller 位址，dictionary key 不同，不會誤刪新對映。舊 controller 的 `DeallocSentinel` 隨後觸發 `ObjCMemoryLeakTracker.unregister(addr)`，完成追蹤器清理。

**無 cross-talk 風險。**

### session 初始化時機

兩枚 `InputSession` singleton 不需要在 `_installConstructorHook` 內建立。可在 `SessionControllerSputnik` 初始化階段（`_configureClassLevelBlocks` 之前或同時）預先配置：

```swift
static let sessionEven = InputSession(preallocated: true)
static let sessionOdd  = InputSession(preallocated: true)
```

`preallocated` 旗標表示此 session 不繫結任何 controller/client，僅配置內部引擎（composer、inputHandler、language model）。實際的 client binding 發生在首次 `reassign` 時。

### 需要修改的檔案

| 檔案 | 變更 |
|------|------|
| `InputSession.swift`（雙倉） | 新增 `preallocated` init 路徑；移除 `sessionsByClient`、`evictLRUIfNeeded`、`hasOrphan`；session 查詢改為 parity-based：`InputSession.session(forParity:)` |
| `SessionControllerSputnik.swift`（雙倉） | `callCoreAtLeastOnce` 改為 parity lookup + reassign 而非新建；新增兩枚 static session 的預配置 |
| `IMKSwift.m`（雙倉） | 無需變更——`_IMKSwift_controllerGeneration` 已存在，極性由 `generation % 2` 決定 |
| `IMKSwift.h`（雙倉） | 可選：暴露 `+ (uint64_t)IMKSwift_currentGeneration` 供 Swift 端查詢（或用 KVC 讀取 associated object） |

### 判定

**可行。** 極性雙緩衝方案簡化了 `InputSession` 的整個生命週期管理：無動態分配、無淘汰、無孤兒偵測。CpLk toggle 的自然行為（每次切換產生一個新 controller）與 generation parity flip 完全吻合，不引入任何新 timing 風險。

| `sessionsByClient` (`NSMutex<[UInt: InputSession]>`) | client-addr-based 快取字典 |
| `globalLRUTick` / `lruTick` | LRU 記數器 |
| `evictLRUIfNeeded()` | 惰性 LRU 淘汰（含 `hasOrphan` 偵測） |

**新增項目**：

```swift
// 極性雙緩衝池
static let sessionEven = InputSession(preallocated: ())
static let sessionOdd  = InputSession(preallocated: ())

static func session(forParity parity: Int) -> InputSession {
    return (parity & 1) == 0 ? sessionEven : sessionOdd
}

// 預配置 init：不繫結 controller/client，僅初始化內部引擎
public init(preallocated: ()) {
    self.theClientAddr = { nil }
    initInputHandler()
    synchronizer4LMPrefs?()
    inputMode = .init(rawValue: prefs.mostRecentInputMode) ?? .imeModeNULL
}
```

**安全修正**：

`unregisterSessionAddr` 加入 guard——僅在 session 仍屬於該 controller 時才清空 `inputControllerAssignedAddr`：

```swift
// Before:
session.inputControllerAssignedAddr = nil  // 無條件清空

// After:
if session.inputControllerAssignedAddr == ctlKey {
    session.inputControllerAssignedAddr = nil
}
```

`reassign` 同步清理舊 controller 的 mapping，避免 `sessionAddrByControllerAddr` 殘留 stale entries：

```swift
// Before: 僅寫入新 mapping，舊 key 殘留
Self.registerSessionAddr(self, for: controllerAddr)

// After: 先清舊 key，再寫入新 key
if let oldAddr { Self.sessionAddrByControllerAddr.withLock { $0[oldAddr] = nil } }
Self.registerSessionAddr(self, for: newAddr)
```

#### 三、SessionControllerSputnik：改為 parity lookup + reassign

**`callCoreAtLeastOnce`** 從 ~40 行縮減為 ~10 行：

```swift
// Before: cachedSession lookup → new InputSession → asyncOnMain reassign
// After:
let parity = Int(IMKInputSessionController.imkSwift_currentGeneration() & 1)
let session = InputSession.session(forParity: parity)
session.reassign(to: controller, clientAddrProvider: Self.getGuardableAddrPair(...))
session.state = IMEState.ofEmpty()
session.resetInputHandler()
InputSession.current = session
return session
```

消除項目：
- `asyncOnMain` 延遲 reassign（Phase 112 §7.1 的 dangling pointer crash 修正不再需要——無新建 session 流程）
- `cachedSession(for:)` 查詢 + `InputSession(controller:)` 新建 + `registerInCache` 路徑
- `vCLog("InputSession reused. ID: ...")` 診斷日誌（reassign 是唯一路徑，無需區分「新建 vs 複用」）

#### 四、Legacy 相容性

- `sessionAddrByControllerAddr` 保留 `nonisolated(unsafe)`（Legacy Swift 5.9+ 支援）

### Phase 115 後續修正（2026-07-24）

#### 一、`performServerActivation` fast path 未更新 `clientBundleIdentifier`

**問題**：`performServerActivation` 的 fast path 僅檢查 `Self.current?.id == id`（同一枚 singleton），未檢查 client 是否已變更。當不同 App 先後對接到同一枚 parity session 時，`clientBundleIdentifier` 殘留上一個 App 的值。

**修復**（`InputSession_CoreProtocol.swift`，雙倉）：

```swift
// Before:
if isActivated, Self.current?.id == id, inputHandler != nil {

// After:
if isActivated, Self.current?.id == id, inputHandler != nil,
   let client, isStillTheSameClientObj(client as? NSObject) {
```

僅當 client 物件本體也相同時才走 fast path；client 變更時強制走 full path 以更新 `clientBundleIdentifier`、`isServingIMEItself`、`isClientElectronBased` 等 per-client 屬性。

#### 二、`ObjCMemoryLeakTracker` → `IMKControllerLifetimeTracker` 全面更名

Swift 端 `ObjCMemoryLeakTracker`（`vChewing_Shared_DarwinImpl`）連同其依賴的 `ObjCUtils` target（`DeallocSentinel`）已被移除。所有 controller 生命週期追蹤統一由 `IMKSwiftModernHeaders` target 內的 ObjC 實作 `IMKControllerLifetimeTracker` 負責，不再依賴 Swift 端自訂 tracker。`IMKInputSessionController.-initWithServer:` 自動呼叫 `trackController:`，無需 Swift 端手動呼叫。

受影響檔案（雙倉）：`InputSession.swift`、`SessionControllerSputnik.swift`、`IMEMenuSputnik.swift`——所有 `ObjCMemoryLeakTracker.shared.*` 呼叫均改為 `IMKControllerLifetimeTracker.shared().*`。
