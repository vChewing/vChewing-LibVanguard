Homa 在插入新讀音的時候是會根據 Maximum Segment Length (maxSegLength) 半徑取出可能的所有讀音排列組合、再據此查詢 Lexicon 的。這很可能是笛卡爾積爆炸的來源，除非掃描半徑內所有讀音 PossibleKey 都只是 singleKey。

## 前期討論與施術方針（2026-04-26）

### 現況確認

1. **`LMCoreEX` 目前為純 exact-match**：`rangeMap` 與 `temporaryMap` 皆為 `[String: …]` Dictionary，查詢時直接以完整讀音鍵 subscript，無任何前綴、子串或聲母索引。
2. **`VanguardTrie.Trie` 雖具 partial match，但不適合取代 `LMCoreEX`**：
   - 缺乏單一 entry 精細刪除 API（僅有 `insert` 與 `clearAllContents`）
   - 持久化成本高昂（每次變更需整棵樹重新編碼為 Plist）
   - 記憶體開銷遠大於 `LMCoreEX`（節點碎片化 + Dictionary 開銷）
   - 使用者語彙資料量小（數百至數千條），`LMCoreEX` 的 O(1) hash lookup 已足夠快，Trie 的查詢加速優勢微不足道
3. **使用者辭典不需要 Reverse Lookup**：原廠辭典（`VanguardTrie.TextMapTrie`）內建的反查資訊已足夠覆蓋需求。

具體設計待研究階段進一步細化，但大方向為：
- 維持現有的 `rangeMap` / `temporaryMap` exact-match 結構不變
- 額外引入一個前綴索引（例如排序後的 `sortedKeys: [String]` 或輕量 `prefixMap: [String: Set<String>]`）
- partial match 時先以前綴索引縮小候選 key 範圍，再對候選 key 走現有的 exact match 路徑
- 前綴索引的維護成本必須低於 Trie 的整樹重建，且不能影響 `saveData()` / `replaceData()` 的效能特性

1. 拼音模式（支持包括漢語拼音與華羅拼音等多種方案）的組音區的直接唸讀恐無法受系統的完美支持。或許應該從後端將注音符號塞給系統的唸讀器？
2. 「用聲調確認」時觸發的唸讀對象或得改從 Homa.Assembler 的 actualKeys 陣列取得，至少這體現了最新的 pathfinding 結果所拾取的讀音。

## 前期討論與施術方針（2026-04-26）

### 現況確認

1. **`narrateTheComposer` 目前以 `composer.phonabetKeyForQuery()` 作為預設朗讀內容**：`phonabetKeyForQuery` 呼叫 `getComposition()`（預設 `isHanyuPinyin: false`），因此無論注音模式或拼音模式，其回傳值均為注音符號（bopomofo）。`SpeechSputnik` 接收到的朗讀文本在絕大多數情況下已是注音。
2. **拼音模式下的潛在漏洞**：雖然 `phonabetKeyForQuery` 回傳注音，但 `narrateTheComposer` 的 `with maybeKey` 參數允許呼叫端傳入任意字串（如 `readingKey.first`）。在拼音無調確認組字路徑中，`readingKeyForQuery` 回傳的是聲調候選桶（如 `["ㄋㄧㄢ", "ㄋㄧㄢˊ", ...]`），`readingKey.first` 為第一聲（陰平）變體，未必反映使用者實際意圖或 pathfinding 結果。
3. **「用聲調確認」的朗讀時機與對象問題**：
   - 正常組字路徑（`composeReadingIfReady`）在第 284 行於 `handler.assemble()` **之前**觸發朗讀，對象為 `readingKey.first`（原始輸入讀音）。
   - 後置聲調覆寫路徑（`performRearIntonationOverrideIfNeeded`）在 `.success` 後**完全沒有朗讀**，使用者無法透過語音回饋確認覆寫結果。
   - 兩種「用聲調確認」情境均未使用 `assembler.actualKeys`，無法反映組句引擎 pathfinding 後的實際讀音。
4. **`actualKeys` 的語義**：`Homa.Assembler.actualKeys` = `assembledSentence.keyArrays.flatMap(\.self)`，即全部節點的實際讀音鍵的扁平陣列。`actualKeys.last` 為最新插入字元的實際讀音；對於後置聲調覆寫，`actualKeys[assembler.cursor - 1]` 為被覆寫字元的實際讀音。

### 施術方針

**A. 朗讀內容強制注音化（防禦性轉換）**

在 `narrateTheComposer` 內新增一層防禦：無論 `maybeKey` 來源為何（`composer.phonabetKeyForQuery`、`readingKey.first`、`actualKeys.last` 等），若內容包含 ASCII 字母（表明可能是拼音殘留），一律嘗試以 `Tekkon.cnvHanyuPinyinToPhona(targetJoined: newToneOne: "")` 轉換為注音。若轉換後仍殘留 ASCII 字元（非漢語拼音方案或無效輸入），則退回 `composer.value.swapping(" ", with: "")`（作曲家內部永遠以注音儲存）。此舉確保 `SpeechSputnik` 永遠收到系統 TTS 可正確朗讀的注音符號。

**B. 正常組字路徑改以 `actualKeys` 為朗讀對象**

將 `composeReadingIfReady` 第 284 行的 `narrateTheComposer` 呼叫後移至 `handler.assemble()` 之後（第 290 行之後），以 `handler.assembler.actualKeys.last` 取代 `readingKey.first` 作為朗讀對象。如此可確保：
- 無調拼音確認時，朗讀的是 pathfinding 實際選中的聲調（而非硬編碼的第一聲變體）。
- 有調注音確認時，朗讀的是 pathfinding 確認後的最終讀音（與輸入通常一致，但語義更正確）。

**C. 後置聲調覆寫路徑補上朗讀回饋**

在 `performRearIntonationOverrideIfNeeded` 的 `.success` 分支內、`handler.assemble()` 之後，新增 `narrateTheComposer` 呼叫，以 `handler.assembler.actualKeys[assembler.cursor - 1]` 為朗讀對象。使使用者在覆寫聲調後能透過語音確認新讀音。

**D. 測試策略**

- 新增 Typewriter 單元測試：模擬拼音模式輸入，驗證 `narrateTheComposer` 接收到的朗讀文本為注音（非拼音）。
- 新增 Typewriter 單元測試：模擬無調拼音確認，驗證朗讀文本與 `actualKeys.last` 一致。
- 新增 Typewriter 單元測試：模擬後置聲調覆寫，驗證覆寫後有觸發朗讀且文本正確。
