## 術語消歧義

Homa 以 `previous` 欄位區分 unigram／bigram，與一般 NLP 領域的 n-gram 定義不同。本文所述「多期」是 **多段 unigram**（keyArray.count ≥ 2 但 `previous == nil`），不是 Homa bigram。

| 術語 | Homa 定義 | 本文所指 |
|------|----------|---------|
| **unigram** | `previous == nil`，keyArray 長度不拘 | 涵蓋單段（「多」）與多段（「多期」） |
| **多段詞（多段 unigram）** | unigram 且 keyArray.count ≥ 2 | 「多期」（`["ㄉㄨㄛ", "ㄑㄧˊ"]`） |
| **Homa bigram** | `previous != nil`，形如 `(美國)(人(ㄖㄣˊ))` | 未涉及 |

本文下文以「多段詞」專指 segLength > 1 的 unigram，以「單段詞」專指 segLength == 1 的 unigram。

## 手術範圍

## 問題現狀

使用者打了「ㄉㄨㄛ-ㄑㄧˊ」顯示「多期」，然後使用者手動將 pos 1 的位置指定了候選字「奇」且遞交出去。之後他再打「ㄉㄨㄛ-ㄑㄧˊ」卻發現本該是「多奇」的情況卻仍是「多期」。

## 根因分析

共有三層缺陷，缺一不可。

### 缺陷一（storage）：consolidateNode 不拆分多段詞

`consolidateNode` 的 `preConsolidate` 參數由 `prefs.consolidateContextOnCandidateSelection` 控制。當該偏好為 `false` 時，`consolidateCursorContext` 不會被呼叫，多段詞「多期」保持完整。

隨後 `overrideCandidate(theCandidate, at: ...)` 嘗試以 keyArray `["ㄑㄧˊ"]` 覆寫，但該節點 `allActualKeyArraysCached` 僅含 `[["ㄉㄨㄛ", "ㄑㄧˊ"]]`，不包含 `["ㄑㄧˊ"]`，導致全部 4 次重試均拋出 `.nothingOverriddenAtNode`。`consolidateNode` 提前 `return`，POM 記憶從未被寫入。

**實證**：覆寫後的 override-model-data 檔案內容為空陣列。

### 缺陷二（retrieval matching）：alternateKeys 多段詞→split 金鑰不匹配

即使缺陷一被修復、記憶成功寫入，查詢時仍會因金鑰結構不同而匹配失敗。

**記憶鍵**（拆分後）：`()&(ㄉㄨㄛ,多)&(ㄑㄧˊ,奇)` — head 為單段詞 `ㄑㄧˊ`，有 previous context
**查詢鍵**（多段詞）：`()&()&(ㄉㄨㄛ-ㄑㄧˊ,多期)` — head 為多段詞 `ㄉㄨㄛ-ㄑㄧˊ`，無 previous context

`matchesPrimaryHead` guard 在查詢鍵 previous 為 nil 時錯誤拒絕匹配。

### 缺陷三（retrieval apply）：retrievePOMSuggestions cursor 位置錯誤

即使缺陷二被修復、`fetchSuggestion` 正確返回建議，`overrideCursor` 指向多段詞起點（0），但建議的單段詞「奇」應作用於位置 1。`overrideCandidateLiteral("奇", at: 0)` 會因多段詞 gram value 為「多期」而失敗。

## Phase 72 手術方案（實作規格）

### 一、根因分析

#### 1.1 缺陷 A：Tooltip 不顯示的根因

Phase 62 將 `ofMarking()` 工廠方法從 Shared protocol extension 上移至 MainAssembly 的 concrete `IMEState`，並在其中呼叫 `generateTooltipForMarking()` 來產生 tooltip 文字。

**資料流**：

```
使用者 Shift+Left/Right → handleMarkingState()
  → State.ofMarking(displayTextSegments:, markedReadings:, cursor:, marker:)
    → generateTooltipForMarking()
      → userPhraseKVPair.value  ← rawDisplayedText.map(\.description)[markedRange].joined()
      → markedTargetExists      ← LMMgr.checkIfPhrasePairExists()
      → markedTargetIsCurrentlyFiltered ← LMMgr.checkIfPhrasePairIsFiltered()
      → readingThreadForDisplay
  → session.switchState(marking)
    → showTooltip(state.tooltip, colorState:, duration: state.tooltipDuration)
```

**根因**：`generateTooltipForMarking()` 永遠會返回非空 tooltip 字串（除非 `markedRange.isEmpty`）。但觀察 `handleMarkingState` 中 Shift+Left/Right 的 call site：

```swift
// InputHandler_HandleStates.swift (兩個倉庫)
var marking = State.ofMarking(
  displayTextSegments: state.displayTextSegments,
  markedReadings: Array(assembler.actualKeys[currentMarkedRange()]),
  cursor: convertCursorForDisplay(assembler.cursor),
  marker: convertCursorForDisplay(assembler.marker)
)
marking.data.rawDisplayTextSegments = state.data.rawDisplayTextSegments
marking.tooltipBackupForInputting = state.tooltipBackupForInputting
session.switchState(marking.markedRange.isEmpty ? marking.convertedToInputting : marking)
```

注意最後一行：當 `marking.markedRange.isEmpty` 時，實際切換的是 `marking.convertedToInputting`（而非 marking 本身）。`convertedToInputting` 的 tooltip 被設為 `data.tooltipBackupForInputting`（即前一個狀態的 tooltip，通常是空字串），**而非** `generateTooltipForMarking()` 產生的 tooltip。

但更關鍵的是：`ofMarking()` 內部呼叫 `generateTooltipForMarking()` 時，`rawDisplayTextSegments` 尚未賦值（在 `ofMarking()` 返回後才賦值）。`userPhraseKVPair` 使用 `rawDisplayedText`，其 fallback 為 `displayedText`。在 BPMFVS 投影啟用時，`displayedText` 可能已經過注音模糊比對投影處理，導致 value 內容與使用者實際標記的文字不一致。然而這僅影響 tooltip 顯示的 value 欄位、不影響 tooltip 是否顯示。

**真正導致 tooltip 不顯示的根因**，經排查確認，`ofMarking()` 在 MainAssembly 的實作中確實正確產生 tooltip 字串，且 `switchState()` 的 `showTooltip(state.tooltip, ...)` 也確實會顯示非空 tooltip（duration=0 表示永久顯示）。因此，若 tooltip 仍不顯示，最可能的原因是 `generateTooltipForMarking()` 在特定條件下返回了空字串（如 `markedRange.isEmpty`），或者 tooltip 被後續的狀態切換隱藏。

查 `InputSession_HandleEvent.swift`（兩個倉庫），對於未被處理的按鍵：

```swift
// InputSession_HandleEvent.swift
if !result {
  // 除非是 .ofMarking 狀態，否則讓某些不用去抓的按鍵起到「取消工具提示」的作用。
  if [.ofEmpty].contains(state.type) { ui?.tooltipUI?.hide() }
}
```

`.ofMarking` 狀態不會觸發 tooltip 隱藏，所以非 marking 狀態下的鍵不會干擾。但若 marking 狀態的 `markedRange` 為空（cursor == marker），則 `generateTooltipForMarking()` 返回空字串，`showTooltip()` 會改為呼叫 `hide()`。

**總結缺陷 A 根因**：需要在 `handleMarkingState` 的 Shift+Left/Right call site 確認 `convertCursorForDisplay()` 的返回值是否可能導致 cursor == marker 的情況。若 cursor == marker，`markedRange` 為空 → tooltip 為空 → `showTooltip` 隱藏。

#### 1.2 缺陷 B：加詞 Hang 的根因

Enter 在 marking state 的完整呼叫鏈：

```
handleMarkingState(input: Enter) (InputHandler_HandleStates.swift)
  ├─ state.isMarkedLengthValid
  ├─ state.markedTargetIsCurrentlyFiltered
  │    └─ LMMgr.checkIfPhrasePairIsFiltered() → mode.langModel.isPairFiltered(...)
  ├─ session.performUserPhraseOperation(addToFilter: false)
  │    ├─ LMMgr.writeUserPhrasesAtOnce(userPhrase, areWeFiltering: false)
  │    │    └─ LMAssembly.withFileHandleQueueSync {       ← sync on serial queue
  │    │         ├─ userPhrase.write(toFilter: false)      ← FileHandle I/O
  │    │         │    └─ withFileHandleQueueSync {          ← re-entrant OK
  │    │         │         └─ FileHandle seek + write
  │    │         │       }
  │    │         ├─ userPhrase.crossConverted.write(...)   ← 同上，cross mode
  │    │         └─ loadUserPhrasesData(type: .thePhrases)
  │    │              └─ mode.langModel.loadUserPhrasesData(path:, filterPath: nil)
  │    │                   └─ LMAssembly.readFileContentAsync(path:...) { content in
  │    │                        └─ asyncOnMain { self.lmUserPhrases.replaceData(textData: content) }
  │    │                      }
  │    ├─ inputHandler.updateUnigramData()
  │    ├─ inputHandler.currentLM.insertTemporaryData(...)
  │    ├─ LMMgr.bleachSpecifiedSuggestions(targets:..., mode:...)     ← 可能觸發 saveData
  │    │    └─ lxPerceptor.bleachSpecifiedSuggestions(candidateTargets:...)
  │    │         └─ saveCallback?() ?? saveData()
  │    │              └─ persistor.saveData(...)
  │    │                   └─ writeFullSnapshot(...) ← JSONEncoder + Data.write(atomic)
  │    ├─ LMMgr.bleachSpecifiedSuggestions(headReadings:..., mode:...)
  │    └─ LMMgr.bleachSpecifiedSuggestions(targets: [valueReversed], mode: .reversed)
  └─ generateStateOfInputting() → session.switchState(newState)
       └─ commit(text:) + updateCompositionBufferDisplay()
```

**根因**：三層同步阻塞累積在一起：

1. **`writeUserPhrasesAtOnce` 內的 `fileHandleQueue.sync`**：呼叫方在 main thread 上以 `fileHandleQueue.sync` 提交閉包。閉包內執行兩次 `FileHandle` 寫入（正體+反體），每次都有 `attributesOfItem` syscall + seek + read + write。對於大型語彙檔案（>1000行），`seek(toFileOffset: fileSize - 1)` + `readDataToEndOfFile()` 的量可能很大。

2. **`bleachSpecifiedSuggestions` → `lxPerceptor.saveData()` → `writeFullSnapshot()`**：每次呼叫會 JSON-encode 整個 POM 資料集並以 `.atomic` 寫入磁碟。`performUserPhraseOperation` 呼叫了三次 `bleachSpecifiedSuggestions`（targets / headReadings / valueReversed），每輪都可能觸發一次完整的 snapshot 寫入。`JSONEncoder.encode()` 對大資料集是 O(n) 的 CPU 操作，`Data.write(options: .atomic)` 則是阻塞式的磁碟 I/O。

3. **Phase 70 對 `replaceData` 的變更**：Phase 70 在 `replaceData` 開頭新增了 `replacingOccurrences(of: "\t", with: " ")`。這個操作在 `readFileContentAsync` 的回呼（`asyncOnMain` → main actor）中執行，對整個使用者片語檔案內容進行字串取代。對於大型檔案（數 MB），`replacingOccurrences` 本身是 O(n) 的 heap allocation，加上後續的 `parse(splitee: "\n")` 全文解析，在主執行緒上執行會直接阻斷 UI。

**這三層阻塞全部發生在 main thread / main actor 上**。由於 IMK 的 `handle(_:client:)` 也在 main thread 上被呼叫，因此整個輸入法（包括 client app 透過 IPC 傳入的事件）都會被阻塞，造成「連帶 client app 一起 hang」的使用者體驗。

Phase 70 之前（即 tab 未被正確處理時），async 路徑的 `replaceData` 收到的 raw data 含有 tab，`strData == rawStrData` 比較永遠為 `false`（因為 sync 路徑的 strData 已無 tab），導致每次都執行全文解析。雖然這也很慢，但因為發生在 `asyncOnMain` 的延遲回呼中，不會直接阻塞 `performUserPhraseOperation` 的 return。Phase 70 的修復讓 async 路徑和 sync 路徑的行為一致了，卻也讓原本就存在的性能瓶頸和執行緒排程問題暴露出來。

```swift
  do {
    if shouldConsolidate { ... }
    let rawStrData = try String(contentsOfFile: path, encoding: .utf8)
    // replaceData 在 fileHandleQueue 上執行，避免阻塞 main
    asyncOnMain { completion(rawStrData) }
  } catch { ... }
}
```

此變更實際上不需修改 `readFileContentAsync` 本身（它已在 `fileHandleQueue.async` 中執行）；需要改的是 completion handler 中呼叫 `replaceData` 的方式。目前 completion 透過 `asyncOnMain` 回到 main actor，然後呼叫 `replaceData`。應改為在 completion 中僅儲存處理後的資料，讓 `replaceData` 的全文解析發生在背景：

```swift
// 變更後：loadUserPhrasesData 內的 async 路徑
LMAssembly.readFileContentAsync(path: path, shouldConsolidate: ...) { [weak self] content in
  guard let self else { return }
  // 在背景執行緒執行 replaceData（全文解析）
  DispatchQueue.global(qos: .userInitiated).async {
    self.lmUserPhrases.replaceData(textData: content)
    asyncOnMain {
      self.lmUserPhrases.filePath = path
      vCLMLog("lmUserPhrases: \(self.lmUserPhrases.count) entries loaded")
    }
  }
}
```

同理適用於 `lmFiltered`、`lmUserSymbols`、`lmAssociates`、`lmReplacements` 的 async 載入路徑。

**⚠ 注意**：`LMCoreEX` 是 struct（值型別），在 `LMInstantiator` 中以 stored property 持有。由於 `LMInstantiator` 是 class（reference type），從多個執行緒對其 `lmUserPhrases` 屬性進行 mutation 會引發 data race。此變更需要確保所有對 `lmUserPhrases` 的 mutation 都在同一個 serial queue 上執行（如既有的 `fileHandleQueue`），或改用 actor 隔離。

**簡化方案**：由於 `replaceData` 的全文解析是純 CPU 操作（不涉及 I/O），可將其保留在 `asyncOnMain` 回呼中，但改為 chunked 處理以避免長時間阻塞 run loop。不過更安全的做法是利用既有的 `fileHandleQueue`：

```swift
// LMInstantiator.loadUserPhrasesData 變更後
LMAssembly.readFileContentAsync(path: path, shouldConsolidate: ...) { [weak self] content in
  guard let self else { return }
  self.lmUserPhrases.clear()
  // 將 replaceData 也放到 fileHandleQueue，確保與其他 mutation 序列化
  LMAssembly.withFileHandleQueueSync {
    self.lmUserPhrases.replaceData(textData: content)
  }
  self.lmUserPhrases.filePath = path
}
```

**3.2.3 `lmPerceptionPersistor.swift`（兩個倉庫的 LMAssembly）— `saveData` / `bleachSpecifiedSuggestions`**：

確保 `saveData()` 和 `writeFullSnapshot()` 的 JSON 編碼與磁碟寫入發生在背景執行緒。`PerceptionPersistor` 已經是 `nonisolated`，其內部方法可從任意執行緒呼叫。但 `LXPerceptor.saveData()` 目前被 `bleachSpecifiedSuggestions` 的 `saveCallback?() ?? saveData()` 間接觸發。由於 3.2.1 已將 bleach 延遲至 `asyncOnMain`，若 POM 資料集不大，JSON 編碼在 main actor 上的延遲可以接受。但為保險起見，可在 `Persistor.saveData()` 內部將 JSON 編碼 dispatch 至 global queue。

**3.2.4 確保 `writeUserPhrasesAtOnce` 不阻塞 main thread**：

目前 `writeUserPhrasesAtOnce` 內部使用 `withFileHandleQueueSync`（sync）。呼叫方 `performUserPhraseOperation` 在 main thread 上，因此 sync 會阻塞 main thread 等待 fileHandleQueue 完成。

變更：將 `writeUserPhrasesAtOnce` 改為 async 介面，或至少在 `performUserPhraseOperation` 中將其 dispatch 至背景：

```swift
// 變更後：performUserPhraseOperation 中
var succeeded = true
let semaphore = DispatchSemaphore(value: 0)
LMAssembly.fileHandleQueue.async {
  LMAssembly.withFileHandleQueueSync {
    let resultA = userPhrase.write(toFilter: areWeFiltering)
    let resultB = userPhrase.crossConverted.write(toFilter: areWeFiltering)
    guard resultA, resultB else {
      succeeded = false
      semaphore.signal()
      return
    }
    if PrefMgr.shared.phraseEditorAutoReloadExternalModifications {
      Broadcaster.shared.postEventForReloadingPhraseEditor()
    }
    loadUserPhrasesData(type: areWeFiltering ? .theFilter : .thePhrases)
    semaphore.signal()
  }
}
semaphore.wait()
if !succeeded { return false }
```

然而 `semaphore.wait()` 仍然會阻塞 main thread。真正的解決方案是將整個 `performUserPhraseOperation` 之後的流程改為 callback-based 或 async/await。但這對現有架構改動過大。

**權衡方案**：`performUserPhraseOperation` 的主要耗時源自 bleach（已在 3.2.1 延遲），而 `FileHandle` 的 seek+write 通常在毫秒級完成。真正造成 hang 的是 Phase 70 後 `replaceData` 在主執行緒上的全文解析 + 三次 bleach 的 JSON 編碼 + 磁碟寫入。若修復了 3.2.1（延遲 bleach）和 3.2.2（replaceData 移出 main actor），`writeUserPhrasesAtOnce` 內的 FileHandle I/O 應不再構成可感知的 hang。因此 `writeUserPhrasesAtOnce` 可保持不變，除非實測後仍見阻塞。

## Phase 73 手術方案（實作規格）

### 一、根因分析

#### 1.1 缺陷 A：條件式子語言模組清除缺口（Conditional Sub-LM Purge Gap）

**資料流**：

```
使用者更改目錄 → Settings UI 設定 userDataFolderSpecified
  → AppDelegate.updateDirectoryMonitorPath()
    → reloadOnFolderChangeHappens(forced: true)
      → LMMgr.initUserLangModels()
        → LMMgr.loadUserPhrasesData()  // type=nil
          → for each mode:
            → mode.langModel.loadUserPhrasesData(path:filterPath:)  // ✅ 清除 lmUserPhrases + lmFiltered
            → mode.langModel.loadUserSymbolData(path:)              // ✅ 清除 lmUserSymbols
          → if associatedPhrasesEnabled: loadUserAssociatesData()   // ⚠ 條件式
          → if phraseReplacementEnabled: loadUserPhraseReplacement() // ⚠ 條件式
```

**根因**：`LMMgr.loadUserPhrasesData(type: nil)` 對 `lmAssociates` 與 `lmReplacements` 的清除／重載是條件式的——僅在對應偏好旗標（`associatedPhrasesEnabled`、`phraseReplacementEnabled`）**當前為 true** 時才執行。

若使用者的操作歷程為：
1. 在舊目錄時，關聯詞語功能啟用 → `lmAssociates` 載入了舊目錄的 `associatedPhrases-cht.txt`。
2. 使用者切換至新目錄，且此次關聯詞語功能**已停用**。
3. `initUserLangModels()` → `loadUserPhrasesData(type: nil)` 因 `associatedPhrasesEnabled == false` 跳過 `loadUserAssociatesData()`。
4. **結果**：`lmAssociates` 仍持有舊目錄的資料，未被清除。

同理適用於 `lmReplacements`（語彙置換）。

#### 1.2 缺陷 B：檔案讀取失敗時略過 `clear()`

在 `LMInstantiator.loadUserPhrasesData(path:filterPath:)` 中：

```swift
// 現有程式碼（兩個倉庫）
public func loadUserPhrasesData(path: String, filterPath: String?) {
  func loadMain() {
    if FileManager.default.isReadableFile(atPath: path) {
      lmUserPhrases.clear()    // ← clear() 在 if 裡面
      lmUserPhrases.open(path)
    } else {
      vCLMLog("lmUserPhrases: File access failure: \(path)")
      // ⚠ 沒有任何 clear()！
    }
  }
  // ...
}
```

**根因**：`lmUserPhrases.clear()` 被包裹在 `isReadableFile` 的成功分支內。若新目錄的檔案無法讀取（權限不足、`chkUserLMFilesExist` 建立檔案失敗、目錄不存在等），舊資料**完全不會被清除**，LMInstantiator 繼續使用舊目錄的資料回應查詢。

此模式在以下五個 LMInstantiator 方法中**完全相同**：
- `loadUserPhrasesData(path:filterPath:)` → 影響 `lmUserPhrases` + `lmFiltered`
- `loadUserSymbolData(path:)` → 影響 `lmUserSymbols`
- `loadUserAssociatesData(path:)` → 影響 `lmAssociates`
- `loadReplacementsData(path:)` → 影響 `lmReplacements`
- `reloadUserFilterDirectly(path:)` → 影響 `lmFiltered`

#### 1.3 缺陷 C：缺乏統一清除入口（No Unified Purge Mechanism）

`LMInstantiator` 沒有一個集中的「清除所有使用者資料」方法。目前各個子語言模組的清除分散在各自的 `load*` 方法中，且：
- 清除與載入耦合在一起（不清除就無法載入，反之亦然）；
- 呼叫方（`LMMgr`）需要記得逐一呼叫各個 `load*` 方法；
- `unigramLRUCache` 僅在 `loadUserPhrasesData` 內清除，其他 `load*` 方法（`loadUserSymbolData`、`loadUserAssociatesData`、`loadReplacementsData`）**不會**清除 LRU cache；
- `inputTokenHashesArray` 完全不會被清除。

#### 1.4 缺陷 D：`LMCoreEX.open()` 的 `isLoaded` 提前返回

```swift
// LMCoreEX.open()（兩個倉庫）
mutating func open(_ path: String) -> Bool {
  if isLoaded { return false }  // ← 若已載入則不做事
  // ...
}
```

若呼叫方忘記先呼叫 `clear()`（或 `clear()` 因 1.2 的條件式略過而被跳過），則 `open()` 會因 `isLoaded == true` 直接返回 `false`，連新資料都不會載入。此 defect 是 1.2 的放大器。

**3.3.1 `LMMgr_Utilities.swift`（兩個倉庫）— 新增 `migrateUserDataFrom(oldPath:to:)`**：

```swift
/// 將舊目錄的使用者片語檔案內容合併至新目錄的對應檔案。
/// 合併策略：append 模式（新目錄檔案內容在前，舊目錄內容追加在後）。
/// - Parameters:
///   - oldPath: 舊的使用者資料目錄路徑。
///   - newPath: 新的使用者資料目錄路徑。
/// - Returns: 成功合併的檔案數量。
@discardableResult
public static func migrateUserDataFrom(oldPath: String, to newPath: String) -> Int {
  var migratedCount = 0
  for type in LMAssembly.ReplacableUserDataType.allCases {
    let oldFileURL = URL(fileURLWithPath: oldPath).appendingPathComponent(
      fileName(for: type, mode: .imeModeCHT).dropLast("-cht.txt".count).description
    )
    for mode in Shared.InputMode.validCases {
      let oldURL = userDictDataURL(mode: mode, type: type, basePath: oldPath)
      let newURL = userDictDataURL(mode: mode, type: type, basePath: newPath)
      guard FileManager.default.isReadableFile(atPath: oldURL.path),
            FileManager.default.isReadableFile(atPath: newURL.path) else { continue }
      do {
        let oldData = try String(contentsOf: oldURL, encoding: .utf8)
        let newData = try String(contentsOf: newURL, encoding: .utf8)
        let merged = newData + "\n" + oldData
        try merged.write(to: newURL, atomically: true, encoding: .utf8)
        migratedCount += 1
      } catch {
        vCLMLog("Failed to migrate \(type) for \(mode): \(error)")
      }
    }
  }
  return migratedCount
}
```

### 四、影響範圍與注意事項

1. **`LMCoreEX.open()` 行為變更**：原本若已載入資料則不做事（caller 必須先 `clear()`）。變更後 `open()` 會自動清除後再載入。對現有所有 call site 無負面影響（現有 call site 皆已在呼叫 `open()` 前呼叫 `clear()`，`clear()` 的重複呼叫是冪等的）。

2. **`purgeUserData()` 不處理 POM 資料**：`lxPerceptor`（POM）的資料儲存位置使用 `isDefaultFolder: true`，不受使用者自訂目錄影響。因此 `purgeUserData()` 不包含 POM 清除。

3. **`purgeUserData()` 不處理原廠辭典**：`factoryTrie` 是原廠辭典（TextMap），與使用者目錄無關，不應清除。

4. **`LMCoreEX` 的 value type 特性**：`LMCoreEX` 是 struct（值型別），`clear()` 會完全重置其內部狀態。由於它被 `LMInstantiator`（class）持有，`clear()` 的 mutation 發生在 `LMInstantiator` 的 stored property 上。現有所有 mutation 皆在 main actor 或 `fileHandleQueue` 上進行。`purgeUserData()` 應在 main actor 上呼叫。

5. **向後相容性**：`updateDirectoryMonitorPath()` 的簽章變更（新增選擇性參數 `oldPath`）對現有 Swift 呼叫者為 source-compatible（選擇性參數預設 nil）。

6. **多 InputMode 的 purge 順序**：`initUserLangModels()` 先對所有 mode 執行 `purgeUserData()`，再逐一 `chkUserLMFilesExist`，最後載入。這個順序確保即使某個 mode 的 `chkUserLMFilesExist` 失敗，其他 mode 也不會殘留舊資料。

# Phase 75

## 手術範圍

本次手術僅限 `vChewing-macOS` 倉庫的 **SettingsUI**（SwiftUI，macOS 14+）。

**SettingsCocoa（AppKit）暫不處理**，`vChewing-OSX-Legacy` 倉庫亦不在本次手術範圍內。相關方案的 OS 版本要求統一調整為 macOS 14+。

## 架構決策

- **系統需求**：macOS 14+（使用 `@Observable` macro 取代舊式 `@ObservableObject`，解決 Combine-based view model 顆粒度不足的問題）。
- **共用 ViewModel**：所有 SettingsUI page 共用一個 `@Observable class SettingsUIViewModel`，包含搜尋狀態、導航狀態、以及 UserDef→Tab 映射的收集與查詢。此 ViewModel 經由 `.environmentObject()` 注入整個 View 樹。

## 原始需求

> 這次先不急著談具體的手術，而是先搞清楚解決方案。

唯音輸入法的偏好設定，無論 SettingsCocoa 還是 SettingsUI，都對電腦文盲而言太過於複雜：幾乎不可能第一時間定位到自己想要改的設定。

Shiki Suen 推測的幾個解決視角：

1. 對 SettingsUI 實作類似於 macOS Settings 那樣的「可檢索」特性。似乎 macOS Preferences (macOS 12 and earlier) 也有這個特性，可以參照著實作給 SettingsCocoa。但實作成本無法估測。
2. 單獨弄一個 View 允許使用者檢索選項。檢索結果以文字的形式告知使用者「要找的結果在哪一頁」。
3. 使用額外的工具對 SettingsCocoa 的 Swift 程式碼做靜態分析、自動讀取 UserDef 以及相關的 i18n 檔案，然後生成「可檢索的」HTML 說明手冊。該手冊可在建置輸入法的時候惰性更新，生成的 HTML Asset 綁定在輸入法內，使用者點了之後會自動叫出 Safari 顯示該說明頁面。這個方法的技術實現工具不一定非得是 Swift，也可以是 C# (.NET 10)。

前兩個特性，尤其是第二個特性，需要在 SettingsCocoa / SettingsUI 以 SwiftUI Mirror 等特性捕撈一個資訊：每個頁面究竟牽涉了哪些 UserDefs 的顯示。檢索用的資料叢集可以從 UserDef 的 Metadata 生成。

## Phase 75 可行性研究

> 以下研究基於對 `vChewing-macOS` 倉庫 SettingsUI 的實地原始碼審查。研究日期：2026-05-21。SettingsCocoa 與 Legacy 倉庫暫不納入本次手術範圍。

### 一、現狀基線

#### 1.1 資料規模

| 指標 | 數值 |

| `PrefUITabs` 分頁數 | **12**（About / General / Candidates / Behavior / Output / Dictionary / Phrases / Cassette / Keyboard / Clients / Services / DevZone） |

#### 1.2 UserDef→Tab 映射現狀

**映射關係僅隱式存在於各 Pane 的原始碼中**，沒有集中式的宣告機制。例如：

- `VwrSettingsPaneCocoaGeneral.swift` 呼叫 `UserDef.kAppleLanguages.render(...)`、`UserDef.kReadingNarrationCoverage.render(...)` 等
- `VwrSettingsPaneCocoaBehavior.swift` 呼叫 `UserDef.kSpecifyShiftBackSpaceKeyBehavior.render(...)` 等
- `VwrSettingsPaneCocoaCandidates.swift` 呼叫 `UserDef.kUseHorizontalCandidateList.render(...)` 等

**結論**：目前沒有任何執行期或編譯期機制能回答「UserDef X 出現在哪個分頁」。這是方案一與方案二的共同前置條件。

##### 多分頁歸屬檢查

經由對全部 12 個 SettingsUI Pane 檔案的交叉比對，**目前所有 109 個 UserDef 均只出現在單一分頁**，不存在同一 UserDef 出現在多個分頁的情況。但映射表的資料結構仍應設計為 `UserDef → [Tab]`（一對多陣列）而非 `UserDef → Tab`（一對一），以備未來重構時某個 UserDef 被多個分頁引用。

##### i18n 解析機制

`.i18n` 的實作位於 `SwiftExtension/SwiftFoundationImpl.swift`：

```
extension StringLiteralType {
  public var i18n: String { NSLocalizedString(description, comment: "") }
}
```

即 `"i18n:UserDef.kAppleLanguages.shortTitle".i18n` 等價於 `NSLocalizedString("i18n:UserDef.kAppleLanguages.shortTitle", comment: "")`，直接從 app bundle 的 `Localizable.strings` 查表。

對搜尋功能的影響：

- **方案一／二（執行期搜尋）**：直接呼叫 `metaData.shortTitle?.i18n`，無需額外 parser——`.i18n` 底層就是 `NSLocalizedString`，當前 IME process 的 bundle 上下文自然正確。
- **方案三（Build-time HTML 生成）**：無法在 build phase script 中依賴 `NSLocalizedString`（因為 executable 不是 IME bundle），需自行 parse `SharedUI/*.lproj/Localizable.strings` 建立 key→value 映射（~550 keys，簡單的 `"key" = "value";` 格式，1–2h）。

#### 1.3 渲染架構（SettingsUI，macOS 14+）

| 層級 | 實作 |
|------|------|
| 容器 | `CtlSettingsUI` (`NSWindowController` + `NSHostingView`) |
| 導覽 | `NavigationSplitView` + `List`（`PrefUITabs.allCases`） |
| Pane 基底 | `View`（`Form` / `Section`，宣告式） |
| 控件生成 | `UserDefRenderableImpl.renderUI()` / `.bind($x).render()` → `some View` |
| 資料繫結 | `@AppStorage` wrapper（直接讀寫 `UserDefaults.standard`） |

### 二、方案一分析：內建搜尋

#### 2.1 方案描述

在現有偏好設定視窗內嵌入搜尋欄，使用者輸入關鍵字後即時過濾／高亮匹配的設定項目，類似 macOS System Settings（Ventura+）或 System Preferences（Monterey-）的行為。

#### 2.2 SettingsUI（SwiftUI）端可行性

**技術路徑**：

macOS 14+ 的 SwiftUI `NavigationSplitView` 搭配 `.searchable()` modifier 和 `.searchSuggestions()` 可實現內建搜尋。核心思路是：

1. 在 `VwrSettingsUI` 上掛 `.searchable(text: $searchText)`。
2. 建立一個 compile-time 或 build-time 生成的 `[UserDef: PrefUITabs]` 映射表。
3. 搜尋時，對每個 UserDef 的 `MetaData` 欄位（`shortTitle`、`description`）做 i18n 解析後的字串比對。
4. 顯示搜尋建議列表；選擇某一項後自動跳轉到對應分頁並高亮／捲動到該控件。

**可行性評估**：🟢 **技術上可行**

- `.searchable()` 在 macOS 14+ `NavigationSplitView` 中已成熟。
- `LocalizedStringKey` / `String.i18n` 可在執行期解析。
- `PrefUITabs.suiView` 已經提供分頁→View 的對應。

**風險與限制**：

| 風險 | 說明 |
|------|------|
| macOS 14+ 限定 | SettingsUI 本身已是 macOS 14+ 限定，但設定視窗有 `CtlSettingsCocoa`（macOS 10.9+）作為 fallback——後者無法受益 |
| 目標控件捲動定位 | SwiftUI 的 `ScrollViewReader` + `scrollTo(_:anchor:)` 需要每個設定項有 stable `id`，而目前的 render 框架未統一暴露 `id` |
| 搜尋延遲 | 若每個按鍵都對 109 個 UserDef 做 i18n 解析，效能無虞（總計 < 1ms），但若未來 UserDef 擴增至數百個可能需要 debounce |
| 特殊控件搜尋盲區 | 部分自訂控件（如 `ComboBox`、`NSPathControl` wrapper、`NSPopUpButton` 手動注入）無法透過 `MetaData` 靜態描述其行為，搜尋可能出現 false negative |

**實作成本估算**：

| 工作項 | 估計工時 | 說明 |
|--------|---------|------|
| UserDef→Tab 映射表 | 2–3h | 見第五節（共同前提） |
| 搜尋邏輯（fuzzy match + i18n resolve） | 2–3h | 對 `MetaData.shortTitle` / `.description` 做 localized string 比對 |
| `VwrSettingsUI` 改造（`.searchable` + 導航） | 3–4h | 加入 `@State searchText`、`NavigationLink` programmatic navigation、搜尋建議 UI |
| 個別 Pane 的 `scrollTo` 支援 | 3–5h | 為每個 UserDef 控件分配 stable `id`（需要修改 `UserDefRenderable` 與各 Pane 的 render call site） |
| 測試與多語系 QA | 3–4h | 確保四語系搜尋結果一致 |
| **合計（SettingsUI 端）** | **13–19h** | |

> **SettingsCocoa 端已擱置**，不納入本次 Phase 75 手術範圍。

### 三、方案二分析：獨立搜尋 View

#### 3.1 方案描述

單獨實作一個搜尋 Sheet，使用者輸入關鍵字，搜尋結果以文字列表顯示「設定項名稱 → 所在分頁名稱」。點選結果後關閉搜尋視窗並自動跳轉到對應分頁。

#### 3.2 可行性評估

🟢 **高度可行，實作成本最低**

- 無需改造現有 Pane 的內部結構。
- 搜尋 View 是完全獨立的新組件，架構乾淨、對現有程式碼侵入性最小。
- 結果僅需知道「UserDef → Tab」映射和 Tab 名稱的 i18n，**不需要控件定位**（只需跳轉到正確分頁，讓使用者自己在該頁找）。
- 可經由 `SettingsUIViewModel`（共用 ViewModel）直接取得映射資料，無需額外資料層。

**降級策略**：若不實作自動跳轉（僅告知使用者「在某某分頁」），成本更可降至近乎僅有 ViewModel 的建置。

**風險與限制**：

| 風險 | 說明 |
|------|------|
| 跳轉後仍需使用者自行尋找 | 搜尋只能帶到「分頁 + Section」層級，進入該 Section 後使用者仍需肉眼掃描數個選項。Section 級別導航可將搜尋範圍從整頁（~15–30 個選項）縮小至單一 Section（~3–8 個選項），大幅降低掃描負擔 |
| 多餘一步操作 | 使用者需要知道「有搜尋功能存在」並主動觸發它（熱鍵 ⌘F 或選單項），不如內建搜尋欄直覺。可透過首次開啟偏好設定時的 tooltip 提示來緩解 |

**實作成本估算**：

| 工作項 | 估計工時 | 說明 |
|--------|---------|------|
| `SettingsUIViewModel`（含 collector 映射收集） | 2–3h | 見 §5.4（路徑 D + ViewModel） |
| 搜尋核心（fuzzy match + i18n resolve） | 2–3h | 對 `MetaData.shortTitle` / `.description` 做 localized 比對 |
| 搜尋 Sheet UI | 2–3h | 一個 TextField + 結果 List，極簡 |
| 觸發入口（選單項「搜尋偏好設定…」+ 熱鍵 ⌘F） | 1h | |
| 跳轉邏輯（`NavigationSplitView` programmatic navigation） | 1h | 透過 ViewModel 設定 `selectedTabID` |
| 測試與多語系 QA | 2h | |
| **合計** | **10–13h** | |

### 四、方案三分析：靜態 HTML 說明手冊

#### 4.1 方案描述

以外部工具（Swift 或 C#/.NET 10）對原始碼做靜態分析，自動擷取：

1. 各 `VwrSettingsPane*` 檔案中引用了哪些 `UserDef`。
2. 從 `UserDef.swift` 讀取每個 `UserDef` 的 `MetaData`（含 `shortTitle`、`description` 等 i18n key）。
3. 從 `.strings` 檔案讀取四語系翻譯。

然後生成一個靜態 HTML 頁面，內含：
- 以分頁為章節、每個設定項為條目的結構化說明。
- 瀏覽器內建搜尋（Ctrl+F / ⌘F）即可檢索。
- 可包含截圖（由人工維護）。

HTML 作為 build phase 產物嵌入 `Bundle`，使用者從選單點選「說明手冊」後以 Safari 開啟。

#### 4.2 可行性評估

🟡 **可行，但投入產出比需仔細權衡**

**技術路徑選擇**：

| 工具 | 優點 | 缺點 |
|------|------|------|
| **Swift（同一生態系）** | 零外部依賴、可直接 import 專案 module 做 type reflection、build phase script 即可整合 | 需要寫一個獨立的 executable target（或 SPM plugin） |
| **C# (.NET 10)** | 正則表達式／字串處理較 Swift 精簡、LINQ 適合資料轉換、跨平台 | 需要 .NET runtime 在建置機器上、與 Xcode build phase 整合較麻煩、團隊需維護兩套語言 |

**建議選擇 Swift**：原因如下：

1. `UserDef` 是 Swift `enum`，Swift 工具可以直接 `import Shared` 並 iterate `UserDef.allCases` 取得完整定義。C# 只能靠 regex 解析原始碼——對 enum case 定義還算可行，但對 `MetaData` 的 switch-case 映射（~500 行）做 regex 解析不可靠且脆弱。
2. vChewing 的 macOS CI 環境已具備 Swift toolchain，無需額外安裝 .NET。
3. Build phase script 直接 `swift run` 即可。

**靜態分析的兩種路徑**：

| 路徑 | 做法 | 可靠度 |
|------|------|--------|
| **A. 編譯期（import module）** | 寫一個 Swift executable，`import Shared`，呼叫 `UserDef.allCases` + `.metaData` + `.dataType`，直接拿到完整結構化資料 | 🟢 100% 可靠 |
| **B. Regex 原始碼掃描** | 對 `VwrSettingsPane*` 檔案做 regex `/UserDef\.(\w+)/`，擷取各 Pane 引用的 UserDef | 🟡 正則無法處理 switch case 中動態組裝的 UserDef 引用；但 vChewing 的 Pane 程式碼風格高度一致（一律 `UserDef.kXXX.render(...)`），regex 匹配率預估 > 95% |

**混合方案**：

- UserDef 定義與 MetaData：用路徑 A（import module，100% 可靠）。
- UserDef→Tab 映射：用路徑 B（regex 掃描 Pane 檔案，對風格一致的程式碼基本可靠，但邊界 case 需人工複核）。
- i18n 文案：從 `SharedUI/*.lproj/Localizable.strings` 直接讀取 key-value，做字串替換。

**風險與限制**：

| 風險 | 說明 |
|------|------|
| 維護成本 | 每當有新的 UserDef 或分頁調整，HTML 需要重新生成——但作為 build phase 自動化，這不是問題 |
| 說明手冊缺乏互動性 | 無法跳轉到實際設定頁面——使用者看完手冊後仍需自行找到對應分頁 |
| Safari 跳出 | 使用者體驗中斷（從設定視窗跳到 Safari）。但可降級為「內嵌 WebView 在設定視窗內顯示」來解決 |
| 截圖維護 | 截圖屬於人工維護內容，隨著 UI 變更會過期 |

**實作成本估算**：

| 工作項 | 估計工時 | 說明 |
|--------|---------|------|
| Swift executable target（`VCGenSettingsManual`） | 1–2h | Package.swift 新增 executable product |
| UserDef 定義讀取（`import Shared` + iterate `allCases`） | 1h | |
| UserDef→Tab 映射 regex 掃描 | 2–3h | 掃描全部 24 個 Pane 檔案（macOS × 12 + Legacy × 12） |
| `.strings` parser（四語系） | 1–2h | 簡單的 key-value 解析 |
| HTML 模板與生成 | 3–4h | 含章節導覽、搜尋框（JavaScript 前端過濾）、RWD |
| Build phase 整合 | 1–2h | Xcode Build Phase script / Makefile target |
| 內嵌 WebView 顯示（選用） | 2–3h | 若不跳出 Safari |
| 測試與多語系 QA | 2h | |
| **合計** | **13–19h** | |

### 五、共同前提：UserDef→Tab 映射表

方案一和方案二都依賴「哪個 UserDef 在哪個分頁」的結構化資料。以下為獲取該資料的幾種路徑。

#### 5.1 路徑 A：標註式（Annotation）

在每個 `UserDef` case 的 `MetaData` 中新增一個 `tab: PrefUITabs` 欄位。

**優點**：
- 100% 準確、無需 static reflection。
- 作為 Single Source of Truth，任何分頁調整都會在編譯期暴露。
- 可同步用於執行期查詢。

**缺點**：
- 需要在 ~105 個 `MetaData` switch case 中手動添加 `tab:` 欄位。
- 相當於一次全面性的 MetaData 補完工程。

**成本**：3–4h（逐一審查 109 個 UserDef 並對照 Pane 原始碼確認歸屬）。

#### 5.2 路徑 B：Build-time 靜態分析

寫一個 Swift script（或 SPM plugin）在 build phase 掃描所有 `VwrSettingsPane*` 檔案的原始碼，擷取 `UserDef.kXXX` 引用，並依據 Pane 類別名稱推斷對應的 Tab。

**優點**：
- 無需修改 UserDef 定義，對現有程式碼零侵入。
- 始終反映實際原始碼狀態。

**缺點**：
- Regex 匹配可能漏掉動態組裝的引用（極罕見，目前未發現此類 pattern）。
- 當 Pane 檔案重新命名或拆分時需要更新 script。

**成本**：2–3h（含 regex 撰寫、測試、CI 整合）。

#### 5.3 路徑 C：Mirror / Runtime Reflection

使用 Swift `Mirror(reflecting:)` 在執行期 iterate `UserDef.allCases`，但無法透過 Mirror 取得「哪個 Pane 引用了哪個 UserDef」——因為 Pane 是 View，不是資料模型。

**結論**：❌ **Mirror 在此場景不適用**。Pane 的 `body` / `loadView()` 是 computed property / method，不是 stored property。Swift Mirror 無法在執行期解析 computed property 的實作內容。

#### 5.4 路徑 D：Shared ViewModel + Runtime Registration（採用方案）

> 此路徑為 Shiki Suen 原創提案，現確定為 Phase 75 的技術路線。

**核心思路**：所有 SettingsUI page 共用一個 `@Observable class SettingsUIViewModel`。該 ViewModel 管理搜尋狀態、導航狀態、以及 UserDef→Tab 映射——後者透過 page 渲染時 `UserDef.renderUI()` 的自我申報自動收集。

**技術機制**：

1. **Collector 定義**：一個 `@Observable class SettingsUIViewModel`（macOS 14+），管理搜尋、導航、及 `userDefToTabs: [UserDef: Set<PrefUITabs>]` 映射。
2. **注入點**：`CtlSettingsUI.windowDidLoad()` 建立 ViewModel 並注入整個 View 樹：
   ```swift
   let viewModel = SettingsUIViewModel()
   window?.contentView = NSHostingView(
     rootView: VwrSettingsUI().environmentObject(viewModel)
   )
   ```
3. **Tab context**：`VwrSettingsUI.body` 在渲染 `selectedTab.suiView` 時注入當前 tab（需定義一個 custom `EnvironmentKey`，~10 行）：
   ```swift
   selectedTab.suiView
     .environment(\.currentSettingsTab, selectedTab)
   ```
   這需要定義一個 custom `EnvironmentKey`：
   ```swift
   struct CurrentSettingsTabKey: EnvironmentKey {
     static let defaultValue: PrefUITabs? = nil
   }
   extension EnvironmentValues {
     var currentSettingsTab: PrefUITabs? {
       get { self[CurrentSettingsTabKey.self] }
       set { self[CurrentSettingsTabKey.self] = newValue }
     }
   }
   ```
4. **註冊點**：`UserDef.renderUI()` → `UserDefRendered.onAppear` 向 ViewModel 登記：
   ```swift
   @Environment(\.currentSettingsTab) var currentTab
   @EnvironmentObject var viewModel: SettingsUIViewModel
   // .onAppear { if let tab = currentTab { viewModel.register(userDef, in: tab) } }
   ```

**優勢**：

| 優勢 | 說明 |
|------|------|
| 🟢 零維護 | 映射隨 Pane 程式碼自動同步 |
| 🟢 反映實際渲染 | 條件式渲染（`if #available`、`if Date.isTodayTheDate(...)` 等）自動被尊重 |
| 🟢 搜尋／導航整合 | ViewModel 同時持有映射、搜尋狀態、導航狀態，單一 `@EnvironmentObject` 搞定所有跨頁狀態 |
| 🟢 欄位級觀察 | `@Observable` 確保搜尋文字變更不會觸發整個 ViewModel 的重新觀察 |

**劣勢**：

| 劣勢 | 說明 |
|------|------|
| 🟡 需定義 `EnvironmentKey` | 一個自訂 key + extension，~10 行，成本可忽略 |
| 🟡 需修改 `renderUI()` | 在 `UserDefRendered` 的 `body` 中掛 `.onAppear` modifier（~5 行） |
| 🟡 `selectedTabID` 遷移 | `VwrSettingsUI` 的 `@State selectedTabID` 移至 ViewModel（~5 行） |

**實作成本**：

| 工作項 | 估計工時 | 說明 |
|--------|---------|------|
| `SettingsUIViewModel` class | 1h | `@Observable`，含 `register()` + `search()` |
| `EnvironmentKey` + extension | 0.5h | ~10 行 |
| `CtlSettingsUI` injection（`.environmentObject`） | 0.5h | 一行改動 |
| `VwrSettingsUI` tab context + selectedTabID 遷移 | 1h | |
| `UserDefRendered` registration（`.onAppear`） | 0.5h | ~5 行 |
| **合計** | **3.5h** | |

**路徑 D 與方案一／二／三的互動**：

- **方案一（內建搜尋）**：ViewModel 的 `search()` 直接查 `userDefToTabs`，`.searchable()` 綁定 `viewModel.searchText`。
- **方案二（獨立搜尋 Sheet）**：Sheet 以 `@EnvironmentObject var viewModel` 讀取同一實例，搜尋結果直接從 `viewModel.search()` 取得。
- **方案三（HTML 手冊）**：ViewModel 資料可 serialized 為 JSON 供 build phase 使用，但需至少執行過一次 IME。

#### 5.5 路徑對比總表

| 維度 | A（標註） | B（Regex） | **D（ViewModel）** |
|------|----------|-----------|-------------------|
| 準確度 | 100%（手動） | ~95% | **100%（實際渲染）** |
| 維護成本 | 🔴 每次新增 UserDef 需手補 | 🟡 Script 隨 Pane 改名更新 | **🟢 零維護** |
| 條件式渲染 | 🔴 需手動標註 | 🔴 無法感知 | **🟢 自動尊重** |
| 搜尋／導航整合 | 🔴 需額外 plumbing | 🔴 需額外 plumbing | **🟢 ViewModel 內建** |
| 實作成本 | 3–4h | 2–3h | **3.5h** |

#### 5.6 建議

**採用路徑 D（Shared ViewModel + Runtime Registration）作為唯一方案**。

由於 SettingsCocoa 已擱置、OS 需求鎖定 macOS 14+，路徑 D 的所有劣勢都不再是問題。路徑 A/B 僅作為 CI guard 備案——build phase regex 掃描結果可與 ViewModel 的註冊結果做 diff，偵測渲染遺漏。

> 路徑 C（Mirror）已確認不適用，不列入比較。

### 七、建議方案

#### 7.1 推薦：分階段推進

```
Phase 75A: SettingsUIViewModel + 獨立搜尋 Sheet（方案二 + 路徑 D）
    ↓ （驗證搜尋功能的使用者接受度）
Phase 75B: 完善方案一（SettingsUI 端內建搜尋）
    ↓ （視需求）
Phase 75C: 方案三 HTML 手冊
```

**Phase 75A**（建議立即執行，採用路徑 D + 方案二，估算 10–16h）:

1. **`SettingsUIViewModel`**（路徑 D）：實作 `@Observable class`，含 `register()`（`onAppear` 自動申報）+ `search()`（fuzzy match）+ `selectedTabID` 遷移至 ViewModel（3.5h）。
2. **獨立搜尋 Sheet**（方案二）：在 `VwrSettingsUI` 加 `.sheet()` + 搜尋框 + 結果列表（「設定項名稱 → 分頁名稱 + Section」），點選後自動跳轉（2–3h UI + 搜尋邏輯 2–3h + 跳轉 1h）。
3. **Section 級別導航**：若 Section 資訊對使用者體驗有顯著幫助，可在 ViewModel 註冊時同時記錄 section header，或後續以 regex 補充擷取。
4. **CI guard**：保留路徑 B（regex）作為 CI 檢查——build phase 掃描 Pane 檔案生成預期映射，與 ViewModel 的註冊結果做 diff，偵測潛在的渲染遺漏。

**為何先做方案二而非方案一**：

- 方案二的投入產出比最高：以方案一 1/3 的成本解決約 70–80% 的使用者需求（「不知道設定在哪一頁」）。
- 路徑 D（ViewModel）使方案二的建置成本進一步降低——不再需要手動維護映射表或 regex script，ViewModel 自動追蹤實際渲染。
- 由於 SettingsCocoa 已擱置、OS 鎖定 macOS 14+，所有方案均在 SettingsUI 上實現，方案二的跨 UI 劣勢不再存在。

- 若使用者回饋顯示「仍不夠直覺」，再追加方案一（`.searchable()` 內建搜尋+高亮）作為強化，且此時 ViewModel 已就緒，`.searchable()` 可直接綁定 `viewModel.searchText`。

#### 7.2 方案三的定位

方案三（HTML 手冊）與方案一／二並不互斥，而是**互補**：

- 手冊可包含**截圖**和**詳細說明**（`MetaData.description` 太簡短時尤為有價值）。

#### 1.2 Pane 預載入機制

`CtlSettingsCocoa.init()` 呼叫 `panes.preload()`，它會逐一調用全部 12 個 Pane 的 `loadView()`。這意味著：

- **所有 UserDef 的 NSView 控件都在視窗初始化時即已建立**。
- 但渲染結果是 `NSStackView` 層級結構，`UserDef.kXXX.render(fixWidth:)` 返回的是匿名 `NSView?`，無法從 view hierarchy 反向追溯對應的 `UserDef`。

#### 1.3 UserDefCocoa 渲染路徑

```
UserDef.kXXX.renderCocoa(fixWidth:prefUITab:)              // → Phase 76 新 API（位於 CtlSettingsCocoa.swift）
  └─ toCocoaRenderable() → UserDefRenderableCocoa           // NSObject, NOT NSView
       └─ .render(fixWidth:) → NSStackView                   // 匿名 NSView hierarchy
```

`UserDefRenderableCocoa` 是 `NSObject`（非 `NSView`），無法從產出的 `NSStackView` 反向找到對應的 `UserDef`。Phase 76 利用新的 `prefUITab` 參數在渲染時直接完成登記，繞過此限制。

### 二、UserDef→Tab 映射收集

#### 2.1 路徑 A：renderCocoa API 內嵌注入（✅ 採用方案）

**策略**：在 `CtlSettingsCocoa.swift` 檔案尾部為 `UserDef` 新增 extension，提供帶 `prefUITab` 參數的 `renderCocoa(fixWidth:prefUITab:extraOps:)`。當 `prefUITab` 非 nil 時，以 `asyncOnMain` 將 `(tab, userDef)` 注入 `CtlSettingsCocoa` 持有的 `userDefMap: [PrefUITabs: Set<UserDef>]`。

**API 設計**：
```swift
// 置於 CtlSettingsCocoa.swift 檔案尾部
extension UserDef {
    func renderCocoa(
        fixWidth: CGFloat? = nil,
        prefUITab: PrefUITabs? = nil,
        extraOps: ((inout UserDefRenderableCocoa) -> ())? = nil
    ) -> NSView? {
        var renderable = toCocoaRenderable()
        extraOps?(&renderable)
        if let tab = prefUITab {
            asyncOnMain {
                CtlSettingsCocoa.shared?.registerUserDef(self, in: tab)
            }
        }
        return renderable.render(fixWidth: fixWidth)
    }
}
```

**CtlSettingsCocoa 新增**：
```swift
var userDefMap: [PrefUITabs: Set<UserDef>] = [:]

func registerUserDef(_ userDef: UserDef, in tab: PrefUITabs) {
    userDefMap[tab, default: []].insert(userDef)
}
```

**Pane 端 call site**（以 General 為例）：
```swift
UserDef.kAppleLanguages.renderCocoa(fixWidth: contentWidth, prefUITab: .tabGeneral) { ... }
UserDef.kReadingNarrationCoverage.renderCocoa(fixWidth: contentWidth, prefUITab: .tabGeneral) { ... }
```

**優勢**：
- 🟢 100% 準確：實際呼叫的 UserDef 才會被登記，條件式渲染自動尊重。
- 🟢 零維護：Pane 新增／移除 UserDef 時自動跟隨。
- 🟢 macOS 10.9 相容：`asyncOnMain` 是 vChewing 內部已有的 helper，無 Swift Concurrency 依賴。
- 🟢 無模組反向依賴：extension 直接寫在 `CtlSettingsCocoa.swift`（`MainAssembly4Darwin` 模組內），不修改 `Shared_DarwinImpl`。
- 🟢 按需注入：只有傳入 `prefUITab` 的 call site 才觸發登記；既有不帶 tab 參數的 `.renderCocoa(fixWidth:)` 不受影響。

**劣勢**：
- 🟡 需修改全部 12 個 Pane 內的 `renderCocoa` call site（每個 `UserDef.kXXX.renderCocoa(fixWidth:)` → `.renderCocoa(fixWidth:prefUITab:.tabXXX)`）。
  - 但這是機械式替換（regex 一鍵完成），實際工時 < 1h。

**成本**：2–3h（含 extension 撰寫、map 管理、12-pane call site 更新、Legacy 同步）。

### 二、六大瓶頸詳析

#### 🔴 瓶頸 #1（最嚴重）：`buildReverseLookupTable` — 每個中文字都產生一個 `String`

**位置**：`collectCharsFromPlainCellBytes` (L642) / `collectCharsFromGroupedCellBytes` (L690)

```swift
let scalar = decodeUTF8Scalar(from: buffer, at: i, length: len)
if !ideographicOnly || scalar.properties.isIdeographic {
    result.append(String(scalar))  // ← 每個字元一個 String 分配！
}
```

然後在 `buildReverseLookupTable` (L451) 中以 `[String: [Int]]` dictionary 儲存：

```swift
charToLineIndices[char, default: []].append(lineIndex)
```

**問題規模**：一個百萬行的辭典，平均每行 2 個中文字，就是兩百萬次 `String(scalar)` 調用。每個 `String` 都是堆積分配，且在 `Dictionary<String, [Int]>` 中被 retained 作為 key。

**在 Intel 上為何更痛**：Intel Mac 的 `malloc` 效率遠不如 Apple Silicon（後者使用 nanov2 記憶體分配器），同樣數量的堆積分配在 Intel 上的耗時可以是 M 系列的 3-5 倍。

**修復方案**：全程使用 `UInt32` scalar value 作為字典 key，只在最後建 `RevLookupEntry` 時才轉換為 UTF-8 bytes。

```swift
// Before: [String: [Int]]
// After:  [UInt32: [Int]]
var charToLineIndices: [UInt32: [Int]] = [:]

// 收集時直接用 scalar value
let scalarValue = decodeUTF8ScalarValue(from: buffer, at: i, length: len)
charToLineIndices[scalarValue, default: []].append(lineIndex)

// 只在最終建 RevLookupEntry 時才轉 UTF-8
for (scalarValue, lineIndices) in charToLineIndices {
    let charString = String(Unicode.Scalar(scalarValue)!)
    result.append(.init(key: ContiguousArray(charString.utf8), ...))
}
```

**預估改善**：減少 ~200 萬次 String 分配，**25-35%**。

#### 🔴 瓶頸 #2：`parseKeyLineMapContent` 第二段 — 每個 key entry 多次 String 分配

**位置**：L374-391

```swift
for (nodeID, idx) in entries.indices.enumerated() {
    let readingKey = extractString(from: data, start: keyEntry.keyStart, end: keyEntry.keyEnd)
    let keyInitials = readingKey.split(separator: separator).compactMap {
        $0.first?.description          // ← 每個 segment 一個 String
    }.joined()
    keyInitialsIDMap[keyInitials, default: []].append(nodeID)
    let segCount = readingKey.split(separator: separator).count  // ← 又 split 一次
    entries[idx] = .init(..., segmentCount: UInt8(clamping: segCount))
}
```

**問題規模**：假設 10 萬個 key entry、每個平均 3 個 segment：

| 操作 | 次數 |
|------|------|
| `extractString` | 10 萬次 String 分配 |
| `.split`（第一次） | 10 萬次 `[Substring]` 分配 |
| `.compactMap { .description }` | 30 萬次 String 分配 |
| `.joined()` | 10 萬次 String 分配 |
| `.split`（第二次） | 10 萬次 `[Substring]` 分配 |
| **總計** | **約 70 萬次堆積分配** |

**修復方案**：直接在 raw bytes 上掃描，不經過任何 String API：

```swift
data.withUnsafeBytes { rawBuffer in
    let buffer = rawBuffer.bindMemory(to: UInt8.self)
    let sep = separator.asciiValue!
    for (nodeID, idx) in entries.indices.enumerated() {
        let ks = entries[idx].keyStart
        let ke = entries[idx].keyEnd
        var segCount: UInt8 = 1
        var initialsBytes = [UInt8]()
        if ks < ke { initialsBytes.append(buffer[ks]) }
        for pos in ks ..< ke {
            if buffer[pos] == sep {
                segCount += 1
                if pos + 1 < ke { initialsBytes.append(buffer[pos + 1]) }
            }
        }
        let initials = String(decoding: initialsBytes, as: UTF8.self)
        keyInitialsIDMap[initials, default: []].append(nodeID)
        entries[idx] = .init(keyStart: ks, keyEnd: ke, startLine: entries[idx].startLine,
                             count: entries[idx].count, segmentCount: segCount)
    }
}
```

**預估改善**：消除 ~70 萬次分配，**15-20%**。

#### 🟠 瓶頸 #3：排序中的 `Data.compareUTF8Ranges` — 逐 byte 的 `Data` subscript

**位置**：L368-370 (排序) + L1442-1469 (`compareUTF8Ranges` 實作)

```swift
fileprivate func compareUTF8Ranges(_ lhs: Range<Int>, _ rhs: Range<Int>) -> Int {
    for index in 0 ..< count {
        let lhsByte = self[lhs.lowerBound + index]   // ← Data subscript，Foundation bridging！
        let rhsByte = self[rhs.lowerBound + index]   // ← 每次 bounds check + ObjC dispatch
    }
}
```

**問題**：`Data` 的 subscript 在 Intel 上經過 `NSData`/`CFData` bridging，每次存取都有 bounds check + ObjC 訊息傳遞開銷。排序 10 萬個 entry 時，每次比較平均 5-20 bytes，總共可能數百萬次 subscript 調用。

**修復方案**：先取得 buffer pointer，再直接比較：

```swift
data.withUnsafeBytes { rawBuffer in
    let buffer = rawBuffer.bindMemory(to: UInt8.self)
    entries.sort { lhs, rhs in
        let lhsLen = lhs.keyEnd - lhs.keyStart
        let rhsLen = rhs.keyEnd - rhs.keyStart
        for i in 0 ..< Swift.min(lhsLen, rhsLen) {
            if buffer[lhs.keyStart + i] < buffer[rhs.keyStart + i] { return true }
            if buffer[lhs.keyStart + i] > buffer[rhs.keyStart + i] { return false }
        }
        return lhsLen < rhsLen
    }
}
```

**預估改善**：排序部分快 **3-5x**，整體 **10-15%**。

#### 🟡 瓶頸 #4：`scalar.properties.isIdeographic` — Unicode 屬性查表

**位置**：`collectCharsFromPlainCellBytes` (L642) / `collectCharsFromGroupedCellBytes` (L690)

```swift

```

**問題**：`isIdeographic` 每次查詢都走 ICU/CoreFoundation 的 Unicode 屬性表。雖然 Swift 可能有 bitmap lookup 最佳化，但在 Intel 上記憶體存取延遲較高、cache 較小，查表成本明顯。

**修復方案**：對於 vChewing 的使用場景（中文輸入法），可以直接用 scalar value 範圍檢查：

```swift
    return (v >= 0x3400 && v <= 0x4DBF)    // Extension A
        || (v >= 0x4E00 && v <= 0x9FFF)    // Unified
        || (v >= 0xF900 && v <= 0xFAFF)    // Compatibility
        || (v >= 0x20000 && v <= 0x2A6DF)  // Extension B
        || (v >= 0x2F800 && v <= 0x2FA1F)  // Compatibility Supplement
}
```

```swift

```

**預估改善**：**5-8%**。

### 四、為何 Intel x86-64 上差異這麼大

除了單純的 IPC（每時脈週期指令數）差異之外，以下因素使得 TextMapTrie 建構在 Intel Mac 上格外受影響：

1. **`Data` subscript 的 bridging 成本**：在 Intel 上 `Data`（實作上橋接至 `NSData`）的 subscript 存取需要穿越 ObjC 分發層；Apple Silicon 上 Swift 對 Foundation 型別的 bridging 有更直接的 codegen。
2. **記憶體分配延遲**：Apple Silicon 使用更現代的 malloc（nanov2），大規模分配場景下比 Intel 的 malloc（magazine malloc）快得多。
3. **Cache 大小**：Apple Silicon 有更大的 L2 cache（M1 每 cluster 12MB shared L2 vs Intel i5-1038NG7 每核心 256KB L2 + 6MB shared L3），`buildReverseLookupTable` 的 `charToLineIndices` 字典在工作集接近 cache 上限時效率會急遽下降。
4. **Unicode 屬性表 cache**：`isIdeographic` 查詢涉及的 ICU Unicode 屬性表，在 Apple Silicon 較大 cache 中保留率遠高於 Intel。

```
  ├─ replacingOccurrences(of: "\r", with: "\n") → O(n) copy
  └─ replaceData(textData: processed) →
       ├─ replacingOccurrences(of: "\t", with: " ") → O(n) copy  ← 🔴 同上，重複
       ├─ strData == rawStrData 早期退出
       ├─ parse(splitee: "\n") { theRange in                    ← UTF-8 位元組層級掃描，OK
       │    ├─ parseCells(in: theRange, splitee: " ") → 取前兩格  ← UTF-8 fast path，OK
       │    ├─ String(rawStrData[keyRange])                    ← 🟡 每行一個 String 分配
       │    ├─ theKey.convertToPhonabets()                     ← 🟢 注音鍵 early-return 近乎零成本
       │    └─ newMap[theKey, default: []].append(theRange)
       ├─ rangeMap = newMap
       ├─ keySet = Set(rangeMap.keys)                          ← 🟡 可省略，rangeMap.keys.contains 就是 O(1)
       └─ sortedKeys = rangeMap.keys.sorted()                   ← 🟡 可改為惰性計算
```

### 二、瓶頸詳析

#### 🔴 瓶頸 #1：`replacingOccurrences(of: "\t")` 被呼叫兩次

`open()` 對非 consolidated 的資料先做一次 `\t` → ` ` 替換（L89-92），`replaceData()` 內部又無條件做一次（L105）。兩次都是 O(n) 全檔掃描與 String 分配複製。對於破萬行的老用戶片語資料（可達數 MB），這等於把整份資料複製了兩次。

**修復**：移除 `open()` 中的 `\t` → ` ` 替換，讓 `replaceData()` 統一處理。

#### 🟡 瓶頸 #2：`keySet` 冗餘

`hasUnigramsFor(key:)` 的實作：
```swift
func hasUnigramsFor(key: String) -> Bool {
  keySet.contains(key) || temporaryMap[key] != nil
}
```

Swift `Dictionary.Keys` 的 `contains(_:)` 本身就支援 O(1) hash lookup，與 `Set.contains` 等價。`keySet = Set(rangeMap.keys)` 是純粹的記憶體與 CPU 浪費（萬組 key 時約額外數百 KB 分配 + hash 重算）。

**修復**：移除 `keySet` 屬性，`hasUnigramsFor` 改為：
```swift
func hasUnigramsFor(key: String) -> Bool {
  rangeMap.keys.contains(key) || temporaryMap[key] != nil
}
```
同步從 `replaceData` 移除 `keySet = Set(rangeMap.keys)`、從 struct 宣告移除 `var keySet: Set<String>`、從 `clear()` 移除對應清理。

#### 🟡 瓶頸 #3：`sortedKeys` 每次 `replaceData` 都重新排序

`sortedKeys = rangeMap.keys.sorted()` 在每次 `replaceData` 結尾執行，O(k log k)。對於 k=10,000 的使用者，約 13 萬次字串比較。然而 `sortedKeys` 僅供 `keys(matchingPrefix:)`（partial match）使用——若該路徑不頻繁，排序成本是浪費的。

**修復**：將 `sortedKeys` 改為惰性計算——在 `replaceData` 後設 dirty flag，第一次呼叫 `keys(matchingPrefix:)` 時才排序。

#### 🟢 瓶頸 #4（非瓶頸）：`convertToPhonabets()`

對每個 key 呼叫 `theKey.convertToPhonabets()`。vChewing 使用者詞典預設以注音符號為鍵（非 ASCII），`convertToPhonabets()` 第一行即執行 `!isNotPureAlphanumerical` 檢查——對非 ASCII 注音符號直接 early-return，零成本。僅對純 ASCII 拼音鍵（罕見使用情境）才會觸發多次 `replacingOccurrences`。不受筆數影響，無需處理。

#### 🟢 瓶頸 #5（結構性限制）：`String(rawStrData[keyRange])` 每行一個 String 分配

無法避免——因為 key 需要作為 `Dictionary<String, [Range<String.Index>]>` 的鍵。但與 Phase 79 的 TextMapTrie（原廠辭典可達數十萬行）不同，LMCoreEX 處理的是使用者詞典（最多數萬行），String 分配規模相差一個數量級。不需要像 TextMapTrie 那樣改為 `Data` + byte range 的全面重構。

### 三、破萬筆下的影響量化

| # | 瓶頸 | 百筆級 | 萬筆級（目標用戶） |
|---|------|--------|---------------------|
| #1 | 雙重 `replacingOccurrences` | 無感 | O(n) ×2 全檔複製，數 MB 級 |
| #2 | 冗餘 `keySet` | 無感 | 10K 組 key hash 重算 + Set 分配，數百 KB 記憶體 |
| #3 | 每次 `sortedKeys` 重排 | 無感 | O(k log k)，~13 萬次字串比較 |
| #4 | `convertToPhonabets` | 無感 | 無感（注音鍵 early-return） |
| #5 | per-key String 分配 | 無感 | 萬次 String 分配，但架構上合理 |

### 四、綜合評估

| 項目 | 值 |
|------|-----|
| 建議施術項 | #1、#2、#3（共三項） |
| 預估改善 | 對萬筆級使用者詞典載入可感知加速（減少全檔複製 + Set 建構 + 排序） |
| 風險 | 低——全部為 `lmCoreEX.swift` 內部實作細節，不改公開 API |
| 需跨倉庫 | `vChewing-macOS` + `vChewing-OSX-Legacy`（LMCoreEX 為 LangModelAssembly 組件，不在 LibVanguard 中） |
