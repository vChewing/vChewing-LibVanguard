## 1. 問題陳述

### 1.1 重複實作

`switchState()` 與 `resetInputHandler()` 在兩個位置有近乎完全相同的實作：

|------|------|

- debug log 條件判斷的 `prefs` 來源不同（`self.prefs` vs `PrefMgr.sharedSansDidSetOps`）
- `resetInputHandler` 的 `commitExisting` 參數與 `mixedAlphanumericalBuffer` 提交（Mock 版缺少這兩項）

### 1.2 無法直接復用的根本原因

當時 `SessionCoreProtocol` 定義在 `Shared` package，`Handler` 的 associated type 約束為 `InputHandlerCoreProtocol`：

```swift
// Shared/SessionCoreProtocol.swift（舊）
public protocol SessionCoreProtocol: AnyObject & CtlCandidateDelegate {
  associatedtype Handler: InputHandlerCoreProtocol  // ← 空殼，無 clear()/committableDisplayText()
  ...
}
```

`InputHandlerCoreProtocol` 僅四行、只宣告 associated type 約束，沒有任何方法。而 `switchState()` 需要呼叫 `inputHandler.clear()` 和 `inputHandler.committableDisplayText()`——這些方法定義在 `InputHandlerProtocol`（Typewriter SPM）的 extension 中。

若在 Shared 端直接收緊 `Handler` 約束至 `InputHandlerProtocol`，會造成 Shared → Typewriter 的循環依賴（`InputHandlerProtocol` 定義在 Typewriter，而 Typewriter 依賴 Shared）。

### 1.3 協議層級過多

手術前的完整協議鏈：

```
Shared:    SessionCoreProtocol ──── InputHandlerCoreProtocol (空殼，4 行)
Typewriter:  (無基底協議)          InputHandlerProtocol : InputHandlerCoreProtocol
MainAssembly: SessionProtocol : SessionCoreProtocol  →  InputHandler
```

`SessionCoreProtocol` 和 `InputHandlerCoreProtocol` 是僅為滿足 associated type 循環約束而存在的空殼層，`Typewriter` package 本身沒有自己的 session 基底協議。

## 2. 手術設計

### 2.1 核心思路

一步到位地解決三個問題：
1. 將 `SessionCoreProtocol` 從 `Shared` 搬至 `Typewriter`
2. 將 `Handler` 約束直接收緊為 `InputHandlerProtocol`（不再需要中間層）
3. 在 `SessionCoreProtocol` 上直接提供 `switchState()` 與 `resetInputHandler()` 的預設實作
4. 將 `InputHandlerCoreProtocol` 的內容內聯進 `InputHandlerProtocol`，消滅空殼

### 2.2 `debugLogCondition`：以 computed property 取代硬編碼差異

兩邊 `switchState()` 的唯一邏輯差異是 debug log 的觸發條件。引入一個 computed property 來抽象這個差異：

- **MockSession** 覆寫為 `PrefMgr.sharedSansDidSetOps.isDebugModeEnabled`
- **Darwin SessionProtocol** 覆寫為 `self.prefs.isDebugModeEnabled`

### 2.3 五大抽象方法：以 protocol requirement 承載平台差異

`switchState()` 內部呼叫的五個平台相關操作，全部以 protocol requirement 形式留在 `SessionCoreProtocol` 中，由各平台自行實作：

| 抽象方法 | Darwin 實作 | Mock 實作 |

| `commit(text:clearDisplayBeforeCommit:)` | IMK `client.insertText` + 繁簡轉換 + 語彙置換 | `recentCommissions.append(text)` |
| `updateCompositionBufferDisplay()` | `setMarkedText` + popup window | 空（no-op） |
| `toggleCandidateUIVisibility(_:refresh:)` | 顯示/隱藏候選窗 NSWindow | 切換 flag |

| `getMitigatedState(_:)` | 依 `clientMitigationLevel` 調整 cursor | 原樣返回 |

這些本身就是「抽象 API」——它們在 `SessionCoreProtocol` 中以 protocol requirement 存在。若未來有輕量級的額外差異點，可用 lambda/computed property（如 `debugLogCondition`）補充，但五大核心操作保持 protocol requirement 以確保型別安全與除錯清晰度。

## 6. 設計取捨

### 為何不是 base class？

`InputSession` 目前為 `final class`（設計選擇，非 IMK 強制——IMKController 是 `SessionCtl`）。即使去除 `final`，Swift 不支援多重繼承。關鍵障礙是 associated type `Handler`：`MockSession.Handler == MockInputHandler` 與 `InputSession.Handler == InputHandler` 是不同型別，一個 concrete base class 無法同時滿足兩者的 associated type 約束。

Protocol + default implementation 是 Swift 中處理「共享邏輯 + 型別參數化」的正確模式。

### 為何不保留 InputHandlerCoreProtocol？

它僅四行、無任何方法，純粹是 associated type 循環約束的佔位符。當 `SessionCoreProtocol` 與 `InputHandlerProtocol` 同屬一個 package 時，循環約束可以直接寫在 `InputHandlerProtocol` 的 `associatedtype Session` 上，無需中間層。

### IMEState / IMEStateData / IMEStateProtocol 為何留在 Shared？

`LangModelAssembly` 透過 `some IMEStateProtocol` 引用 `IMEStateProtocol`。`LangModelAssembly` 是資料/模型層，不應反向依賴引擎層 Typewriter。`SessionCoreProtocol` 引用 `IMEState` 透過 `import Shared` 解析，無循環依賴。

TooltipUI 系統在連續接收到兩個顯示任務的時候，如果第二個任務是恆久顯示、且第一個任務是短暫顯示、且第二個任務傳入時第一個任務還沒消失的話，則第二個任務會顯示一瞬就消失。推測是被第一個任務的消失機制誤殺了。

## Phase 84 手術方案

### 一、根因分析

問題出在 `TooltipUI.show()` 的延遲消失機制：

```swift
// 現有程式碼 (macOS & Legacy 相同邏輯)
public func show(..., duration: Double) {
    // ...
    window?.setIsVisible(true)
    if duration > 0 {
        asyncOnMain(after: duration) {
            self.window?.orderOut(nil)  // ← 這個 closure 沒有被追蹤、無法取消
        }
    }
}
```

**觸發場景**：
1. 第一個 `show(duration: 1.0)` 被呼叫 → 排程了 1 秒後的 `orderOut(nil)`。
2. 0.3 秒後，第二個 `show(duration: 0)`（恆久顯示）被呼叫。`showTooltip()` 雖然先呼叫了 `hide()`，但 `hide()` 只做 `orderOut(nil)`，**並不會取消第一個 show 排程的延遲 closure**。
3. 第二個 show 正確顯示了 tooltip。但再過 0.7 秒（即第一個呼叫後的 1 秒），第一個 show 的延遲 closure 準時觸發，把恆久顯示的 tooltip 也關掉了。

**根因一句話**：`asyncOnMain(after:)` 產生的延遲 closure 沒有被追蹤/取消，`hide()` 也沒有取消它的機制。

### 二、修復策略：Generation Counter（世代計數器）

採用一個 `Int` 世代計數器，每次 `show()` 被呼叫時遞增。在延遲 closure 內比對：若世代已變，表示已有新的 show 覆寫，則跳過本次 `orderOut(nil)`。

```swift
// 新增屬性
private var showGeneration: Int = 0

// 修改 show()
public func show(..., duration: Double) {
    showGeneration += 1
    let currentGeneration = showGeneration
    // ... 現有顯示邏輯 ...
    if duration > 0 {
        asyncOnMain(after: duration) {
            guard self.showGeneration == currentGeneration else { return }
            self.window?.orderOut(nil)
        }
    }
}
```

**選擇 generation counter 而非 `Task.cancel()` 的理由**：
- `asyncOnMain(after:)` 在 macOS <12 走 `DispatchQueue.main.asyncAfter`，不支援 cancellation。
- generation counter 對兩條路徑（`Task` / `DispatchQueue`）都有效。
- 邏輯簡單、無額外 allocation。

```

    }

```

改為：

```swift

    asyncOnMain(after: duration) {
        guard self.showGeneration == currentGeneration else { return }

    }

```

### 已知審計發現

本節記錄 Phase 86 初次實作後的程式碼審計結果，供後續 Phase 參考。

#### `computeCandidateOnlySize()` 與 `updateMetrics()` 的邏輯重複

兩個方法共享約 80% 的 cell 排版邏輯，但各自為政。共同的步驟包括：重置 shitCell、呼叫 updateMetrics 取得 minimumCellDimension、遍歷 cell 計算 visualDimension、horizontal matrix 下使用 cellWidthMultiplied、vertical/短詞寬度歸一化、累積行尺寸並推進 origin、以及 vertical 模式下同列共寬。

差異僅在：

- `updateMetrics()` 只遍歷 `lineRangeForCurrentPage`，補 dummy 空白行，計算高亮 rect / Peripherals / Reading，輸出至 `self.metrics`，origin 為 `(originDelta, originDelta)`。
- `computeCandidateOnlySize()` 遍歷全部 `candidateLines`，不補空白行，不計算高亮與底部欄位，輸出至 `self.candidateOnlySize`，origin 為 `(0, 0)`。

**Origin `(0, 0)` 是刻意設計**：GSI view 的 `drawScrollMode()` 透過 `NSAffineTransform.translateX(by:yBy:)` 手動疊加 `originDelta`，以此製造 scrollView-scrollBar 與 window 邊框之間的 padding 間距。若將其改為與 `updateMetrics()` 一致的 `(originDelta, originDelta)`，transform 層會重複偏移、破壞排版。

**改善建議**：提取共通排版邏輯為 `private func layoutCells(in:initialOrigin:) -> CGSize`，由兩個方法各自呼叫。可消除約 40 行重複、杜絕未來 origin 意外同步的風險，同時保持各自獨立的 origin 策略。

> ✅ **已實作**：共通排版邏輯已提取至 `layoutCells(in:initialOrigin:)`。`updateMetrics()` 與 `computeCandidateOnlySize()` 均透過該 helper 排版，各自僅保留差異化邏輯（highlight rect 反推、peripherals 計算、blank cells 補償等）。兩個倉庫均已重構並通過建置。

#### `pageCandidateSize`、`computeCandidateOnlySize()`、`scrollByLines` 皆為刻意設計

以下三點經審計確認均為正確的刻意行為：

- **`pageCandidateSize` 使用 `lineRangeForCurrentPage`**：滑鼠捲動僅供 peek -> click 用途，鍵盤巡覽仍需正確影響捲動對齊。`pageCandidateSize` 反映的是鍵盤巡覽當前所在頁的尺寸，此為正確語意。
- **highlight 導航時呼叫 `computeCandidateOnlySize()`**：非過度呼叫。column matrix 模式下每個 visible column 的寬度會因該 column 內最長 cell 而變化，因此 highlight 移動後確實需要重算。
- **`scrollByLines` vertical 模式掃描 `visualOrigin.x`**：流程為 pool 先 scroll → 得出此時該顯示的四個 column → 計算它們的寬度 → 影響 scrollView width。此為正確的依賴順序。

### 根因分析

#### P0：pinEdges(to:) 約束累積

updateNSWindowModern() 每趟呼叫都會執行 candidateView.pinEdges(to: containerView)。pinEdges 實作（AppKitImpl_NSView.swift:112）每次都新建 4 個 NSLayoutConstraint 並透過 view.addConstraints() 疊加，從不清理舊約束。

一個 SCPC 逐字選字 session 可觸發上百次 updateDisplay()，累積四百多個孤立的 constraint 物件。這是記憶體持續增長的主要來源。舊 TDK 雖有相同 pattern，但 GSI 因額外的 configureScrollMode / configureNormalMode 內部開銷，總體更快突破閾值。

#### P1：attributedStringHeader / attributedStringPhrase 為 computed property

CandidateCellData4AppKit 中這兩個屬性每次存取都新建 NSAttributedString。GSI scroll mode 下 drawScrollMode() 繪製全部候選行（非僅當前頁）。50 個候選字 = 每幀 100 個 attributed string。觸控板慣性捲動 60 fps = 每秒 6000 個臨時物件。

#### P2：同幀內重複 flatMap

drawScrollMode() 和 computeHighlightRects() 各自對 candidateLines 呼叫 flatMap { $0 }，對同一批資料建立了兩份臨時陣列。

#### P2：NSAffineTransform() 每幀新建

NSAffineTransform 是 class（非 struct），drawScrollMode() 每次都新建一個。慣性捲動期間每秒 60 個 heap allocation。

#### 已排除

## Phase 88 實作結果

### 根因分析

就地加詞的完整流程為 `writeUserPhrasesAtOnce → write(toFilter:)（追加一行到檔案）→ loadUserPhrasesData → replaceData（全量重新解析 ~10k 行，重建 rangeMap）`。

原有的 `writeUserPhrasesAtOnce` 使用 `withFileHandleQueueSync`（即 `dispatch_sync`）從 MainActor 呼叫整個流程：

1. **檔案追加**（微秒級，不是瓶頸）
2. **`loadUserPhrasesData`**：走 `asyncLoadingUserData = true` 的非同步路徑 → `readFileContentAsync` → `fileHandleQueue.async { ... }`，這個 async block 被排入同一 serial queue 但不立即執行。
3. **`withFileHandleQueueSync` 返回**（MainActor 短暫卡 µs 級）
4. **async block 在 fileHandleQueue 上執行**：consolidate（pragma 命中則跳過）+ 讀取檔案 + `asyncOnMain { completion }`
5. **completion 回到 MainActor**：`withFileHandleQueueSync { replaceData }` → MainActor 被 `dispatch_sync` 卡住，等待 fileHandleQueue 完成全量解析 ~10k 行 → **這就是使用者感受到的 ~1 秒卡頓**。

核心問題是步驟 5：`replaceData` 本應在背景執行，但 completion handler 用 `withFileHandleQueueSync` 從 MainActor 同步等待 fileHandleQueue 完成工作。

### 修復策略

**單刀：將 reload 從同步改為非同步。**

`writeUserPhrasesAtOnce` 保持原有的 `withFileHandleQueueSync`（DispatchQueue 相容，可移植至 macOS 10.9 legacy），但在尾端呼叫 `loadUserPhrasesData(type:async: true)` 而非原本不傳參數（預設走 `asyncLoadingUserData` 判斷）。

`async: true` 會走 `readFileContentAsync` → `fileHandleQueue.async`。由於當前正處於 `withFileHandleQueueSync` 內部（已在 fileHandleQueue 上），這個 async block 會被排入同一 serial queue、但直到當前 sync block 返回後才執行。之後 completion 回到 MainActor 做 `replaceData`，MainActor 會短暫卡 ~1s——但這發生在非同步的未來，不影響使用者當下的打字體驗。

**時序對比**：

| | 修改前 | 修改後 |
|---|---|---|
| 使用者點擊加詞 | MainActor 卡 ~1s | MainActor 不卡（µs 級返回） |
| ~1s 後 | （已卡完） | `replaceData` 全量解析，MainActor 短暫卡 |
| reload 完成 | 同步完成 | 非同步完成 |

**跨平台相容**：全程使用 `DispatchQueue`（`withFileHandleQueueSync` / `asyncOnMain` / `fileHandleQueue.async`），無 Swift Concurrency 依賴，可直接移植至 macOS 10.9 legacy 分支。

### 為何不採用「兩刀」方案

原本設計的「兩刀」方案（外層 `withFileHandleQueueAsync` + 內層 `async: false` sync reload）可完全消滅 MainActor 阻塞，但在 vChewing-macOS 的 Swift 6 模組級 `@MainActor` 環境下無法編譯：`write(toFilter:)` 等 implicit `@MainActor` 方法無法從 `fileHandleQueue.async` 的 nonisolated closure 呼叫，而將其標為 `nonisolated` 會引發深層級聯（`LMMgr` / `LMInstantiator` 所有依賴鏈均需改 `nonisolated`），手術範圍過大。另 `Task.detached` + `await MainActor.run` 方案則會在 tooltip 顯示時引發 deadlock，且 Swift Concurrency API 無法移植至 legacy 分支。

### 方案選擇

| 方案 | 說明 | DispatchQueue 相容 | 選擇 |
|------|------|:-:|:-:|
| 單刀：reload 改 async:true | 寫入 µs 返回，reload 非同步後臺完成 | ✅ | ✅ 採用 |
| 兩刀：async 外層 + sync 內層 | MainActor 零阻塞，但 Swift 6 @MainActor 無法編譯 | ✅ | ❌ 級聯過深 |
| Task.detached + MainActor.run | deadlock + 無法移植 legacy | ❌ | ❌ |
