### 修復的 Bug

**Bug 1: `Homa_SwiftImpl.sliced(by:)` 空分隔符號回傳空陣列**

- 根因：`sliced(by: "")` 原實作沒有處理空分隔符號。`arrSeparator.count == 0` 時，每輪迴圈切出的 `ripped` 皆為空陣列，隨即被 `if ripped.isEmpty { continue }` 跳過，導致所有字元都沒進入 `buffer`，最後只能回傳空字串結果。
- 修復：在 `sliced(by:)` 開頭新增 `guard !arrSeparator.isEmpty else { return selfArray.map { String($0) } }`，空分隔符號時改為逐 Unicode Scalar 拆分。
- 影響：修復 `testHardCopyAndWordSegmentation` 測試失敗。

**Bug 2: 候選權重不應寄生在 `CandidatePair` 上**

- 根因：當時為了快速打通 POM 路徑，暫時把 LM 權重塞進 `CandidatePair.score`，再由 Typewriter 透過 `.map(\.pair)` 萃取 raw candidates。這讓 `CandidatePair` 同時扮演身份型別與帶權重型別，POM 過濾也因此必須依賴錯誤的資料模型。
- 修復：最終 follow-up 直接把 `score` 從 `CandidatePair` 拆出，讓 `CandidatePair` 僅保留 `(keyArray, value)`；若要攜帶權重，統一使用 `CandidatePairWeighted` 或 `CandidatePair.weighted(_:)`。`fetchCandidates()`、`overrideCandidate()` 的 weighted overload、Typewriter 的 raw candidate 過濾與 POM 測試路徑全部同步改走 weighted 型別。
- 影響：修復 `test_IH301_POMBleacherIntegrationTest` 與 `test_IH303_POMIgnoresLowerWeightSuggestedUnigramMatchingRawQueriedUnigram` 測試失敗。

**Bug 3: `CandidatePair` 的 `Hashable` / `Equatable` 合約違反**

- 根因：只要 `CandidatePair` 內含 `score`，就很容易讓身份相等與權重差異兩種概念糾纏在一起，進一步放大 `Hashable` / `Equatable` 契約風險。
- 修復：初始 follow-up 先手動實作 `hash(into:)` 對齊 `raw` 身份；本輪則更進一步把 `score` 整個移出 `CandidatePair`。現在 `CandidatePair` 與 `CandidatePairWeighted` 各司其職，資料模型本身就不再鼓勵混裝。
- 影響：`CandidatePair` 的身份一致性問題被結構性移除，並補齊純候選 / weighted 候選的回歸測試與 POM / revolver / perceptor 路徑驗證。

## 功能需求及其背景

Phase 34 (`1c55b577b9d77ed86de93bd632dc90a6741e1121`) 實測發現 factory dictionary 在 Intel Mac mini 2018 (i7) 上的載入 CPU workload 過重，client app 會 freeze 兩秒以上。當時的核心問題不是功能缺漏，而是 `TextMapTrie` 在這個 workload 下的資料載入與查詢抽象過於通用，熱路徑沒有真正對準 app-side factory dictionary 的使用形狀。

最初討論時有兩個方向：

1. 暫時把下游退回 `FactoryTextMapLexicon` 這類專用 loader。

## 最終定案（by GPT-5.4）

Phase 35 的最終答案不是讓 `FactoryTextMapLexicon` 長期留在下游，而是把那條驗證過的專用 loader 設計上推為 canonical `VanguardTrie.TextMapTrie`。換句話說，這一輪真正留下來的不是「LangModelAssembly 私有岔路」，而是「TrieKit 自身的 TextMap backend 重新定義」。

原因很明確：中間驗證已證明，真正有效的不是「把程式名字改回 FactoryTextMapLexicon」，而是以下幾個結構特徵本身：raw `Data` 常駐、排序後 full-key index、binary search 精準查詢、longer-segment prefix-range scan、以及 eager reverse lookup table。既然如此，把這套設計留成下游 fork，只會導致 TrieKit 與 LangModelAssembly 各自維護一份 TextMap 語義，未來還會再次漂移。

真正還需要補的，是 `VanguardTrieProtocol` 的抽象形狀。舊的 protocol extension 只會先經過 `getNodes(...)`，因此一旦呼叫端把具體型別抹成 `any VanguardTrieProtocol`，`queryGrams(...)`、`hasGrams(...)`、`queryAssociatedPhrasesAsGrams(...)` 這些高階 API 就無法有效命中 `TextMapTrie` 的 specialized backend。本 phase 的定案因此是：保留 `getNode` / `getNodes` 既有 surface，不破壞舊 API；同時新增 `getEntryGroups(...)` requirement，讓高階查詢直接以 entry groups 為工作單位，從而讓 materialized-node backend 與 specialized-entry backend 都能在同一個 protocol 下各走各的快路。

如果要把這輪定案壓縮成一句話：Phase 35 最終做的不是「把下游退回專用 loader」，而是「把專用 loader 的有效設計 canonicalize 進 TrieKit，並修正 protocol 抽象，讓這條 fast path 在 existential dispatch 下仍然成立」。在 `vChewing-macOS` 的最終 history 當中，Phase 35 因此只留下 TrieKit mirror commit；先前短暫存在過的 downstream fallback loader commit 不屬於最終定案。

## Phase 39 實作結果

1. **根因 A（HIGH IMPACT）**：`mergedAlternativeBucketUnigrams(for:)` 對 Phase 37 的 tone bucket `&` alternatives 做笛卡爾積展開後逐一完整查詢，查詢量為 O(tone_variations^syllable_count)，6-syllable 免聲調句子可達 5⁶ = 15,625 次完整 `unigramsFor()` 呼叫。`VanguardTrieProtocol` 已有 `getNodes(keysChopped:...)` / `getEntryGroups(keysChopped:...)` API 在 trie 層級處理 `&` alternatives 並以 node ID 去重，但 LMI 層完全未使用。

### Phase B：消除 `getNodeIDsForKeyArray` 的雙重 NSLock 與字串比較

**根因**：`getNodeIDsForKeyArray` 對每個 key 呼叫 `getCachedFirstChar`（NSLock + Dictionary subscript + `_stringCompareInternal`），佔該方法 35% CPU。再加上 `internKey` 的第二次 NSLock，每個 key element 產生 2N lock acquisitions 與 N 次字串比較。

**vChewing-OSX-Legacy**:
- `TrieTextMap_Core.swift`：同步改動。

**影響**：消除 35% 時間開銷，直接落地對 `getNodeIDsForKeyArray` 的最大型單一最佳化。

### Phase C：`QueryBuffer<T>` 去 GCD 化與單調時鐘化

**根因**：`QueryBuffer` 的 `DispatchQueue.sync` + `Date()` 在笛卡爾積放大下累積嚴重。每次 get/set/remove 都觸發 GCD 排隊、`Date()` 系統呼叫與 `TimeInterval` 浮點運算。

**實作**：

**vChewing-macOS** & **vChewing-LibVanguard**:
- `TK_QueryBuffer.swift`：完整改寫：
  - 移除 `DispatchQueue`，改用 `private let lock = NSLock()`
  - 時間戳從 `Date()` 改為 `DispatchTime.now().uptimeNanoseconds`（單調時鐘，無系統呼叫）
  - 改以 `operationCount` 每 64 次操作才觸發一次 `removeExpiredEntriesLocked(now:)` 清理，而非每次操作都建立 `Date()`
  - **重要**：`cleanupCheckInterval` 必須宣告為 `static var cleanupCheckInterval: UInt64 { 64 }`（computed property），因 Swift 不允許 generic type 內的 `static let` stored properties

**vChewing-OSX-Legacy**:
- `TK_QueryBuffer.swift`：同步改動。

**影響**：消除 GCD 基礎設施開銷與系統呼叫，每個 query 層級查詢可節省數微秒至數十微秒。

### Phase D：`Homa.Assembler.gramQueryCache` 從 Dictionary 到 Int hash key + LRU 半量淘汰
