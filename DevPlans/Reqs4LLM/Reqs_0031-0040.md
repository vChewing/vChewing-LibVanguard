### 修復的 Bug

**Bug 1: `Homa_SwiftImpl.sliced(by:)` 空分隔符號回傳空陣列**

- 根因：`sliced(by: "")` 原實作沒有處理空分隔符號。`arrSeparator.count == 0` 時，每輪迴圈切出的 `ripped` 皆為空陣列，隨即被 `if ripped.isEmpty { continue }` 跳過，導致所有字元都沒進入 `buffer`，最後只能回傳空字串結果。
- 修復：在 `sliced(by:)` 開頭新增 `guard !arrSeparator.isEmpty else { return selfArray.map { String($0) } }`，空分隔符號時改為逐 Unicode Scalar 拆分。
- 影響：修復 `testHardCopyAndWordSegmentation` 測試失敗。

**Bug 2: 候選權重不應寄生在 `CandidatePair` 上**

- 根因：當時為了快速打通 POM 路徑，暫時把 LM 權重塞進 `CandidatePair.score`，再由 Typewriter 透過 `.map(\.pair)` 萃取 raw candidates。這讓 `CandidatePair` 同時扮演身份型別與帶權重型別，POM 過濾也因此必須依賴錯誤的資料模型。
- 修復：最終 follow-up 直接把 `score` 從 `CandidatePair` 拆出，讓 `CandidatePair` 僅保留 `(keyArray, value)`；若要攜帶權重，統一使用 `CandidatePairWeighted` 或 `CandidatePair.weighted(_:)`。`fetchCandidates()`、`overrideCandidate()` 的 weighted overload、Typewriter 的 raw candidate 過濾與 POM 測試路徑全部同步改走 weighted 型別。
- 影響：修復 `test_IH301_POMBleacherIntegrationTest` 與 `test_IH303_POMIgnoresLowerWeightSuggestedUnigramMatchingRawQueriedUnigram` 測試失敗。

**Bug 3: `CandidatePair` 的 `Hashable` / `Equatable` 合約違反**

- 根因：只要 `CandidatePair` 內含 `score`，就很容易讓身份相等與權重差異兩種概念糾纏在一起，進一步放大 `Hashable` / `Equatable` 契約風險。
- 修復：初始 follow-up 先手動實作 `hash(into:)` 對齊 `raw` 身份；本輪則更進一步把 `score` 整個移出 `CandidatePair`。現在 `CandidatePair` 與 `CandidatePairWeighted` 各司其職，資料模型本身就不再鼓勵混裝。
- 影響：`CandidatePair` 的身份一致性問題被結構性移除，並補齊純候選 / weighted 候選的回歸測試與 POM / revolver / perceptor 路徑驗證。
