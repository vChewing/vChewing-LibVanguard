## Phase 103 實作結果（2026-07-06, GLM-5.2）

### 根因分析

倚天傳統佈局（ETen）的數字鍵映射與大千傳統佈局（Dachen）本質不同：

| 鍵位 | ETen | Dachen |
|------|------|--------|
| `1` | ˙（聲調） | ㄅ（聲母） |
| `2` | ˊ（聲調） | ㄉ（聲母） |
| `3` | ˇ（聲調） | ˇ（聲調） |
| `4` | ˋ（聲調） | ˋ（聲調） |
| `7` | ㄑ（韻母） | ˙（聲調） |
| `8` | ㄢ（韻母） | ㄚ（韻母） |
| `9` | ㄣ（韻母） | ㄞ（韻母） |
| `0` | ㄤ（韻母） | ㄢ（韻母） |
| `.` | ㄔ（聲母） | ㄡ（韻母） |
| `5` `6` | 未映射 | ㄓ / ˊ |

關鍵差異：**ETen 的 `1`-`4` 是聲調鍵，`7`-`9`/`0` 是韻母鍵**，因此純數字序列可組成合法注音讀音（如 `92` = ㄣˊ = én → 嗯）。Dachen 的 `1`/`2` 是聲母，`9` 是韻母、`2` 是聲母，`92` = ㄞㄉ 違反 CSVT 順序（韻母在前聲母在後）→ 不構成合法讀音。

`bestAutoSplitCandidate()` 的 `blockedPrefixLength` 機制將開頭聲調數字鍵（如 ETen 的 `1`→˙）視為「被阻斷的 ASCII 前綴」。當輸入 `192` 時：
1. `1`→˙ 為聲調鍵 → `blockedPrefixLength = 1`，`1` 被當作 ASCII 前綴。
2. `92` = ㄣˊ 為合法注音讀音（én → 嗯），且詞庫有命中 → auto-split 候選成立。
3. `validCandidates` 的 `blockedPrefixLength > 0` 分支僅檢查 `prefixText.count >= blockedPrefixLength`（`1`.count=1 >= 1 ✓），未檢查後綴是否全為數字 → 候選通過。
4. 結果：`192` 被拆為 `1`（ASCII）+ `嗯`（ㄣˊ），後續 `.1` = ㄔ˙ → 饬，`68` 因 `6` 未映射而保留為 ASCII，`.1` 再次產生饬……最終 `1嗯饬68饬00饬`。

### 修復方案

在 `bestAutoSplitCandidate()` 的 `validCandidates` 過濾器中，於 `blockedPrefixLength > 0` 分支新增檢查：**後綴全為 ASCII 數字時拒絕該候選**。

此修復的精確性：
- 僅在開頭有被阻斷鍵（聲調數字或大寫字母）時生效（`blockedPrefixLength > 0`）。
- 後綴含字母（如 `gj;3`）或符號（如 `.1`）時不受影響 → 不破壞既有 `4gj;3`→`4`+`爽`（IH426）等用例。
- `blockedPrefixLength = 0` 時（如 `cOS` 前綴、`?c` 前綴）完全不觸發 → Izanami `cOS` 前綴用例不受影響。
- Dachen 佈局下 `1`→ㄅ 非 聲調鍵，`blockedPrefixLength = 0` → 不觸發 → Dachen 行為完全不變。

### 程式碼變更

| # | 位置 | 修復 | 說明 |
|---|------|------|------|
| 1 | `bestAutoSplitCandidate()` `validCandidates` 過濾器 | `blockedPrefixLength > 0` 且後綴全為 ASCII 數字時 `return false` | 阻止「聲調數字前綴 + 純數字注音後綴」的誤拆，使 IP 位址、日期等純數字序列保持 ASCII |

### `.digit`（句點+聲調數字）評估結論

事主提報 `.1`-`.4` 在 ETen 仍產生注音（`.1`=ㄔ˙=飭、`.2`=ㄔˊ=持、`.3`=ㄔˇ=齒、`.4`=ㄔˋ=赤）。經評估：

- `.2`/`.3`/`.4` 對應的 ㄔˊ/ㄔˇ/ㄔˋ 是倚天 DOS lexicon 內的常見讀音，阻斷將導致 mixed mode 下無法輸入持/遲/齒/尺/赤/翅等常用字。
- `.1`=ㄔ˙ 不在倚天 DOS lexicon 內，但在 vChewing factory core dictionary 內（飭/齒/嗤），屬合法 BPMF 讀音。
- `.digit` 的歧義是 mixed mode 的內在特性：`.` 是合法注音鍵（ETen 為 ㄔ），`1`-`4` 是合法聲調鍵。阻斷所有 `.digit` 將威脅 mixed mode 的核心職能（無需切換模式即可混輸中英文）。
- IP 位址 `192.168.100.1` 的問題已由 auto-split 修復（項 1）完全解決：連續輸入時 leading tone digit 阻斷 full-phonetic 路徑，buffer 過長亦跳過該路徑。

**結論：`.digit` 的全面阻斷不合理，不予實施。** 僅保留 auto-split 修復。`.1`(ㄔ˙→飭) 在 buffer 為空時單獨輸入的情境為 mixed mode 的內在歧義，使用者可藉由先按 Enter/Space 提交 ASCII 或切換非混合模式解決。

### 補充規則評估：ETen `.` + Intonation 於純數字句點 buffer 時視為 pure alnum

事主提出補充規則：「在 Tekkon Parser 為 ETen 傳統時，對 `.` + Intonation 的處理追加判斷：只要是在 `.` 之前輸入的 mixedAlnumBuffer 內容僅由數字與 `.` 構成，就當作 pure alnum 輸入。」經逐例追蹤評估，**該規則不正當，不予實施**。

#### 評估分析

**非空 buffer（僅含數字與 `.`）時，規則為 no-op：** full-phonetic 路徑在所有此類情境下均已被既有機制遮蔽，規則無法改變任何結果：

| 情境 | full-phonetic 路徑被遮蔽的原因 |
|------|-------------------------------|
| Buffer `"192"`，輸入 `.1` → fullInput `"192.1"`（5 字） | `fullInput.count > maxSingleSyllableKeyCount (4)` — 路徑跳過 |
| Buffer `"192."`，輸入 `1` → fullInput `"192.1"`（5 字） | 同上 — 過長 |
| Buffer `"8"`，輸入 `.1` → fullInput `"8.1"` | `receiveSequence("8.1")` → ㄢ(韻) ㄔ(聲) ˙ → CSVT 違規 → `isPronounceable` false |
| Buffer `"7"`，輸入 `.1` → fullInput `"7.1"` | `receiveSequence("7.1")` → ㄑ ㄔ ˙ → 兩聲母 → not pronounceable |
| Buffer `"1"`，輸入 `.1` → fullInput `"1.1"` | `isLeadingToneBlocked`: 首字 `"1"`→˙ → 阻斷 |
| Buffer `".0."`，輸入 `1` → fullInput `".0.1"`（4 字） | `receiveSequence(".0.1")` → ㄔ ㄤ ㄔ ˙ → CSVT 違規（聲母在韻母後）→ not pronounceable |

在所有非空數字/句點 buffer 情境中，`.` 均透過 fallthrough（`handler.composer.clear()`）以 ASCII 留在 buffer 內。規則不會改變任何結果。

**空 buffer 時，規則有害：** full-phonetic 路徑唯一對 `.` + 聲調數字觸發的情境是 `.` 之前 buffer 為空：

```
Buffer = "" → 輸入 "." → composer 收 ㄔ → buffer = "."
輸入 "1" → fullInput = ".1"（2 字 ≤ 4）
  → receiveSequence(".1") → ㄔ ˙ → 合法 CSVT、可發音、有聲調
  → hasUnigramsForFast(["ㄔ˙"]) → true（飭/齒/嗤 在 factory core dictionary）
  → 讀音接受 → 飭
```

若規則將空 buffer 視為「僅由數字與 `.` 構成」（vacuously true），將阻斷獨立 `.digit`，導致 ETen mixed mode 下無法輸入 **持**(ㄔˊ=`.2`)、**齒**(ㄔˇ=`.3`)、**赤**(ㄔˋ=`.4`) 等常用字——與全面阻斷方案面臨相同問題。

#### 結論

該規則增加複雜度（parser 專屬判斷、buffer 內容分析）但不解決任何實際問題：
- 非空 buffer：規則為 no-op，full-phonetic 路徑已被既有機制遮蔽。
- 空 buffer：規則阻斷合法注音輸入，威脅 mixed mode 核心職能。
- IP 位址已由 Phase 103 auto-split 修復完全解決。

**不予實施。**

### 華碩輸入法比較觀察與延遲提交（lazy commit）評估

事主觀察到華碩輸入法（ASUS 輸入法）能在 ETen 佈局下直接輸入 IP 位址，完全不需要使用者在讀音形成前先按 Space/Enter 提交 ASCII。vChewing mixedAlnum 目前無法做到此點：`.digit` 在 buffer 為空時立即走 full-phonetic 路徑並提交讀音，無法事後撤銷。

#### 根本原因：eager commit 架構

vChewing mixedAlnum 的 full-phonetic 路徑採用 **eager commit**（即刻提交）：`.digit` 形成合法讀音後，第一分支立即 `insertKey()` 將讀音插入 assembler 並清空 buffer。讀音一旦進入 assembler 即無法回退為 ASCII。

#### 延遲提交（lazy commit）構想

full-phonetic 路徑已有兩個分支：
1. **第一分支（eager）**：`insertKey()` + 清空 buffer — 讀音立即進入 assembler
2. **第二分支（lazy）**：`composer = trialComposer` + `buffer = fullInput` — 讀音留在 composer、buffer 保留

若對 `.digit`（句點+聲調數字）跳過第一分支、改走第二分支，則：
- `.1` 單獨輸入 → 讀音 ㄔ˙ 留在 composer，buffer = `".1"`（未提交）
- `.1` + `9` → fullInput `".19"`，full-phonetic 因 CSVT 違規跳過，auto-split 失敗 → fallthrough 清空 composer，buffer = `".19"` → **ASCII** ✓
- `.2` + Space → space handler 視 `bufferIsSingleSyllablePhonetic=true`，走 BPMF 全匹配 → 提交 ㄔˊ → **持** ✓
- `.2` + Enter → 取決於 upper-level handler 對非空 buffer + 非空 composer 的 Enter 處理

此構想的關鍵優勢：**下一個按鍵自動消歧**。繼續打數字/句點 → ASCII；打 Space → 確認注音。

#### 未實施原因

此構想雖然理論上可行，但在本次 Phase 未實施，理由如下：

1. **行為差異需全面驗證**：lazy commit 使讀音留在 composer 而非 assembler，改變了使用者可見的 composing area 呈現行為（候選字出現時機延遲一鍵）。Izanami 測試的 empty-prefix 分支雖不會觸發既有斷言失敗（`assembler.length > 1` 及 `actualKeys.first != reading` 均不觸發），但測試的設計意圖（「in-assembler reading intact」）與 lazy 行為不一致，需重新評估測試斷言。
2. **Enter 路徑未驗證**：Enter 在 mixed handler 回傳 nil 後由 upper-level triage 處理，其對「非空 buffer + 非空 composer」的提交行為需逐一追蹤確認。
3. **影響範圍不僅 ETen**：`isPeriodFollowedByToneDigitsOnly` 為通用檢查，Dachen `.3`=ㄡˇ(偶) 等讀音亦會改為 lazy commit，需確認 Dachen 行為不受損。
4. **本次 Phase 範圍**：Phase 103 的提報問題（`192.168.100.1` 在連續輸入時被誤拆）已由 auto-split 修復完全解決。standalone `.digit` 的歧義為混合模式內在特性，非本次提報範圍。

#### 後續行動建議

若事主欲追究 standalone `.digit` 的即時消歧，可考慮於後續 Phase 評估 lazy commit 方案：
- 對 `isPeriodFollowedByToneDigitsOnly(fullInput)` 為 true 的情境，跳過 full-phonetic 第一分支（eager insert），改走第二分支（keep in buffer）。
- 重新設計 Izanami 測試斷言以容忍 lazy commit 行為（assembler 可能為空、讀音在 composer）。
- 全面驗證 Enter/Space/候选選擇/游標移動等路徑。
- 評估是否僅限 ETen parser 或通用於所有靜態佈局。

### 已知限制

- 若 ETen 使用者刻意以「數字前綴 + 純數字注音後綴」進行混合輸入（如欲打 `1` + `嗯`），此修復會阻止該拆分，整段保留為 ASCII。但此情境極罕見且高度歧義（與數字序列無法區分），保留為 ASCII 為更合理的預設行為。
- `.digit`（句點+聲調數字）在 buffer 為空時仍會走整段注音路徑產生注音讀音（如 `.1`=ㄔ˙、`.2`=ㄔˊ）。這是 mixed mode 的設計行為，非 bug。華碩輸入法能直接處理此情境（可能採用 lazy/deferred commit 架構），vChewing 的 eager commit 架構暫無法做到，需架構層面評估（見上方「延遲提交構想」）。
- Phase 100 已知的 ETen `ai` 前綴與 `ello` 前綴限制仍然存在，與本次修復無關。

## 施術目標

重述 `Phase104_PreResearchedSpecs.md` 的產品需求：

Homa 的 candidate revolver 目前僅支援 hard revolve，輪替時會把候選列表中 segLength 最大者直接覆寫到組字區，即使該候選跨越了鄰近「已被使用者明確覆寫（explicit）」的節點也照樣覆蓋。這破壞了「一個字一個字選」的使用者體驗：

- 情境一（`ji4-gong1`）：使用者先將 `ji4` 選成「濟」（explicit），再對 `gong1` 輪替；hard revolve 首輪即把「濟」回退成「技工」，需再輪一次才得「濟公」。
- 情境二（real-world `你 現在 要 搬 的話`）：使用者選了「要」後輪替「斑」→「搬」，「要」被回退，結果變回 `你 現在 耀斑 的話`。

實證根因（已於 PreResearchedSpecs 釘死）：損害一律來自「多音節候選覆寫跨越鄰近 explicit 節點」——`calculateNextCandidateIndex` 的 non-explicit 分支回傳索引 0（segLength 最大候選），`overrideCandidate` 的 targetRange 跨越鄰近 explicit 節點並覆蓋之。鞏固非兇首（情境一鞏固被跳過、情境二鞏固活躍，結局相同）。

施術內容：
1. Homa `revolveCandidate` 新增 `softRevolve: Bool = false` 參數；`softRevolve == true` 時在候選選擇層做「安全子集」過濾（剔除 segLength > 1 且覆寫範圍涵蓋非當前節點之 explicit 節點的候選），再在安全子集上計算輪替索引。不動鞏固邏輯。
2. Typewriter 端：`revolveCandidate(reverseOrder:)` 透傳 `softRevolve`；凡由 SPACE 鍵（無論修飾鍵有無）啟動的輪替一律傳 `softRevolve: true`；其餘入口（Tab、ContextMenu、方向鍵、方括號）維持 hard。

```swift
/// Check whether any given flag is being hold.
/// - Parameter flags: Given flags. If empty, this API will return false.
/// - Returns: Bool result.
public func isHoldingAny(_ flags: KBEvent.ModifierFlags) -> Bool {
  guard !flags.isEmpty else { return false }
  return !keyModifierFlags.isDisjoint(with: flags)
}

/// Check whether all given flags are being hold.
/// - Parameter flags: Given flags. If empty, this API will return false.
/// - Returns: Bool result.
public func isHoldingAll(_ flags: KBEvent.ModifierFlags) -> Bool {
  guard !flags.isEmpty else { return false }
  return keyModifierFlags.contains(flags)
}
```

本次手術同時牽涉到 `vChewing-macOS` 倉庫。

## 要解決的問題

> 該 Phase 的意圖與 Phase 108 一致，但手術對象不是 app-bundle-based installer，而是 pkg installer。

目前這個倉庫的 PKG Installer 是由 WhiteBox Packages App 創建的。但這就導致了所有步驟必須得藉由 GUI 完成、無法自動化處理。

請創建一個 shell 腳本利用 pkgbuild 重現 `vChewing.pkgproj` 所處理的事務。這個腳本可以是 Apple Script 或 perl 或任意的 macOS built-in 語言。注意：

- 如果使用者沒有用管理員權限運行該安裝包的話，則嚴禁其喚起管理員權限，除非確實有偵測到 `/Library/Input Methods` 有可疑物件需要更名。
- 如果使用者有使用管理員權限運行安裝包的話，仍得注意將內容安裝包安裝到當前使用者的 `~/Library/Input Methods` 目錄、且該使用者對此有全部控制權。
