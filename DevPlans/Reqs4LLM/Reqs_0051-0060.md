## Phase 52 實作結果（2026-04-28, KIMI-K2.6）

### 根因結論

經徹查，**TextMapTrie 本身無缺陷**，`VanguardTextMap` 產物亦正確收錄了假名與鴨蛋零條目。

故障根因位於 **下游 `vChewing-macOS` / `vChewing-OSX-Legacy` 的 `LMInstantiator` 查詢邏輯**。

具體來說：

1. `char-misc-nonkanji.txt` 的內容在 VanguardLexicon 建構時被歸類為 `typeID = 8`（`nonKanji`）。
2. 下游 `LMInstantiator` 的 `CoreColumn.theDataMISC` 對應 `textMapTypeIDs = [4, 8]`，同時涵蓋 `letterPunctuations`（4）與 `nonKanji`（8）。
3. 但 `unigramsFor` / `hasUnigramsForFast` 中，對 `.theDataMISC` 的查詢被條件 `keyChain.hasPrefix("_")` 限制——只有 `_` 開頭的特殊 key（如標點、半形標點）才會查詢 MISC。
4. 這導致普通讀音 key（如 `ㄚ`、`ㄌㄧㄥˊ`）下的 `nonKanji` 條目（假名、鴨蛋零）永遠不會被返回。

該 regression 始於 Tag 4.3.3 引入 `VanguardTextMap` 作為原廠辭典格式時；舊版 SQLite 或其他格式下這些內容可能通過不同路徑被查詢。

### 實作策略

修復以「**拆分 nonKanji 查詢路徑**」落地：

1. **新增 `CoreColumn.theDataNonKanji`**（`textMapTypeIDs = [8]`），專門對應 `nonKanji` 內容。
2. **在 `unigramsFor` 中**，對所有普通讀音 key 加入 `.theDataNonKanji` 的 `factoryUnigramsFor` 查詢，與 `_` 開頭的 MISC 查詢互不影響。
3. **在 `hasUnigramsForFast` 中**，同步加入 `.theDataNonKanji` 的快速可用性檢查。
4. **保留既有行為**：`_` 開頭的 key 仍走 MISC 路徑（包含 typeID 4 + 8），普通讀音 key 新增的 nonKanji 查詢不會混入 `letterPunctuations` 結果。

#### 需求

在 mixed Chinese-English typing fallback 模式下：

#### 根因結論

根因不在 Session 或 IMK event bridge，而是在 **`MixedAlphanumericalTypewriter` 自身把所有 Option-modified 輸入一律提前排除**：

1. mixed handler 在進入 ASCII / phonetic 決策前，存在 `guard !input.isControlHold, !input.isOptionHold, !input.isCommandHold else { return nil }`。
2. 這使所有 `Alt+主鍵盤 ASCII` 輸入根本無法進入 mixed handler 的有效決策區，先前只補 `Alt+主鍵盤區數字鍵` 的窄分支，已不足以覆蓋修訂後需求。

4. 對字母 / 數字 / 符號的 Alt 組合而言，不能直接採用 macOS event 提供的 glyph，因為那通常是 Option-modified 的 locale-dependent 字元；本需求要的是忽略 Option glyph 汙染、但保留 Shift 對 ASCII base/shift variant 的影響，因此必須以 keyCode + `LatinKeyboardMappings` 反推原始 ASCII。

1. 在 `MixedAlphanumericalTypewriter` 的 `Space` 分支之後、一般標點/CJK punctuation 判斷之前，新增 `Option+主鍵盤 ASCII` 的 direct-commit fast path。
2. 以 `inferredLatinKeyboardLayout().mapTable[keyCode]` 統一處理主鍵盤字母鍵、數字鍵、符號鍵：不按 Shift 時取 base ASCII，按住 Shift 時取 shifted ASCII。

## Izanami 測試設計原理

`test_IH400_MixedAlnumKanjiInputTest_Izanami` 的設計哲學借鑑「伊邪那美」——一旦存在系統性缺陷，測試會像無限迴圈一樣窮舉所有觸發條件，直到根本修復為止。

**運作原理**：

1. **參數化前綴**：`@Test(arguments: ["", "ai", "AI"])`
   - `""`（empty prefix）：驗證 basic typing——純注音序列能被 composer 正確吸收、注音確認後可提交中文。
   - `"ai"`：小寫字母前綴，本身為合法注音序列（大千 `a=ㄇ, i=ㄛ`），最容易被 composer 誤吸收。
   - `"AI"`：大寫字母前綴，buffer 為空時會被阻斷，但 mixed buffer 非空時仍可能被捲入。

2. **窮舉覆蓋**：對每個前綴，遍歷 `lmPlainBopomofo.sortedKeys` 中**所有**有單字條目的讀音。對每個讀音，將其對應的 raw key sequence（由 `reading.backwardQWERTYDachenMapping` 產生）拼接在前綴之後，模擬「連續鍵入 prefix + reading keys → Enter 提交」的完整使用者流程。

3. **核心斷言（曝光機制）**：
   - **Empty prefix**：Enter 後 `mixedAlphanumericalBuffer` 必須為空（composer 已吸收），且 `recentCommissions` 必須非空（中文成功提交）。
   - **Non-empty prefix**：Enter 後 `recentCommissions.joined()` 必須包含前綴文字。若前綴被 composer 誤當注音吸收（如 `aijo6` 被當成 `ㄇㄧㄛㄐ`），則 commissions 不會出現 `"ai"`，斷言失敗，該 reading 被記入 failure report。

4. **聲調區分**：對無聲調 reading（如 `ㄨ`），測試會在末尾追加空白以觸發注音確認；對有聲調 reading 則直接提交。這確保兩種注音確認路徑都被覆蓋。

5. **為何稱之為 Izanami**：過去對 mixed mode 的修復往往是「補一個邊界 case」，但新 case 立刻冒出。Izanami 測試從源頭窮舉所有 reading，任何會導致前綴被吸收的邏輯缺陷都無所遁形——正如伊邪那美一旦啟動，中招者會在幻術中反覆經歷同一命運，直到真正改變為止。

### 根因分析

Izanami 測試所揭露的故障核心為：**MixedAlphanumericalTypewriter 在處理「ASCII 前綴 + 注音後綴」的 mixed 輸入時，`isFullyParserCovered` 純注音路徑與 auto-split 排序策略導致前綴被過度吸收。** 經三輪迭代，共定位八類機制缺陷：

1. **純注音路徑過度吞噬長輸入**：當 `fullInput` 超過單一注音音節最大鍵位數時，`trialComposer.receiveSequence(fullInput)` 只能保留最後一個音節，前面的鍵位被無聲覆寫。但舊程式碼仍會在 `hasIntonation()` 與詞庫命中時直接提交，導致前面的 ASCII 前綴被誤吞。
2. **auto-split 排序偏向高機率長後綴**：舊排序以 `bestProbability` 為主、suffix length 為輔，使得 `aicj4` 中 `icj4`（ㄏㄨㄛˋ，機率較高）勝過 `cj4`（ㄏㄨˋ），導致 prefix 從 `ai` 被削減為 `a`。
3. **Space 確認分支缺乏非 word-like fallback**：Space 分支的 auto-split 僅嘗試 `requiresWordLikePrefix: true`，對於 `ai5j/ ` 這類 prefix='ai' 不符合 word-like 規則的輸入，直接落入 BPMFFullMatchTypewriter，導致 buffer 被清空而前綴丟失。
4. **`hasUnigramsForFast` 過濾 ETenDOS 條目**：非 word-like fallback 下的 auto-split 因 `hasUnigramsForFast` 拒絕了僅存在於 ETenDOS 的讀音（如 ㄨˇ、ㄏㄨˋ），使得合法 suffix 被誤判為無效。
5. **Tekkon auto-correct 干擾 suffix 判斷**：`buildAutoSplitCandidate` 使用 `trialComposer.receiveSequence(suffixText)` 時，`phonabetCombinationCorrectionEnabled = true`（預設）會自動修正注拼組合。這導致 `zj/4` 被修正為與 `z/4` 相同的 reading key（ㄈㄥˋ），deduplication 後保留較短的 `z/4`，使得 `aizj/4` 被誤拆為 `aiz` + ㄈㄥˋ，而非 `ai` + ㄈㄨㄥˋ。
6. **`shouldPreferASCIIWordPath` 誤傷 2-char 前綴**：`aihjp` 被判定為 ASCII word path（2 次覆寫），導致 Space 確認時 bypass auto-split，`aiq ` 被誤提交為 ASCII。
7. **整段注音路徑優先於 auto-split**：正常輸入路徑先嘗試 `isFullyParserCovered` 整段注音路徑，再嘗試 auto-split。這導致 `xu.6`（suffix 長度 2）的 auto-split 被整段路徑阻擋，無法正確拆出 `xㄨˋ`。
8. **`hasNoDestructiveOverwrite` 分支未檢查詞庫命中**：`aiu3`（大千 `a=ㄇ, i=ㄧ, u=ㄛ, 3=ˇ`）恰好佔滿 composer 四個槽位（聲、介、韻、調），使得 `hasNoDestructiveOverwrite = true`。舊程式碼將此視為「整段輸入可安全保留為單一注音音節」的充分條件，直接保留為 reading key `ㄇㄧㄛˇ`。但 `ㄇㄧㄛˇ` 在詞庫中無任何條目，導致 assembler 無法插入、輸出為空，且 auto-split 被提前返回阻斷。此問題僅發生於 `ai` prefix（小寫字母），`AI` prefix 因 `fullInputHasUppercase` 被排除於 `isFullyParserCovered` 之外而不受影響。

## Phase 57 合理性評估 & Tekkon 效能瓶頸分析 (by Deepseek-v4-pro)

### 一、Phase 57 方案評估：
Instruments 軌跡顯示的絕對熱點是：

```
cnvPhonaToHanyuPinyin → StringProtocol.swapping  [7.46G, 14.9%]
  └─ RefCounts::doDecrementSlow + _swift_release_dealloc  [4.39G, 8.8%]
```

兩者合計佔總 CPU 的 **23.7%**，全部是 **String 操作**（反覆建立/釋放中間字串）。而 `Composer` struct 本身的複製開銷**完全沒有出現在熱點中**——Composer 只有 ~100 bytes 左右的 stored properties（4 個 `Phonabet` + 1 個 `String` romajiBuffer + 1 個 enum + 1 個 Bool），複製成本在 Instruments 中根本看不見。

把 struct 換成 MainActor class，也就是 Shiki Suen 的原始提案，所帶來的實際影響：
- ❌ 不會減少 `cnvPhonaToHanyuPinyin` 的呼叫次數
- ❌ 不會減少 String 分配
- ❌ 會引入 reference counting 開銷（class 是 reference type）
- ❌ `MainActor` 會限制 Composer 只能在 main actor 使用，破壞現有 `Sendable` 語意
- ❌ `MixedAlphanumericalTypewriter` 中的 `var trialComposer = handler.composer` 從 cheap copy 變成 retain

**結論：Shiki Suen 的原始提案對解決實際效能問題無效。**

### 二、真正的效能瓶頸（依嚴重程度排列）

#### 🔴 瓶頸 1：`cnvPhonaToHanyuPinyin` 的 O(N×M) 字串替換（14.9% CPU）

`arrPhonaToHanyuPinyin` 有大約 **250 對**對映，`cnvPhonaToHanyuPinyin` 對每一對呼叫一次 `String.swapping()`。`swapping()` 內部把字串轉成 `Array(unicodeScalars)`、逐字元比對、拼接結果。對於一個 3 字元的輸入（如 `ㄓㄨㄥ`），就是 250 次 × O(3) 的字元陣列比對與字串拼接，產生約 250 個中間 `String`。

這可以改為 **單次掃描的 dictionary-based 字元對映**：用 `Dictionary<UnicodeScalar, String>` 做 O(1) 查表，或對多字元組合用 Trie。將 O(N×M) 降為 O(N)。

#### 🔴 瓶頸 2：`updateRomajiBuffer()` 在 trial 上下文中無意義地被呼叫（17.1% CPU）

`receiveKey(fromPhonabet:)` **無條件呼叫** `updateRomajiBuffer()` → `cnvPhonaToHanyuPinyin`。但在 auto-split 的 trial composer 中，`romajiBuffer` **永遠不會被讀取**——它只用於 `getInlineCompositionForDisplay()`（組字區顯示）。

**修復建議**：將 `romajiBuffer` 改為 lazy/dirty-flag 模式，或提供一個 `receiveKey(fromPhonabet:updateRomaji:)` 參數讓 trial 路徑略過 romaji 更新。

#### 🟡 瓶頸 3：auto-split 迴圈中重複建立 trialComposer 並呼叫 `receiveSequence`（27.8% CPU）

`bestAutoSplitCandidate` 對每個 suffix length (1..maxSyllableKeyCount，最多 6) 呼叫 `buildAutoSplitCandidate`，每次都：
- `var trialComposer = handler.composer`（struct copy）
- `trialComposer.clear()`
- `trialComposer.receiveSequence(effectiveSuffixText)`（對每個字元呼叫 `receiveKey` → `receiveKey(fromPhonabet:)` → `updateRomajiBuffer()`）

Phase 58 建議的「trialComposer 固定在一個 nullable pointer 上」是有道理的，但這不要求把整個 Tekkon 改成 class。只需在 `MixedAlphanumericalTypewriter` 層面用 `inout` 參數傳遞、或重用單一 composer 變數（mutating struct），避免每個 suffix length 都重新複製。

#### 🟡 瓶頸 4：`receiveSequence` 的非 romaji 模式逐字元處理

```swift
givenSequence.forEach { receiveKey(fromString: $0.description) }
```

`$0.description` 每個字元都會建立一個新的 `String`。可以直接迭代 `unicodeScalars`。

#### 🟢 瓶頸 5：`String.swapping` 本身實作偏重

`swapping()` 把整個字串轉成 `Array(unicodeScalars)`，然後逐字元比對。正確的做法是使用 `String.range(of:)` 做一次定位再替換，或直接用 `replacingOccurrences(of:with:)`（Foundation）。

### 三、對 Phase 57 的修正建議

Phase 57 應該**重新定位**為：

1. **不要**做 struct→class 轉換。Composer 保持 struct + `Sendable`。
2. **優先修復** `cnvPhonaToHanyuPinyin` 的 O(N×M) 字串操作 → 改為單次掃描的 dictionary mapping 或 Trie lookup。
3. **讓 `updateRomajiBuffer()` 延遲計算**：只在 `romajiBuffer` 被實際讀取時才呼叫 `cnvPhonaToHanyuPinyin`。
4. **在 `MixedAlphanumericalTypewriter` 層面重用 trial composer**：避免 auto-split 迴圈中重複複製/清除/重建。

這四項修改的總效能提升預期遠大於將 Composer 改為 class。

Phase 56 並未完全解決問題。Prefix 為 `Hello` `hello` 時仍舊仆街，出現 **讀音被扯斷** 與 **讀音沒被判斷成可以發音的讀音** 的情況。Izanami 測試揭露 `Hello` 與 `hello` 各 **182 筆 failure**（而 `""` / `"ai"` / `"AI"` 均為 0 筆）。

Shiki Suen 的推想：
- 對於 `MixedAlphanumericalTypewriter` 的 `ascii漢字` 拆分機制恐需徹底檢討。
- 實時偵測 prefix 的 length 增長與 trialComposer.value() 的 length 增長——一旦不同步，則根據最大可能碼長判定是否拋棄此情形，篩除因 Tekkon key-override-based absorbing 而導致的 `thePrefix 被撕開`。

## 根因分析

`Hello`/`hello` 在 Dachen 鍵位佈局下所有五個字元都是合法注音鍵位。`o`=ㄟ（vowel, slot 3）是關鍵——當後接 `j`=ㄨ（semivowel, slot 2）＋聲調時，composer 形成完整音節（`oj`→ㄟ+ㄨ→ㄨㄟˇ），觸發 auto-split 將 `o` 從 prefix 中「撕開」。

根本矛盾在於兩個典型案例有相反需求：

| 案例 | 正確拆分 | 錯誤拆分 |
|------|---------|---------|
| `Helloj3` | Hello + j3 (ㄨˇ) | Hell + oj3 (ㄨㄟˇ) |
| `Hellosu3` | Hello + su3 (ㄋㄧˇ) | Hellos + u3 (ㄧˇ) |

任何單一排序準則無法同時覆蓋兩者。

## 解決方案：三管齊下

### 1. Tekkon `enforceCSVTOrdering`

**洞察**：`ㄨ`/`ㄧ`/`ㄩ` 被歸類為 **semivowel**（slot 2），`ㄟ` 是 **vowel**（slot 3）。不正確 suffix `oj3` 中 vowel(slot3) 先填、semivowel(slot2) 後到——違反聲介韻調自然順序。正確 suffix `j3` 則是 semivowel→intonation，符合順序。

**實作**：為 `Composer` 新增 `enforceCSVTOrdering: Bool` 屬性。在 `receiveKey(fromPhonabet:)` 內檢查 phonabet 的 slot 順序：若新 phonabet 要填入的 slot 比已填最大 slot 更早，拒絕（`return false`）。聲調（slot 4）永遠允許。

同時將所有 `receiveKey` overload（`fromPhonabet`/`fromScalar`/`fromString`/`fromCharCode`）改為 `@discardableResult -> Bool`，並修改 `receiveSequence` 的 `forEach` 為 `for` 迴圈以在 CSVT 拒絕時提前終止。

僅在 trial composer 中啟用（`buildAutoSplitCandidate` 內設定 `enforceCSVTOrdering = true`），不影響實際輸入。

### 2. Space handler 長 buffer 保護

Space 鍵的 `shouldPreferASCIIWordPath` 啟發式會將 "helloq" 判定為英文詞，跳過 auto-split 直接提交為 ASCII。新增條件：當 buffer >= 5 字元時，即使 `shouldPreferASCIIWordOnSpace` 為 true 也嘗試 auto-split，以支援長 ASCII prefix 的混輸情境。

### 3. Subset-superset 過濾

當多個 suffix 候選存在 subset/superset 關係（如 `u4` 是 `qu4` 的後綴），捨棄 subset，僅保留 superset（更完整的讀音）。這解決如 `helloqu4` 中 `u4`（ㄧˋ，subset）擊敗 `qu4`（ㄆㄧˋ，superset）的問題。

## 涉及檔案

   - `Tekkon_SyllableComposer.swift`：`enforceCSVTOrdering` 屬性 + CSVT 檢查 + `receiveKey` 系列 `@discardableResult -> Bool` + `receiveSequence` for-loop 化

```swift
// 在 `if input.isSpace {` 之前插入：
```

### 二、Task 2：Popup Composition Buffer 位置未隨 auto-commit 更新

#### 根因分析

追蹤 `applyAutoSplitCandidate` 的完整呼叫鏈：

```
applyAutoSplitCandidate()
  ├─ session.commit(text: priorChineseText)          // ① async
  ├─ handler.assembler.insertKey(readingKey)
  ├─ let textToCommit = prefixText + overflow        // prefixText = "Hello"
  └─ session.switchState(.ofInputting(textToCommit:))
       └─ switchState() [InputSession_HandleStates.swift:27]
            ├─ commit(text: textToCommit)             // ② async
            └─ updateCompositionBufferDisplay()       // ③ SYNC → lineHeightRect 查詢
```

`commit(text:)` 內部（InputSession_HandleStates.swift#L127）：

```swift
asyncOnMain(bypassAsync: !isServingIMEItself || UserDefaults.pendingUnitTests) {
    doCommit(buffer)  // client.insertText(...)
}
```

在正式服務 IME 客體時（`isServingIMEItself == true`），`bypassAsync` 為 `false`，因此 `insertText` 被包在 `Task { @MainActor in ... }` 中**非同步執行**。

時序如下：

| 步驟 | 執行緒 | 說明 |
|------|--------|------|
| 1 | Main Actor sync | `switchState` 呼叫 `commit(text: "Hello")`，建立 `Task` |
| 2 | Main Actor sync | `updateCompositionBufferDisplay()` → `lineHeightRect(u16Cursor: 0)` → **此時 IMK cursor 尚未移動**（insertText 未執行） |
| 3 | Main Actor sync | `pcb.show(at: staleOrigin)` → **浮動組字窗顯示在舊位置，遮住將被遞交的 "Hello"** |
| 4 | Main Actor sync | `switchState` 返回 |
| 5 | Main Actor next job | `Task` 內 `insertText("Hello")` 執行 → IMK cursor 前移 |

**關鍵缺陷**：步驟 5 之後，**沒有任何機制重新呼叫 `updateCompositionBufferDisplay()`**。PCB 停留在步驟 3 設定的舊位置，直到使用者敲下一個鍵為止。在 VSCode（Electron）環境中，由於其 IMK 處理管線較長，該位置偏差會特別明顯。

#### 施工方案

有三種方案，依推薦度排序：

**方案 A（推薦）：在 `applyAutoSplitCandidate` 中將 ASCII prefix commit 改為同步執行**

在 `MixedAlphanumericalTypewriter` 內部直接呼叫 `client.insertText` 繞過 `commit(text:)` 的非同步包裝，確保 commit 在 `updateCompositionBufferDisplay()` 之前已確實完成。

具體做法：在 `SessionCoreProtocol` 的 extension 中新增一個 `bypassAsync` 版本的 commit（不改變既有協定要求）：

```swift
// Shared/Sources/Shared/Protocols/SessionCoreProtocol.swift extension
public func commit(text: String, clearDisplayBeforeCommit: Bool, bypassAsync: Bool) {
    // 與既有 commit 相同邏輯，但將 asyncOnMain 的 bypassAsync 參數暴露出來
}
```

然後在 `applyAutoSplitCandidate` 中：

```swift
// 將既有 async commit 替換為：
session.commit(text: priorChineseText, clearDisplayBeforeCommit: false, bypassAsync: true)
// ...
session.commit(text: prefixText, clearDisplayBeforeCommit: false, bypassAsync: true)
// 再切換至 inputting（此時 textToCommit 為空或僅含 commitOverflownComposition）
var inputting = handler.generateStateOfInputting()
inputting.textToCommit = handler.commitOverflownComposition  // 不含 prefixText
session.switchState(inputting)
```

**優點**：最小侵入、不改動既有 commit 語意、commit 順序可預測。
**缺點**：需新增 protocol extension method；需確認 synchronous insertText 在 IMK 中的行為一致性（已知在 main actor 上直接呼叫 `insertText` 是安全的）。

**方案 B：在 switchState 返回後異步重定 PCB 位置**

在 `applyAutoSplitCandidate` 的 `session.switchState(inputting)` 之後，以 `DispatchQueue.main.async` 延遲呼叫 `updateCompositionBufferDisplay()`：

```swift
session.switchState(inputting)
DispatchQueue.main.async {
    session.updateCompositionBufferDisplay()
}
```

**優點**：不改變 commit 語意。
**缺點**：可能產生一幀閃爍（PCB 先出現在舊位置再跳至新位置）；依賴 `DispatchQueue.main.async` 的執行時機，在不同 macOS 版本間行為可能不一致。

**方案 C：commit 後不立即顯示 PCB，待下一按鍵再更新**

不變更 commit 邏輯，接受一幀的位置偏差。在下一個按鍵事件中，`switchState` 會自然觸發 `updateCompositionBufferDisplay()`，此時 IMK cursor 已更新，PCB 位置正確。

**優點**：零程式碼變更。
**缺點**：未解決問題；VSCode 中該幀偏差可能持續較長時間（因 Electron 的 IME 處理延遲）。

```swift
  for lineIndex in keyEntry.startLine ..< endLine {  // ~275,000 次內層總迭代
    let line = extractString(from: data, start: start, end: end)  // ← String 分配
    let parsedEntries = parseValueLine(line, ...)                  // ← 完整解析
    for parsedEntry in parsedEntries {
      // 收集 ideographic characters
    }
  }
}
```

每個瓶頸點：

1. **隨機訪問 VALUES 區段**：外層依 key 迭代，內層以 `lineIndex` 跳躍取值。Data 雖是 mmap'd，但跳躍訪問仍造成 CPU cache miss 與 potential page fault。

2. **每行 String 分配**：`extractString` 每次呼叫 `String(decoding: data[start..<end], as: UTF8.self)`，對 275,000 行各分配一個 Swift String — 這在 Instruments 中會是有意義的 heap churn。

3. **每行完整 `parseValueLine`**：`parseValueLine` 做完整的 tab-delimited 解析、Entry 結構體建構、機率反查。但 reverse lookup 只需要**每個 entry 的 value 字串中的 ideographic 字元**，不需要機率、不需要 Entry 結構。

**三者疊加**：在 Intel Mac 上，對 ~275,000 行 VALUES 做隨機訪問 + String 分配 + 完整解析，耗時可達 1-3 秒。async loading 雖將此移至背景佇列，但 `factoryTrie` 為 nil 期間所有原廠辭典查詢均返回空，輸入法在此視窗內無法提供正常候選。

### 四、能否與 `parseKeyLineMapContent` 直接耦合？

**不能。** 兩函式操作不同的檔案區段：`parseKeyLineMapContent` 讀 KEY_LINE_MAP（檔案末段），`buildReverseLookupTable` 讀 VALUES（檔案中段）。合併意味著需要在掃描 KEY_LINE_MAP 的同時跳到 VALUES 區段取值，這不但無法減少 I/O，反而會讓 KEY_LINE_MAP 的 sequential scan 變成 random access，適得其反。

### 五、正確的優化策略

#### 策略 A：單次 sequential VALUES pass + 輕量解析（推薦）

**核心思路**：不逐 key 跳躍訪問 VALUES，而是在 `parseKeyLineMapContent` 完成、`buildLineOwnerIndex` 建立之後，做 **一次** sequential pass over VALUES（行 0 → 行 N-1）。用 `valueLineToKeyEntryIndex[lineIndex]` 查得 owning key，再決定是否提取 ideographic 字元。

```swift
// 取代現有 buildReverseLookupTable 的 sequential-pass 版本
var charToLineIndices: [String: [Int]] = [:]
for lineIndex in 0 ..< valueLineOffsets.count {
    let keyEntryIndex = Int(valueLineToKeyEntryIndex[lineIndex])
    guard keyEntryIndex >= 0 else { continue }
    let keyEntry = keyEntries[keyEntryIndex]
    let readingKey = extractString(from: data, start: keyEntry.keyStart, end: keyEntry.keyEnd)
    let segmentCount = readingKey.split(separator: separator).count

    // 輕量解析：只提取 ideographic chars，不建構完整 Entry
    let start = valueLineOffsets[lineIndex]
    var end = lineIndex + 1 < valueLineOffsets.count
        ? valueLineOffsets[lineIndex + 1] : valuesEndOffset
    while end > start, data[end - 1] == 0x0A || data[end - 1] == 0x0D { end -= 1 }

    let chars = extractIdeographicCharsForReverseLookup(
        from: data, start: start, end: end,
        isTyping: isTyping, defaultProbs: defaultProbs,
        includeGroupedTypingLine: line.first == "@" && segmentCount == 1
    )
    for char in chars {
        charToLineIndices[char, default: []].append(lineIndex)
    }
}
// ... subsequent sort + dedup unchanged
```

**效益**：
- 訪問模式從 random → sequential（cache-friendly，減少 page fault）
- 若進一步實作 `extractIdeographicCharsForReverseLookup` 以 byte-level 解析（掃描 tab、typeID、value 的 UTF-8 bytes，直接用 `UnicodeScalar.properties.isIdeographic` 過濾），可**完全消滅每行的 String 分配與 Entry 建構**

**保守估計加速**：3× – 10×（取決於是否 byte-level parse）

#### 策略 B：將 VALUES 行偏移掃描與 reverse lookup 合併

目前 `scanValueLineOffsets`（步驟 1）與 `buildReverseLookupTable`（步驟 4）兩次遍歷 VALUES。可合併為一次：

1. 先 `parseKeyLineMapContent` → `buildLineOwnerIndex`（此時尚無 `valueLineOffsets`）
2. 單次 sequential pass over VALUES，同時產出 `valueLineOffsets` 陣列與 reverse lookup map

但這需要 `valueLineOffsets.count` 來分配 `buildLineOwnerIndex` 的結果陣列。解決方法：先用 `Data` 快速計數行數（`countNewlines`），或者讓 `buildLineOwnerIndex` 改用 dictionary-based sparse mapping 而非 dense array。

**效益**：消除一次完整的 VALUES 遍歷。但 `scanValueLineOffsets` 本身極快（byte-level newline scan），合併的邊際收益遠小於策略 A 的 random→sequential 轉換。

#### 策略 C：惰性建構 reverse lookup table

不於 init 時建構完整 reverse lookup table。改為：
- 首次 `reverseLookup(for:)` 被呼叫時，用 `valueLineToKeyEntryIndex` 做一次 sequential scan 提取結果並快取
- 或使用 Bloom filter 加速 negative lookup（告訴 caller "此字元一定沒有 reverse entry"）

**效益**：init 時間降為接近零；代價是首次 reverse lookup 會較慢。
**風險**：reverse lookup 用於 candidate window 中顯示讀音提示，若首次查詢延遲過高會影響 UX。但可透過在 idle 時預熱解決。

## Phase 60 實作結果（2026-05-05, Deepseek-v4-Pro）

### 實作策略（最終版）

以策略 A（sequential VALUES pass + byte-level 輕量解析）落地，涵蓋五項變更：

1. **`KeyEntry.segmentCount` 預計算**：
   - `KeyEntry` 結構新增 `segmentCount: UInt8` 欄位。
   - `parseKeyLineMapContent` 在構建 `keyInitialsIDMap` 時同步計算每個 reading key 的 segment count，避免 reverse-lookup 階段重複解析 reading key。
   - 在 sequential pass 中直接讀取 `keyEntry.segmentCount` 判斷 `isSingleSegment`，無需分配 `String`。

2. **Byte-level `collectReverseLookupCharsFromLine`**：
   - 新增函式直接於 `Data` 位元組層級掃描 VALUES 行，以 tab 定位、首字元判斷格式（A/B/C/legacy），僅針對含 ideographic 字元的 cell 做最低限度解碼。
   - 處理 grouped-cell 的 `|` 分隔與 `\` 轉義：快速路徑（無特殊字元）直接走 plain-cell 解碼；有轉義時逐位元組掃描、跳過轉義對、僅解碼多字節 UTF-8 ideographic scalars。
   - 額外提供 `utf8SequenceLength`、`decodeUTF8Scalar`、`parsePositiveInt32` 等 zero-allocation helper。

3. **Sequential-pass `buildReverseLookupTable`**：
   - 外層從 `for keyEntry in keyEntries`（55K 次外層 + 隨機訪問）改為 `for lineIndex in 0..<valueLineOffsets.count`（275K 次 sequential）。
   - 以 `valueLineToKeyEntryIndex[lineIndex]` 查 owning key，用預計算的 `segmentCount` 決定 `includeGroupedTypingLine`。
   - 排序與去重邏輯保持不變。

4. **`buildReverseLookupTable` 簽名擴充**：
   - 新增加 `valueLineToKeyEntryIndex` 參數（static 函式無法直接存取 instance property），init 呼叫處同步傳入。

5. **三倉庫同步**：LibVanguard → `vChewing-macOS` 的 `LangModelAssembly` TrieKit → `vChewing-OSX-Legacy` 的 LMAssembly TrieKit，以直接複製完整檔案達成對位。
