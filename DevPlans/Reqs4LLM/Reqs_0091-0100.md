## 問題現狀

本次的問題全部與 `LMAssembly.LMCassette` 有關。

1. `LMCassette` 目前以 `Data` 的形式貯存資料，可能會受到 Darwin 平台「強制 fallback 到 NSData API」所帶來的效能問題。或許可以檢討是否以 UInt8 陣列的方式來貯存資料，以提升效能。我記得此前有 Phase 針對 TextMapTrie 做過類似的處理、換取了顯著的效能提升。
2. `LMCassette` 或許可以檢討 `quickDefMap` 與 `quickPhraseMap` 這兩個 property 的處理方式、改用 contiguous-memory index。原因：單元測試裡面給的 CIN2 檔案特別小。實際上的檔案會很大，比如這個臨時檔案：`./vChewing-macOS/tmp/ar30-big-v2026-1.04-20260530.cin2`。

## 術前方案

### 前置調研

#### 現有架構

`LMCassette` 的資料層現分兩類：

- **已遷移至 contiguous-memory index**（`CassetteSortedMap` / `CassetteOctagramMap` / `CassetteOctagramDividedMap`）：以 `rawData: Data` + sorted byte-range entries + 二分搜尋取代 Dictionary。
- **仍為 Dictionary**：`quickDefMap: [String: String]`、`quickPhraseMap: [String: [String]]`，以及 `keyNameMap: [String: String]`（~30 entries，保留無妨）。

所有 contiguous-memory 型別均以 `Data` 貯存原始 UTF-8，byte-level 比較輔助方法（`cassetteCompareUTF8Range`、`cassetteCompareUTF8RangePrefix`）直接使用 `Data` subscript（例如 `self[range.lowerBound + i]`），**未繞過 Darwin 平台 `Data` → `NSData` 橋接**。

#### 規模量化

以 `ar30-big-v2026-1.04-20260530.cin2`（178,579 行）為基準：

| 區段 | 行範圍 | 約略條數 | 現有結構 |
|------|--------|----------|----------|
| `%keyname` | 18–59 | ~31 | `[String: String]` 保留 |
| `%quick` | 60–991 | ~930 | `[String: String]` **→ 需遷移** |
| `%chardef` | 992–? | ~29k | `CassetteSortedMap` ✅ |
| `%octagram` | ?–118208 | ~60k | `CassetteOctagramMap` ✅ |
| `%quickphrases` | 118209–178579 | ~60k | `[String: [String]]` **→ 需遷移** |

`quickPhraseMap` 在實機 CIN2 中可達 6 萬條，每條獨立的 `String` key + `[String]` value 陣列的 heap allocation 量可觀。

#### Data → [UInt8] 的前例

Phase 66（Reqs_0061-0070.md）曾對 `TextMapTrie.rawData` 測試 `Data → [UInt8]` 遷移，結論為「無改善」——因為 `TextMapTrie` 內部 byte 操作已繞過 `Data` subscript 橋接。**`LMCassette` 的情況不同**：二分搜尋每步迭代都走 `Data` subscript，大型 CIN2 的 3 萬條 chardef / 6 萬條 quickphrases 的查詢累積橋接開銷不可忽略。

### 手術項目

#### 項目 A：`Data` → `[UInt8]`

將所有 contiguous-memory 型別的 `rawData: Data` 改為 `rawData: [UInt8]`。

**受影響型別**：
- `CassetteSortedMap.rawData`
- `CassetteOctagramMap.rawData`
- `CassetteOctagramDividedMap.rawData`
- `Data` extension 上的 `cassetteCompareUTF8Range` / `cassetteCompareUTF8RangePrefix` → 改為 `[UInt8]` extension

**變更要點**：

| # | 位置 | 現有 | 目標 |
|---|------|------|------|
| A1 | 型別宣告 | `var rawData: Data = .init()` | `var rawData: [UInt8] = []` |
| A2 | Builder 初始化 | `var rawData = Data(capacity: totalBytes)` | `var rawData = [UInt8]()` + `rawData.reserveCapacity(totalBytes)` |
| A3 | Builder 附加 bytes | `rawData.append(contentsOf: key.utf8)` | 同（`[UInt8]` 原生支援 `String.UTF8View`） |
| A4 | Byte 比較 | `rawData.cassetteCompareUTF8Range(...)` | 移為 `[UInt8]` extension；`self[i]` 直接存取，無橋接 |
| A5 | String 解碼 | `String(decoding: rawData[start..<end], as: UTF8.self)` | 同（`ArraySlice<UInt8>` 可傳入 `String(decoding:as:)`） |

**預期收益**：消除二分搜尋與 prefix scan 中每 byte 比對的 `Data → NSData` 橋接。對 `charDefMap`（3 萬條）、`quickPhraseMap`（6 萬條）的查詢熱路徑影響最大。

#### 項目 B：`quickDefMap` → contiguous-memory index

建立新 contiguous-memory 型別 `CassetteQuickMap` 用於 key→single value 對照。

**型別設計**（仿 `CassetteOctagramMap`，但 value 為 UTF-8 byte range 而非 Int）：

```
CassetteQuickMap {
  rawData: [UInt8]
  entries: [CassetteQuickEntry]  // keyStart, keyEnd, valueStart, valueEnd
}
```

- Binary search 查詢：`valuesFor(key:) -> String?`
- Builder：`build(from: [String: String])`，依 key UTF-8 排序後寫入 contiguous rawData

**使用端變更**：
- `quickDefMap` 型別：`[String: String]` → `CassetteQuickMap`
- `quickSetsFor(key:)` 中 `quickDefMap[key]` → `quickDefMap.valuesFor(key:)`
- `open()` 中 parsing 階段保持 `var tmpQuickDef: [String: String] = [:]` → 最終 `quickDefMap = .build(from: tmpQuickDef)`
- `clear()` 中 `quickDefMap.removeAll(...)` → `quickDefMap = .init()`

#### 項目 C：`quickPhraseMap` → contiguous-memory index

`quickPhraseMap` 的結構 `[String: [String]]` 與 `CassetteSortedMap`（key→[values]）完全吻合，可直接複用該型別。

**使用端變更**：
- `quickPhraseMap` 型別：`[String: [String]]` → `CassetteSortedMap`
- `quickPhrasesFor(key:)` 中 `quickPhraseMap[key]` → `quickPhraseMap.valuesFor(key:)`
- `open()` 中 parsing 階段保持 `var tmpQuickPhraseMap: [String: [String]] = [:]` → 最終 `quickPhraseMap = .build(from: tmpQuickPhraseMap)`
- `clear()` 中 `quickPhraseMap.removeAll(...)` → `quickPhraseMap = .init()`

### 測試影響

三項手術均不改變 `LMCassette` 的 public API：
- `quickSetsFor(key:)`、`quickPhrasesFor(key:)`、`unigramsFor(key:)` 回傳型別不變
- `CassetteSortedMap.valuesFor(key:)`、`containsKey(_:)`、`prefixScan(prefix:)` 介面不變
- 現有 `LMCassetteTests` 的三個 test case（wubi86, array30, quickphrases_multi）預期全部通過

只需確認型別變更後測試中的 access pattern（`.isEmpty`、`.count`）仍可編譯——`CassetteQuickMap` 和 `CassetteSortedMap` 均已暴露這些屬性。

### 施術順序

1. **項目 C**（quickPhraseMap → CassetteSortedMap）— 複用既有型別，風險最低
2. **項目 B**（quickDefMap → CassetteQuickMap）— 新增一個簡單型別，結構仿 OctagramMap
3. **項目 A**（Data → [UInt8]）— 影響範圍最廣但全為內部實作變更，最後執行以便 git bisect 隔離問題

### 風險

| 風險 | 等級 | 緩解 |
|------|------|------|
| `[UInt8]` 結構體賦值時觸發陣列複製 | 低 | 查詢路徑為唯讀，不觸發 CoW；`open()` 為一次性建構，成本可控 |
| `String(decoding: [UInt8], as:)` vs `Data` 版行為差異 | 極低 | 兩者接受 `Collection<UInt8>`，簽章相同 |
| Legacy 版與 macOS 版的 struct 語法差異（`nonisolated`） | 低 | 分別確認兩個 repo 的宣告語法；Legacy 不使用 `nonisolated` |

- `%ename`：以 `;` / `:` 分拆，從 `Wubi:en;五笔:zh_CN` 取出 `Wubi`

#### 效能最佳化

`open()` 熱路徑中，`cells` 保持為 `[Substring]`（零複製，共享行緩衝），僅對實際取用的 cell 做 `.trimmingCharacters(in: .newlines)` 取得 `String`。避免對每一行所有 cells 做強制 `String` heap 分配，對於 17.8 萬行的 CIN2 檔案可省去數十萬次不必要的分配。

## 施術目標

GSICandidates 在 expandable 且 unexpanded 状态时，任何可能間接觸發展頁的 **滑鼠滾輪** 操作都請僅展頁、不 flip line。

> 該施術目標僅與 **滑鼠滾輪** 操作有關，且不影響 TDKCandidates。

## Phase 92 實作結果（2026-06-04, Deepseek-v4-Pro）

### 實作策略

當 GSICandidates 的候選池為 expandable（`_maxLinesPerPage > 1`）且 unexpanded（`isExpanded == false`）時，使用者以滑鼠滾輪捲動會透過 `handleMouseScroll` 或 Option+scroll 路徑呼叫 `showNextLine()` / `showPreviousLine()`，進而呼叫 `consecutivelyFlipLines` → `expandIfNeeded` 先觸發展頁，但現有邏輯在展頁後仍繼續 flip line，導致展頁同時跳行。

修法：在 `CtlCandidateGSI4AppKit` 的兩個滑鼠滾輪入口點——`scrollWheel(with:)`（處理 Option+scroll）與 `handleMouseScroll(deltaX:deltaY:)`（處理一般滾輪）——分別在呼叫 `showNextLine()` / `showPreviousLine()` 之前，檢查 `isExpandable && !isExpanded`。若為真，則直接呼叫 `expandIfNeeded(isBackward:)` + `updateDisplay()` 並 early-return，不再執行後續的 flip line 邏輯。

此改法僅影響滑鼠滾輪路徑，不影響鍵盤觸發的翻頁/翻行（`showNextPage` / `showPreviousPage` / 鍵盤熱鍵等）、不影響 TDKCandidates、也不影響展開後（`isExpanded == true`）的滾輪行為。

### 程式碼變更

**兩個 repo 的 `CtlCandidateGSI4AppKit.swift` 均已同步完成以下變更**：

| 變更位置 | 變更內容 |
|------|------|
| `scrollWheel(with:)` | 在 Option+scroll 的 `showNextLine()` / `showPreviousLine()` 呼叫前，新增 `isExpandable && !isExpanded` 判斷，若命中則呼叫 `expandIfNeeded(isBackward:)` + `updateDisplay()` 並 early-return |
| `handleMouseScroll(deltaX:deltaY:)` | 在 `showNextLine()` / `showPreviousLine()` 兩個 case 內，各新增相同判斷與 early-return 邏輯，以 Phase 92 註解標記 |

## 手術範圍

本次手術同時牽涉到 `vChewing-macOS`、`vChewing-OSX-Legacy` 兩個倉庫。但在早期調查階段僅拿 `vChewing-macOS` 作為實驗田。等鞏固了手術結果之後再同步到 `vChewing-OSX-Legacy`。

## 手術目的

檢討 Homa 引擎的檢索效率。必要的話得重構 Homa 引擎與 LMInstantiator 之間的資料交換介面設計。

## 先行研究與發現

### 一、`insertKeys()` warmup 用完整 `unigramsFor()` 做 key 存在性檢查

`Homa.Assembler.insertKeys()`（`Homa_Assembler.swift` L169-180）在插入 key 之前，會對每個 alt key 調用 `queryGrams()` 來確認「這個讀音有沒有任何結果」。`queryGrams()` miss 時觸發完整 `LMInstantiator.unigramsFor()`（原廠 Trie + 使用者片語 + 磁帶 + DateTime + InputToken 展開 + 濾除 + consolidate + 排序）。

然而 LMInstantiator 已有輕量級 `hasUnigramsForFast()`（Phase 44 設計，僅做 O(1) hash lookup），且已透過 `LookupHub.hasGrams(for:)` 暴露。瓶頸在於 Assembler 只持有 `gramQuerier: ([String]) -> [GramRAW]` 這個 closure，無法存取 `hasGrams`。

→ **每次 `insertKeys()` 都對每個 key 做完整查詢，只為了確認 existence。**

### 二、Gram ↔ GramRAW 的無意義來回轉換

`LMInstantiator.unigramsFor()` 回傳 `[Homa.Gram]`（完整物件），但 `LookupHub.grams(for:)` 將其 `.map` 為 `[GramRAW]` 元組（丟棄 `previous`、`backoff`、`id`），然後 `Homa.queryGrams()` 再用 `Homa.Gram($0)` 從元組重建 Gram 物件。

→ **每次查詢經歷 Gram→拆散→重建，純屬 overhead。** 若 `gramQuerier` 型別直接定為 `([String]) -> [Homa.Gram]` 可省去整段轉換。

### 三、同一批資料被多次排序

- `unigramsFor()` L827：`rawAllUnigrams.consolidate()` 後 `sort { $0.probability > $1.probability }`（Phase 77 加入，修復「在/再」排序 bug）
- `queryGrams()` L539：`gramQuerier(keyArray).sorted(by: sortGramRAW)`（按 keyArray.count > probability > lexicographic）
- `queryGramsForAlternatives()` L503：Cartesian product 合併後再次 `sorted`（同 `sortGramRAW` 邏輯）

`unigramsFor()` 的排序對非 Homa 路徑（直接調用 `unigramsFor()` 的少數場景）是必要的，但對 Homa 主路徑而言，後續 `queryGrams()` 的 `sortGramRAW` 使用不同 primary key（keyArray.count），會推翻前一輪排序。

### 四、`assignNodes()` 內本地快取與 instance 快取功能重疊

`assignNodes()` 內有 `var queryBuffer: [GramQueryCacheKey: [Homa.Gram]] = [:]` 作為本地快取，而 `queryGrams()` 內部另有一層 instance 級 `gramQueryCache`。`queryGrams()` 先查本地、再查 instance、miss 才真正查詢，結果同時寫入兩層。

在一次 `assignNodes()` 內，不同 `(position, length)` 組合查出的 `keyArray` 極少重複，本地 cache 的主要收益僅是省下一次 `gramQueryCache` 字典查詢。

### 五、Phase 77 排序的依賴關係

Phase 77 在 `unigramsFor()` 的 `consolidate()` 之後加入的 `sort { $0.probability > $1.probability }`，解決了使用者片語 prepend（`.reversed()`）後導致 `consolidate` 出陣列順序不按 score 排列的問題（典型 case：`ㄗㄞˋ` 的「再」排在「在」前面）。該排序不能移除。

### 六、`queryGramsForAlternatives()` 的 Cartesian product 路徑

當 `alternativesSlice` 中有 `.multipleKeys` 時，走笛卡爾積展開路徑，每個 combination 各自調用 `queryGrams()`。Phase 39 已在 LMInstantiator 側引入 `factoryChoppedUnigramsFor` 將原廠端查詢量降為 O(1 per trie column)，但 Homa 側的展开仍在 Assembler 層級進行。

## 當前 Phase 決議處理的正式手術條目

處理剛才發現的這三個問題：

1. 「`insertKeys()` warmup 用完整 `unigramsFor()` 做 key 存在性檢查」。但注意檢討 LMInstantiator 的專屬的在庫檢查 API 是否能真正勝任其職責（而不受 filter table 的影響），且在發現 LMInstantiator 的這種 API 有這種問題的時候修理這些問題。
2. 「Gram ↔ GramRAW 的無意義來回轉換」。
6. 「`queryGramsForAlternatives()` 的 Cartesian product 路徑」。

## Phase 95 最終實作結果（2026-06-07, Deepseek-v4-Pro）

### 設計收斂：`GramQuerier = ([PossibleKey]) -> [Gram]`

最終型別簽章從 `([String]) -> [GramRAW]` 改為 `([PossibleKey]) -> [Gram]`。Homa 不再做任何 String 拼接（`&` joining）或 Cartesian product，直接將 `[PossibleKey]` 透傳給 LMInstantiator。LMInstantiator 內部：
- **原廠 Trie**：使用既有的 `factoryChoppedUnigramsFor` / `factoryChoppedCoreUnigramsFor`（chopped path），在 trie 層級處理替代鍵展開，O(1 per entry type）。
- **非原廠子模組**（使用者片語、符號、磁帶、DateTime、濾除表）：透過 `expandPossibleKeyArrays` 展開後逐一查詢。這些是 hash-based 資料結構，本質上無法支援替代鍵合併查詢。展開量受 `maxSegLength=4` 封印保護（`5^4=625` 次 hash lookup）。

笛卡爾積從 **trie traversal 層級**（舊版：每個組合一次完整 trie 查詢）降到了 **hash lookup 層級**（新版：每個組合一次 O(1) hash 查詢），這是本 Phase 的核心成果。

竊以為 cassette quick 模式 不需要在選字窗顯示 reading disambiguation。

但這個問題可能犯不著修改與這個 candidate window contents 有關的顯示邏輯。

## Phase 96 實作結果（2026-06-08, Deepseek-v4-Pro）

### 根因確認

`Typewriter_Cassette.swift` 的 `renderQuickSetsIfNeeded` 及 `InputHandler_HandleStates.swift` 的 cassette quick 路徑均將 candidate 的 `keyArray` 設為 selection key index（`"1"`, `"2"`...），而非真正的讀音鍵。這導致 `CandidatePool4AppKit.updateReadingDisambiguation()` 的 Rule 2（池內單字 reading 種類 > 1 → 全顯示）必然觸發，候選窗底部出現無意義的 `"1"` / `"2"` 等文字。

GSICandidates 的 vertical layout scrollbar 的頂端恐需新增 y 方向上的 padding，且 padding 的量應該是 thePool.windowRadius。

## Phase 97 實作結果（2026-06-14, Deepseek-v4-Pro）

### 根因確認

`VwrCandidateGSI4AppKit.swift` 的 `scrollerTrackRect(candidateAreaSize:)` 在 vertical scroller（`isVerticalScroller == true`）的情況下，scroller track 的頂端起點為 `y = thePool.originDelta`（= 5pt）。但視窗背景以 `thePool.windowRadius`（= `originDelta + cellRadius`，macOS 11-25 下為 11pt）做圓角裁切，scroller track 頂端落入圓角區域內，可能在視覺上被裁切或產生不美觀的重疊。

### 修法

在 vertical scroller 的 track rect 計算中，將 track 頂端 padding 從 `originDelta` 改為 `windowRadius`，並同步縮減 track 高度以維持底端與 candidate area 底端對齊：

- `y`: `thePool.originDelta` → `thePool.windowRadius`
- `height`: `candidateAreaSize.height` → `candidateAreaSize.height - (thePool.windowRadius - thePool.originDelta)`（即扣掉 `cellRadius`）

不影響 `scrollerThumbRect`（thumb 的計算以 track rect 為基礎，自適應調整），不影響 `fittingSize`（scrollbar 仍在既有 candidate area 垂直區間內，無需擴張視窗）。

## 施術目標

根據 `vChewing-LibVanguard/DevPlans/Research/Phase99_PreResearch.md` 的調研結果來改進 vChewing 輸入法的記憶體佔用量測量方式，使得測量結果不會被 neural engine RAM 與 graphics RAM 欺騙。注意：輸入法選單顯示的結果也得是這個資料值、而不是被欺騙的資料值。相關的 l10n 描述應該由「記憶體」改為「私有記憶體」（涉及所有語言）。vCLog 的描述應該是「PRIVATE MEMORY」而非「MEMORY FOOTPRINT」（也不要用外行看不懂的「ANONYMOUS MEMORY」）。

（輸入法的自盡的私有記憶體佔用量閾值仍保持為 1_536 MiB。）

## Phase 99 實作結果（2026-06-21, Deepseek-v4-Pro）

### 歷程

| 嘗試 | API | 結果 | 原因 |
|------|-----|------|------|
| 1 | `task_vm_info.internal` | ≈ `phys_footprint`（148.95 MB） | `internal` 是匿名私有記憶體的子集，但因該程序 graphics/neural ledger 為零，值與 `phys_footprint` 無異 |
| 2 | `proc_pidinfo(PROC_PIDTASKINFO)` | 編譯失敗 | `proc_taskinfo` 無 `pti_private_pages` 欄位 |
| 3 | `proc_pid_rusage` → `ri_resident_size` | 99.64 MB（比 `phys_footprint` 更大） | `resident_size` 含共享頁面（dyld cache 等） |
| 4 | `proc_pid_rusage`（Optional 雙指標 ABI） | Stack canary crash | Swift 匯入器將 `rusage_info_t *` 扭曲為 `UnsafeMutablePointer<rusage_info_t?>?`，Optional layout 與 C `void *` 不同 |
| **5** | **`phys_footprint − ledger_tag_graphics − ledger_tag_neural − ledger_purgeable_nonvolatile`** | **148.95 MB（ledger 均零時 ≈ phys_footprint）** | ✅ 採用 |

### 最終實作

`memoryFootprintAnonymous` 從 `task_vm_info` 取 `phys_footprint`，再扣除三個 kernel ledger 欄位：
- `ledger_tag_graphics_footprint` — Liquid Glass / WindowServer GPU 表面
- `ledger_tag_neural_footprint` — Neural Engine 配置
- `ledger_purgeable_nonvolatile` — 可 purge 的非揮發頁面

**邏輯**：當 Liquid Glass 繪製中，這些 ledger 欄位會被 kernel 寫入非零值，自動從 `phys_footprint` 中扣除，使洩漏監測不受干擾。當它們為零（Liquid Glass 未繪製），結果即等於 `phys_footprint`——這已是公開 API 能給出的最誠實答案。

### 已知限制

Activity Monitor「專用記憶體」（實測 117.4 MB vs `phys_footprint` 148.95 MB）來自 kernel ledger `tkm_private`，僅可透過 `sysmond`（private framework）存取，**無 public API**。差額 ~31 MB 為 dyld shared cache / 系統 framework 共享頁面，無法經由任何公開 API 剝離。

## Phase 100 實作結果

共三項：

1. **`buildAutoSplitCandidate()` 標點前綴防禦**（核心修復）：當前綴為長度 1 的純標點符號（無 ASCII 英數字元）時，直接排除該 auto-split 候選。修復 `;`(ㄗ) `,`(ㄓ) `.`(ㄔ) `/`(ㄕ) `'`(ㄘ) 等 ETen 標點兼注音鍵被誤拆為 ASCII 前綴的問題。

2. **`bestAutoSplitCandidate()` word-like 前綴偏好**：當兩個候選中僅一方前綴為 word-like（`[A-Za-z]{3,}[A-Za-z0-9]*`）時，優先保留 word-like 前綴。原 `bothWordLike` / `else` 二路分支改為 `bothWordLike` / `neitherWordLike` / `else` 三路。

3. **`bestAutoSplitCandidate()` 同為 word-like 時的長前綴偏好**：雙方皆 word-like 且概率相同時，偏好較長的前綴（＝較短的後綴），避免 `ello`→`el` 等誤拆。
