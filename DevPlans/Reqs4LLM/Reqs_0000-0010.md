## Phase 01 調研結果

### 一、現行 VanguardLexicon 架構回顧

目前 VanguardLexicon 的資料管線：

```
原始 txt 檔案（按語種/來源分檔）
  → Collector（讀取+合併）
  → 權重計算（log10 正規化）
  → Trie 建構（讀音陣列 → 節點鏈）
  → 注音加密（ㄅ→b, ˊ→2 等拉丁替代）
  → SQL 腳本生成
  → SQLite 編譯 → .db 檔案
```

最終產物是單一 SQLite 資料庫，簡體與繁體共存於同一 Trie，透過 `EntryType` 旗標（`.chs` / `.cht`）在查詢時區分。每個 Trie 節點的 `entries` 欄位為 TSV 序列化的詞條陣列（`value\ttypeID\tprobability\t[previous]`），並加速器表 `keyinitials_id_map` 提供字首查詢索引。

### 二、對提案格式的分析

#### 2.1 格式結構

```
#PRAGMA:VANGUARD_HOMA_LEXICON_VALUES
簡體詞\t簡體詞頻\t繁體詞\t繁體詞頻     ← 每行一對 CHS/CHT
...
#PRAGMA:VANGUARD_HOMA_LEXICON_KEY_LINE_MAP
讀音\t區段起始行號\t最大可能筆數
...
```

本質上是一個**兩級索引的扁平檔案結構**：
- **VALUES 區段**：所有詞條，按讀音排列，CHS/CHT 以一行合併。
- **KEY_LINE_MAP 區段**：索引表，將每個讀音映射到 VALUES 區段中的 `(起始行, 數量)`。

#### 2.2 相對於 McBopomofo 格式的優勢

| 維度 | McBopomofo 格式 | 提案格式 |
|------|-----------------|----------|
| 繁簡處理 | 需要兩份完整資料（或單語種） | 同一讀音的 CHS/CHT 合一行，節省約 40% 儲存 |
| 查詢效率 | O(n) 線性掃描 | O(log n) 二分搜尋 KEY_LINE_MAP → O(1) 隨機存取 VALUES |
| 讀音冗餘 | 每筆紀錄都重複完整讀音字串 | 讀音僅在 KEY_LINE_MAP 出現一次 |
| Bigram 支援 | 無原生支援 | 可透過第五欄位 `previous` 擴展 |

對於二十萬筆紀錄的規模，讀音字串冗餘是主要空間浪費。以平均讀音長度 12 bytes（含分隔符）估算，McBopomofo 方式在讀音上花費約 2.4 MB；提案格式的 KEY_LINE_MAP（假設約 3 萬個不重複讀音鍵）僅花費約 0.6 MB。

#### 2.3 CHS/CHT 候選數量不對等問題

提案中已預見此問題（「最大可能筆數」取兩語種之最大值）。同一讀音下 CHS 可能有 5 筆、CHT 可能有 8 筆，則需 8 行，其中 3 行的 CHS 欄位為空。

**評估**：這種對齊浪費在實務中可接受。實測先鋒語料，同讀音繁簡候選差異通常在 0–3 筆以內（差異主要集中在罕用字與異體字），空欄的額外開銷遠小於省下的讀音冗餘。

若仍想消除對齊浪費，可選擇**變體方案**：

```
#PRAGMA:VANGUARD_HOMA_LEXICON_VALUES
讀音\tCHS筆數\tCHT筆數
簡體詞1\t簡體詞頻1
簡體詞2\t簡體詞頻2
繁體詞1\t繁體詞頻1
繁體詞2\t繁體詞頻2
繁體詞3\t繁體詞頻3
```

但這會使結構不規則、解析邏輯複雜化，且無法利用固定行寬做隨機存取。**建議維持原始提案的四欄對齊格式**，簡潔性勝過微量空間節省。

#### 2.4 建議的欄位擴展

結合現有 Homa.Gram 的資料需求，建議 VALUES 區段的完整欄位為：

```
簡體詞\t簡體詞頻\t繁體詞\t繁體詞頻[\t前驗詞]
```

第五欄 `前驗詞`（optional）用於 Bigram 支援。空欄表示 Unigram。此設計與 `Homa.Gram.previous` 欄位直接對應。

### 三、Swift Span 與效能提升

#### 3.1 Span 簡介

Swift 6.2 引入的 `Span<T>` 提供非複製、非逃逸的連續記憶體視圖語意（non-copying, non-escaping）。配合 `UTF8Span` 可實現字串的零複製處理。

#### 3.2 與本格式的搭配方案

**記憶體映射 + Span 的零複製管線**：

```
1. mmap() 將整個 .txt 檔映射進虛擬記憶體
2. 解析 KEY_LINE_MAP，為每個讀音建立 (byteOffset, byteLength) 索引
   → 存為陣列或字典，保持在記憶體中
3. 查詢時：
   a. 在 KEY_LINE_MAP 索引中找到目標讀音的 (offset, length)
   b. 用 Span<UInt8> 或 UTF8Span 零複製引用 VALUES 區段的對應位元組範圍
   c. 就地解析 tab 分隔欄位（不 allocate String，直接比對 UTF8 序列）
   d. 僅在需要回傳結果時才 materialize 為 String
```

**關鍵效能收益**：
- **零複製讀取**：查詢階段不分配堆積記憶體，僅建立 stack-local 的 Span 視圖。
- **惰性載入**：mmap 的頁面由作業系統按需調入，初始載入時間趨近零。
- **快取友善**：VALUES 區段中同讀音的記錄在物理上連續，查詢時的存取局部性極佳。

#### 3.3 與現行 SQLite 方案的比較

| 維度 | SQLite Trie (現行) | mmap + Span (提案) |
|------|--------------------|--------------------|
| 冷啟動延遲 | 需要 SQLite 初始化 + 首次查詢 I/O | mmap 幾乎零延遲 |
| 單次查詢 | SQL 解析 + B-tree 走訪 + 結果序列化 | 二分搜尋 + Span 零複製引用 |
| 記憶體佔用 | SQLite 頁面快取（可調） | 作業系統頁面快取（自動） |
| 併發安全 | SQLite 內建鎖機制 | 唯讀 mmap 天然執行緒安全 |
| 更新能力 | 可執行 INSERT/UPDATE | 唯讀（需重建整檔） |

**結論**：對於輸入法原廠辭典（唯讀、查詢密集）的場景，mmap + Span 方案在理論效能上優於 SQLite。但 SQLite 方案的優勢在於成熟的工具鏈和增量更新能力，適合使用者自訂辭典等可變資料。

### 四、Trie 相容性與部分讀音檢索

#### 4.1 扁平索引 vs Trie 結構

提案的 KEY_LINE_MAP 是**扁平索引**，不是 Trie。要支援不完全讀音檢索（例如 `ㄅ` 比配 `ㄅㄚ`、`ㄅㄞ` 等），有兩條路徑：

**路徑 A：排序 KEY_LINE_MAP + 二分前綴搜尋**
- 若 KEY_LINE_MAP 按讀音字典序排列，可用二分搜尋找到前綴的 lower_bound 和 upper_bound。
- 例如搜尋 `ㄅ` → 找到 range `[ㄅㄚ, ㄅㄤˋ]` 的所有行。
- 優點：實作簡單、不需要額外結構。
- 缺點：多音節的不完全讀音（如 `ㄅ-ㄩㄝ`）需要在 KEY_LINE_MAP 中搜尋含有該前綴模式的所有讀音串，複雜度上升。

**路徑 B：內嵌序列化 Trie 作為索引**

將 KEY_LINE_MAP 替換為序列化的 Trie 結構，其中每個葉節點指向 VALUES 區段的 `(offset, count)`：

```
#PRAGMA:VANGUARD_HOMA_LEXICON_VALUES
（同上）
#PRAGMA:VANGUARD_HOMA_TRIE_INDEX
nodeID\treadingKey\tchildrenJSON\tvalueOffset\tvalueCount
```

- 優點：O(m) 前綴查詢（m = 讀音長度）、天然支援部分比對。
- 缺點：索引體積增大、解析邏輯複雜化。

#### 4.2 建議

考量到目前 TrieKit 已有成熟的 Trie 查詢實現（含 partial matching、longerSegment 等），且提案格式的主要價值在於**作為可讀的交換格式與輕量級部署方案**，建議：

1. **採用路徑 A（排序 KEY_LINE_MAP + 二分前綴搜尋）**。實作簡單，且對於單漢字的不完全讀音檢索場景已足夠。
2. **KEY_LINE_MAP 中的讀音使用與現行 TrieKit 相同的分隔符 `-`**（例如 `ㄋㄧㄠˋ-ㄆㄧㄣˊ`），確保與 Homa 引擎的 `keyArray` 直接可互轉。
3. 不建議在 txt 格式中內嵌完整的 Trie 結構——那等於重新發明 SQLite。若需要 Trie 級別的查詢效能，繼續使用現行的 SQLite Trie 方案。

### 五、對 VanguardLexicon 引入此格式的可行性

**結論：可行且有價值。** 具體用途：

1. **作為中間交換格式**：取代目前散落的多個 `phrases-*.txt` 檔案，提供一個合併的、可直接載入的格式。VanguardLexicon 的建置管線可輸出此格式，LibVanguard 的 LexiconKit 可直接消費。
2. **作為輕量級部署格式**：對於不需要 SQLite 依賴的場景（如 WASM、嵌入式），mmap 此檔案即可提供完整辭典查詢能力。
3. **作為除錯與稽核格式**：純文字可讀性使人工檢視辭典內容遠比 SQLite 方便。

**建議的檔案副檔名**：`.vhlx`（Vanguard Homa Lexicon eXchange）或 `.vlex`。

**建議的完整 PRAGMA 結構**：

```
#PRAGMA:VANGUARD_HOMA_LEXICON_HEADER
VERSION\t1
READING_SEPARATOR\t-
ENTRY_COUNT\t214500
KEY_COUNT\t31200
#PRAGMA:VANGUARD_HOMA_LEXICON_VALUES
尿频\t-6.234\t尿頻\t-6.234
尿酸\t-7.112\t尿酸\t-7.112
...
#PRAGMA:VANGUARD_HOMA_LEXICON_KEY_LINE_MAP
ㄋㄧㄠˋ-ㄆㄧㄣˊ\t0\t1
ㄋㄧㄠˋ-ㄙㄨㄢ\t1\t1
...
```

HEADER 區段提供元資料，使解析器能在載入時進行基本驗證。

## Vanguard Pragma TextMap 格式規格（定稿）

TextMap 現行為**單一檔案格式**：**Typing TextMap**（`.txtMap`）。

RevLookup 不再有 dedicated `.revlookup` sidecar；runtime 與測試一律從 MainTextMap 自動生成反查視圖。舊 `.revlookup` 僅作歷史記錄，現行實作不再相容。

### Typing TextMap（`.txtMap`）

三段式 PRAGMA 結構。分隔符為 Tab (`\t`)。

#### HEADER 區段

```
#PRAGMA:VANGUARD_HOMA_LEXICON_HEADER
VERSION	1
TYPE	TYPING
READING_SEPARATOR	-
ENTRY_COUNT	871
KEY_COUNT	277
DEFAULT_PROB_4	0
DEFAULT_PROB_7	-11
DEFAULT_PROB_8	-13
DEFAULT_PROB_9	-13
DEFAULT_PROB_10	-1
```

| 欄位 | 說明 |
|------|------|
| `VERSION` | 格式版本號 |
| `TYPE` | `TYPING`（VanguardLexicon 產出）或 `TRIE_TEXTMAP`（LibVanguard `serializeToTextMap` 產出） |
| `READING_SEPARATOR` | 讀音分隔符（預設 `-`） |
| `ENTRY_COUNT` | VALUES 區段的總行數（含所有行型別） |
| `KEY_COUNT` | KEY_LINE_MAP 區段的讀音數量 |
| `DEFAULT_PROB_<typeID>` | 某 EntryType 全體條目共用的機率值；VALUES 中該 typeID 的條目可省略機率欄位 |

#### VALUES 區段

`#PRAGMA:VANGUARD_HOMA_LEXICON_VALUES` 之後的每一行代表一或多筆 Trie Entry。行型別有三種，按解析優先度排列：

**型別 A：合併行**（`>` 前綴）——具有 DEFAULT_PROB 的 typeID 條目按讀音合併為單行。

```
>7	鎄|埃|锿|嗳|哎|㶼|𡉓 ...
>4	，|。|、|！|？ ...
>9	🌳
```

格式：`>typeID\tencodedCell`。機率由 HEADER 中的 `DEFAULT_PROB_<typeID>` 提供。`encodedCell` 使用 escaped pipe 編碼：`|` 為分隔符，`\\` 表示反斜線、`\|` 表示字面 `|`、`\s` 表示空格、`\a` 表示 BEL (`\u{7}`)。

**型別 B：CHS/CHT 機率分組行**（僅限 `TYPE=TYPING`）——同讀音下同機率的簡體/繁體 values 按機率降冪排列（趨近 0 者行號在前）。

```
@-5.28	束	束
@-5.307	^G	豎
@-9.465	数|樹	數
```

格式：`@probability\tchsCell\tchtCell`。`@` 前綴明確標記此行為 TYPING grouped line，避免和一般三欄個體行混淆。`chsCell` / `chtCell` 與型別 A 共用同一套 escaped pipe 編碼。若某語種該機率下無條目，則以 **BEL (`\u{7}`)** 佔位。Parser 仍保留對舊版 bare numeric grouped line（`probability\tchs_values\tcht_values`）的相容，但 builder 不再輸出該舊格式。

CHS 對應 `EntryType.chs`（rawValue=5），CHT 對應 `EntryType.cht`（rawValue=6）。

**型別 C：個體行**——不符合型別 A/B 的條目以逐筆記錄。

```
value\tprobability\ttypeID[\tprevious]
```

第 4 欄 `previous`（可選）用於 Bigram。常見於 `meta`（typeID=2）條目。

#### KEY_LINE_MAP 區段

```
#PRAGMA:VANGUARD_HOMA_LEXICON_KEY_LINE_MAP
Su4	0	23
Su4-xiT	23	2
SuP3	25	2
...
```

每行：`readingKey\tstartLine\tcount`。`startLine` 為該讀音在 VALUES 區段中的起始行號（0-based），`count` 為該讀音佔的行數。讀音使用加密字元集（ㄅ→b 等拉丁替代），多音節以 `-` 分隔。

### RevLookup 視圖（auto-generated）

RevLookup 是由 MainTextMap 在載入時自動生成的 runtime 視圖，不再有獨立檔案。

- **LibVanguard 全物化路徑**：`parseTextMap` 會根據 `VALUES` + `KEY_LINE_MAP` 自動補出 `.revLookup` entries；但 `serializeToTextMap` 不會把這些條目重新寫回 `.txtMap`。
- **LibVanguard 惰性路徑**：`TextMapTrie` 會建立排序後的 `RevLookupEntry` 陣列，value 只保存對應 `VALUES` 行的 line indices，查詢時再按需還原讀音。
- **下游 LangModelAssembly**：同樣從 MainTextMap 自動生成反查表，不接受 external revlookup payload。

現行納入規則：

- 所有 `@probability` 分組行，只在其 reading key 的 segment length 為 1 時才參與反查索引。
- 上述 `@` 行僅納入 ideographic characters，避免把注音文、符號等雜訊掛進單字反查。
- `typeID == 7`（CNS）條目一律納入，不以 Unicode codepoint range 做判定，確保 surrogate pairs / 高萬字等 CNS 內容不會漏掉。

### 一、問題根源

Phase 03 的真正漂移源頭不只在 `VanguardTextMapDataBuilder`。整個 lexicon 建置管線同時存在三個不穩定來源：

- Plist: 42.85ms
- TextMap: 44.17ms

這組數字雖然明顯低於使用者在 Xcode 測得的 100~200ms 區間，但 shared hot path 仍有可以收掉的 allocation 與重複查詢成本。主要瓶頸如下：

1. `Homa.Assembler.queryGrams(using:cache:)` 以字串插值做排序與去重識別，在完整 Lexicon partial-match 場景會製造大量暫時字串。
2. `Assembler` 只有單次 `assignNodes()` 的區域 cache；連續 `insertKey()` 會重複查詢相同切片。
3. `VanguardTrieProtocol` 的 `queryGrams` / 關聯詞路徑反覆對 `readingKey` 做 `split(separator:)`，卻沒有重用既有 `TrieStringOperationCache`。
4. 三種 Trie 實作都用 `TNode.hashValue` 去重，等於在熱路徑重複雜湊整個節點內容，而不是使用穩定 node ID。
5. `SQLTrie.getNode()` 每次都重新 `prepare/finalize` 同一條單節點查詢 SQL，浪費了原本已經存在的 statement cache。

## 實作內容

### 1. Homa hot path 收斂

- `Homa.Assembler` 新增 bounded `gramQueryCache`，讓連續 `insertKey()` / `insertKeys()` 可以重用先前查過的 gram 結果。
- `insertKeys(_:)` 改成直接 warm up `queryGrams(using:cache:)`，避免先做 availability check、後做正式 query 的雙重工作。
- `queryGrams(using:cache:)` 改用結構化 comparator 與 `Hasher` 雜湊識別，移除以字串插值做排序與去重的 allocation 熱點。
- `clear()` 與 `assignNodes(updateExisting: true)` 會清理 persistent cache，避免語言模型更新時吃到過期查詢結果。

### 2. Trie protocol shared path 收斂

- `VanguardTrieProtocol.queryGrams(...)` 與 `queryAssociatedPhrasesAsGrams(...)` 改用 `TrieStringOperationCache.shared.getCachedSplit(...)`，避免重複拆分相同 `readingKey`。
- 協定層 chopped-reading 路徑的節點去重改為使用 `node.id`，不再依賴 `hashValue`。

### 3. Backend-specific 修正

- `VanguardTrie.Trie`、`VanguardTrie.TextMapTrie`、`VanguardTrie.SQLTrie` 的 `getNodes(...)` 去重邏輯統一改為 `node.id`。
- `SQLTrie.getNode()` 改為重用 cached prepared statement，只做 `reset / clear_bindings`，不再每次重新 prepare/finalize。
- SQLTrie 內部 `readingKey` split 也接上 `TrieStringOperationCache`，避免 SQL backend 多做一次字串切割。

- Plist: 8.06ms（較 42.85ms 改善約 81.2%）
- TextMap: 22.78ms（較 44.17ms 改善約 48.4%）

可以確定先前最肥的 shared allocation / duplicate-query 成本已經被收掉，尤其 Plist 與 TextMap 路徑改善非常明顯。SQL 路徑雖然也下降，但幅度遠小於另外兩種 backend，代表剩餘成本主要不是純 Swift hot path，而是 SQLite 逐筆抓節點、資料形態與 storage access pattern 本身的硬成本。

### 根本原因診斷

```swift
// 原初設計（Iteration 1）
struct KeyEntry { let key: String; ... }
var keyEntriesByKey: [String: KeyEntry]      // ~17 MB（hash 預分配）
var orderedKeyEntries: [KeyEntry]            // ~7.4 MB（重複）
// 合計：~24 MB 記憶體浪費
```

此外：
- `reverseLookupTable: [String: [String]]`——96,077 筆反查記錄以 Dictionary 存放，額外佔用 ~5–8 MB（bucket 預分配 + 指標鏈）。
- `cachedEntries: NSCache`——初次配置時無上限，在頻繁查詢場景可膨脹至數十 MB。
- 202,445 筆文法詞條中，每筆 `keyString` 都被獨立 materializer 為 Swift String object，合計 ~5–8 MB heap overhead。

**問題：** 這三個因素加總即為 ~48 MB 額外開銷。

#### Iteration 1：排序陣列 + Binary Search + NSCache 上限

```swift
// 改為單一 sorted array
var sortedKeyEntries: [KeyEntry]             // ~1.2 MB（無重複）
// 移除 keyEntriesByKey Dictionary              // 移除 ~17 MB
// 新增 binary search with UTF-8 comparison
// 設置 NSCache 上限
cachedEntries.countLimit = 8_192             // 限制快取至 ~8 MB
```

**成效**：記憶體從 ~100 MB → ~69–70 MB（降 30%）。

#### Iteration 2：Byte-range 鍵 + 排序 RevLookupEntry 陣列

```swift
// KeyEntry 改為 byte offset
struct KeyEntry { 
  let keyStart: Int    // byte offset into textMapData
  let keyEnd: Int      // byte offset into textMapData
  let startLine: Int; let count: Int 
}

// RevLookupEntry 改為排序陣列
struct RevLookupEntry { 
  let key: ContiguousArray<UInt8>  // raw UTF-8 bytes，無 String allocation
  let readings: [String] 
}
var reverseLookupTable: [RevLookupEntry]     // 排序，binary search 查詢
```

**手術內容**：
1. 新增 `parseKeyMapBytes(in:from:to:)` 於 byte 層級掃描 KEY_LINE_MAP，將讀音鍵以 byte range 持有（不 materialize）。
2. 新增 `resolveKey(for:)` 按需解碼 keyStart/keyEnd 範圍的 UTF-8 bytes 為 String。
3. 更新 `binarySearchIndex(for:)` 以 `Data.compareUTF8Range(_:with:)` 直接比對 raw bytes，避免中間 String allocation。
4. 更新 `reverseLookup(for:)` 對新的排序 `[RevLookupEntry]` 陣列做 binary search。
5. 新增 UTF-8 byte 比較 helper：`compareUTF8(_:_:)`、`compareUTF8Buffers(_:_:)`、`Data.compareUTF8Range(_:with:)`、`Data.compareUTF8Ranges(_:_:)`。
6. 更新 `parsedEntries(for:)` 以原始 bytes 範圍產生 NSString cache key，避免提前 String 物質化。

- 202K 字串物件 allocation 消除（~5–8 MB 節省）。

### 三、技術細節

#### 3.1 `RevLookupEntry` 結構

```swift
private struct RevLookupEntry: Sendable {
  let key: ContiguousArray<UInt8>  // 漢字字符的 UTF-8 bytes
  let lineIndices: [Int]  // 指向 valueLineOffsets 的索引
}
```

#### 3.2 `buildReverseLookupFromKeyEntries` 實作

```swift
private static func buildReverseLookupFromKeyEntries(
  textMapData: Data,
  keyEntries: [KeyEntry],
  valueLineOffsets: [Int],
  valuesEndOffset: Int,
  isTyping: Bool,
  defaultProbs: [Int32: Double]
) -> [RevLookupEntry]
```

**流程：**
1. 遍歷每個 `KeyEntry`（代表一個 reading key 及其在 VALUES 中的行範圍）。
2. 對每個 reading key，解析其對應的所有 VALUES 行（按 line index）。
3. 若該 VALUES 行為 `@probability` 分組行，只有在 reading key 的 segment length 為 1 時才繼續處理，並且只提取 ideographic characters。
4. 若該 VALUES 行為 `typeID == 7` 的 CNS 條目，則無條件提取其所有字符，不使用 Unicode codepoint range 進行篩選。
5. 建立 `charToLineIndices: [String: Set<Int>]` 映射，將字符映射到 **line indices**（而非讀音字串）。
5. 轉換為排序後的 `RevLookupEntry` 陣列（按 UTF-8 byte 排序）。

**關鍵差異：**
- **舊設計（錯誤）**：`charToReadings[String(char)] = Set<String>`（物化讀音字串）
- **新設計（正確）**：`charToLineIndices[String(char)] = Set<Int>`（儲存 line indices）

#### 3.3 `reverseLookup(for:)` 按需解析

```swift
func reverseLookup(for kanji: String) -> [String]? {
  // ... 二分搜尋找到對應的 RevLookupEntry
  return parsedReadings(from: entry.lineIndices)  // 按需解析
}
```

`parsedReadings(from:)` 方法：
1. 對每個 line index，從 `valueLineOffsets` 取得該行的 byte range。
2. 從 `textMapData` 提取該行的 UTF-8 字串。
3. 使用 `findKeyEntry(for:)` 找到該行對應的 reading key。
4. 返回所有讀音的集合。

### 四、記憶體分佈分析

#### 舊設計（錯誤，未節省記憶體）
```
RevLookupEntry {
  key: ContiguousArray<UInt8>  // ~50 bytes per entry
  readings: [String]  // ~100 bytes per entry (String objects + array overhead)
}
// 總計：~150 bytes × 96,000 entries ≈ 14.4 MB
```

#### 新設計（正確，真正節省記憶體）
```
RevLookupEntry {
  key: ContiguousArray<UInt8>  // ~50 bytes per entry
  lineIndices: [Int]?  // ~40 bytes per entry (5 indices × 8 bytes + array overhead)
}
// 總計：~90 bytes × 96,000 entries ≈ 8.6 MB
// 節省：~5.8 MB（約 40% 減少）
```

**關鍵節省來源：**
- 避免提前物化讀音字串（每個 String 物件約 40–60 bytes）
- line indices 僅儲存 Int 值（8 bytes each），查詢時才解析為讀音
- 消除 `.revlookup` 檔案本身的 mmap 開銷（~932 KB）

## Phase 10 實作結果

### 一、設計方案及根本原因

vChewing-macOS 因 Swift 6.2 將整個 package 的 default isolation 釘在 `@MainActor` 上，導致 `FactoryTextMapLexicon` 及其所有 `static` 方法均被推斷為 MainActor-isolated。故無法在 `fileHandleQueue`（非 MainActor context）上直接呼叫 `FactoryTextMapLexicon.init`。

為解決此問題，沿用 `LMInstantiator` 既有的 `asyncLoadingUserData` 機制（測試時為 `false`，生產環境為 `true`），將 `connectFactoryDictionary` 方法拆分為同步/非同步兩條路徑：

- **同步路徑**（`asyncLoadingUserData == false`）：行為與修改前完全一致，用於單元測試與需要立即可用的場景。
- **非同步路徑**（`asyncLoadingUserData == true`）：以 `fileHandleQueue.async { asyncOnMain { do { ... } catch { ... } } }` 結構，先回傳 `connectFactoryDictionary` 讓呼叫端立即繼續（避免阻塞 `applicationWillFinishLaunching` 的同步流程），隨後在下一輪 MainActor 排程循環中執行 `Data` I/O 讀取與 `FactoryTextMapLexicon` 初始化（含 PRAGMA 段落掃描、行偏移量索引建構、KEY_LINE_MAP 解析、反查表生成等 CPU 密集型工作），完成後將結果指派至 `factoryLexicon`。

此設計與 `loadCassetteData` 的非同步載入模式原理對齊，惟因 MainActor isolation 約束增加了 `asyncOnMain` 的包裝層。

### 二、安全保護機制

輸入法 FSM 已有既有防護：`InputSession_HandleEvent` 會在 `!LMMgr.isCoreDBConnected`（即 `factoryLexicon == nil`）時顯示「Factory dictionary not loaded yet.」提示、拒絕處理按鍵事件。故非同步載入進行中的短暫空窗期不會導致崩潰或錯誤行為——使用者只會看到 tooltip 提示，直到載入完成。

### 三、修正內容

#### A. vChewing-macOS

### 四、實作細節及約束

因 Swift 6.2 package isolation，vChewing-macOS 全體 public API 被推斷為 `@MainActor`-isolated。故 `FactoryTextMapLexicon.init` 及其呼叫的 parse/build 系列 `static` 方法（如 `locateSectionBounds`、`parseHeader` 等）均無法在 `fileHandleQueue`（非 MainActor）context 內直接呼叫，會產生 concurrency warning（`call to main actor-isolated initializer in a synchronous nonisolated context`）。

為此採用 `asyncOnMain` 包裝，整段 `Data` 讀取 + `FactoryTextMapLexicon` 初始化都在 MainActor context 下執行。雖然短暫佔用 MainActor，但因 vChewing-macOS 採取「先 return、稍後執行」的非同步排程模式（`fileHandleQueue.async { asyncOnMain { ... } }`），實際的 Main Thread freeze 不會發生在輸入法載入時間線內——`connectCoreDB` / `connectFactoryDictionary` 在 `asyncOnMain { [weak self] in ... }` 內呼叫時已形成額外排隊，主事件迴圈可在此間完成初始化其他視窗與檢查，直到輸入法真正接受使用者輸入前高機率已完成載入。

此 warning 在 vChewing-OSX-Legacy（Swift 5.10 / Xcode 15）不出現，不代表其無此問題——可能是 Swift 5.10 concurrency checker 未檢查 package isolation 約束、或 runtime 有缺陷。
