## 施術後記（2026-04-12，作廢）

本次手術完整實作並通過了所有單元測試（LibVanguard 92 項、LangModelAssembly 51 項、Typewriter 28 項、Legacy `make debug-core` 建置成功），但**實機測試發現記憶體佔用量反而飆升 20 MB ~ 30 MB**，故全數撤銷。

### 失敗根因分析

施術方向在理論層面是合理的（不將整份 .txtMap 常駐記憶體），但實際執行的資料存取模式與 Darwin malloc/VM 的特性相左，導致效果逆轉：

1. **pread() 碎片化配置**：每次 VALUES 行查詢都呼叫 `pread()` 產生一個新的短暫 `Data` buffer，malloc 的 free-list 不會立即將這些釋放後的細碎記憶體歸還給 OS，導致 heap 持續膨脹。這即是使用者所描述「各種 Data 片段無法固定有效利用同一片記憶體區域」的根本成因。

2. **物化讀音鍵的額外開銷**：原始 in-memory 模式的讀音鍵以 byte range 直接指向 `Data`（零額外 allocation）；DFD 模式為了支援 binary search 必須將所有讀音鍵物化為 Swift `String` 陣列（~50,000+ 個 String 物件，每個都有完整的 ARC bookkeeping 與 COW 開銷），本身就造成一筆顯著的額外記憶體負擔。

3. **`Data(contentsOf:)` 的隱性優勢被低估**：Darwin 的 `Data(contentsOf:)` 底層可能走 `mmap()`，OS 可將這些頁面標記為 purgeable / discardable，在記憶體壓力時主動回收；整塊 mmap 記憶體也允許 kernel 的 page sharing。`pread()` 碎片化的 heap allocation 完全無此好處。

4. **NSCache 行為改變**：in-memory 模式下解析後的 VALUE 結果可能讓 NSCache 的 eviction cost 估算更精確；DFD 模式下各快取項目皆為獨立 heap 物件，整體 cache footprint 的行為與原本不同。

### 結論

對這類「大塊單一連續資料、查詢時使用 byte-range subsetting」的場景，`Data(contentsOf:)` 搭配既有的惰性解析架構（只解析 VALUES 行，不物化全部字串）**已是最佳平衡點**。DFD 化（FileHandle + pread）帶來的 heap 碎片與物化開銷超過節省的整塊 Data 空間；本方向不建議再次嘗試，除非改用 OS-level 的 memory-mapped file（`mmap` + `MAP_NOEXCACHE`）並以類 arena 的方式管理查詢 buffer。

## Phase 12 實作結果

### 設計概要

仿照 `FactoryTextMapLexicon` 的 contiguous-memory pattern，將 LMCassette 的大型 Dictionary 轉化為連續 `Data` blob + sorted byte-range index + binary search。初版完成後，因實測載入期 HEAP 峰值仍偏高，於 2026-04-13 追加第二輪修正，將建構流程改為 grouped Dictionary + lightweight prototype grouping，以壓低載入峰值記憶體。

### 新增型別（`extension LMAssembly`）

| 型別 | 用途 | 結構 |
|------|------|------|
| `CassetteSortedMap` | 通用 key→[value] sorted map | `rawData: Data` + `entries: [CassetteMapEntry]`（sorted by key UTF-8）+ `allValues: [CassetteByteRange]`（flat array） |
| `CassetteMapEntry` | key 的 byte range + values 的 index range | `keyStart/keyEnd/valuesStart/valuesEnd: Int` |
| `CassetteByteRange` | 單個 value 的 byte range | `start/end: Int` |
| `CassetteOctagramMap` | key→count (字根八股文) | `rawData: Data` + `entries: [CassetteOctagramEntry]` |
| `CassetteOctagramDividedMap` | key→(count, reading) (音韻八股文) | `rawData: Data` + `entries: [CassetteOctagramDividedEntry]` |

### 資料結構替換

| 屬性 | 舊型別 | 新型別 | 預估 key 數 |
|------|--------|--------|-------------|
| `charDefMap` | `[String: [String]]` | `CassetteSortedMap` | ~23K (嘸蝦米86) |
| `charDefWildcardMap` | `[String: [String]]` | `CassetteSortedMap` | ~8K |
| `symbolDefMap` | `[String: [String]]` | `CassetteSortedMap` | 變動 |
| `reverseLookupMap` | `[String: [String]]` | `CassetteSortedMap` | ~23K |
| `octagramMap` | `[String: Int]` | `CassetteOctagramMap` | ~15K |
| `octagramDividedMap` | `[String: (Int, String)]` | `CassetteOctagramDividedMap` | 變動 |
| `keyNameMap` | `[String: String]` | **不變** | ~30 |
| `quickDefMap` | `[String: String]` | **不變** | 中型 |
| `quickPhraseMap` | `[String: [String]]` | **不變** | <10 |

### 查詢 API

- `CassetteSortedMap.valuesFor(key:)` / `subscript[key]`：O(log n) binary search，按需從 rawData 解碼 UTF-8 字串
- `CassetteSortedMap.containsKey(_:)`：O(log n) 存在性檢查
- `CassetteSortedMap.prefixScan(prefix:)`：O(log n + k) 前綴掃描，取代舊版 `compactMap { $0.key.starts(with:) }` 全表 O(n) 掃描
- `CassetteOctagramMap[key]`、`CassetteOctagramDividedMap[key]`：O(log n) 頻次查詢

### 建構流程

`open()` 解析期間現在只保留必要的 grouped Dictionary：`tmpCharDef`、`tmpSymbolDef`、`tmpOctagram`、`tmpOctagramDivided`。初版的 `tmpReverseLookup` 與 `tmpCharDefWildcard` 兩份大型暫存 Dictionary 已移除，改由 `CassetteSortedMap.buildReverseLookup(charDefs:symbolDefs:)` 與 `CassetteSortedMap.buildWildcard(from:wildcard:)` 直接從 grouped source data 以 lightweight prototype grouping 建構最終索引。`CassetteSortedMap.build(from:)` / `CassetteOctagramMap.build(from:)` / `CassetteOctagramDividedMap.build(from:)` 也改為直接接受 grouped Dictionary，不再經過 `tmpDict.map { ... }` 物化中間 tuple array。

### 2026-04-13 追加修正 by GPT-5.4

初版雖改善穩態資料結構，但實測顯示載入後 HEAP 釋放體感偏慢、峰值記憶體仍高。根因確認為：

- `tmpReverseLookup` 與 `tmpCharDefWildcard` 兩份大型暫存 Dictionary 與 final map 在建構期間同時共存。
- `tmpDict.map { ... }` 額外物化一整份 tuple array。
- builder 初版於 sort comparator 內部反覆建立 `Array(rawData[range])`，在 `O(n log n)` 排序過程產生大量短命 allocation。

本輪修正內容：

- `CassetteSortedMap.build(from:)` 改為直接從 grouped Dictionary 生成 final rawData / entries / allValues。
- `CassetteSortedMap.buildWildcard(from:wildcard:)` 改以 sorted prototype grouping 直接生成 wildcard map，不再建立 wildcard 暫存 Dictionary。
- `CassetteSortedMap.buildReverseLookup(charDefs:symbolDefs:)` 改以 sorted prototype grouping 直接生成 reverse lookup，不再建立 reverse 暫存 Dictionary。
- `CassetteOctagramMap` / `CassetteOctagramDividedMap` builder 同步改為直接接受 Dictionary。

macOS 版使用 `nonisolated private extension Data` 標記 byte-level comparison helper，避免 Swift 6 strict concurrency 下的 main actor isolation 問題。Legacy 版無需此標記。

## Phase 13 實作結果

### 設計概要

本輪最終沒有直接將 `lmAssociates` / `lmCoreEX` / `lmReplacements` 的 `strData` 全面改為 `Data`，而是分兩刀改動三者共用的 parser / tokenizer 熱路徑。理由如下：

- 這三個 SubLM 目前都仍是「可編輯、可熱置換、可回存」的使用者文字資料模組，對外 API 與 Phrase Editor 仍以 `String` 為核心。
- 直接改成 `Data` 雖可讓掃描路徑 byte-based 化，但仍得保留大量 `String` I/O / consolidate / save / retrieve 介面，工程成本高。
- 實測顯示，現行 `String.parse(splitee:)` 的第一個瓶頸主要來自 `Character` 逐字走訪；將其改成 `String.UTF8View` 單 byte 分隔符快路徑即可拿到顯著收益，且保留 `Range<String.Index>` API。
- 進一步 profiling 後又確認第二個瓶頸轉移到每行 `split(separator: " ")`、字串物化與 Dictionary 寫入，因此第二刀改為收斂共用 cell tokenization，而不是把 parser 自 UTF8View 再往全面 `Data` 化推進。

### 施術內容

- `RangeParserAPI.parse(splitee:task:)` 現在對「單一 ASCII / 單 byte UTF-8 分隔符」使用 `String.UTF8View` 掃描；目前三個 SubLM 實際使用的 `"\n"` 皆會走這條快路徑。
- 對非 ASCII separator 則保留原本 `Character` 掃描 fallback，維持既有泛用語義與 API 相容性。
- `RangeParserAPI` 內部新增共用 `parseRanges(in:splitee:task:)`，並對外新增 `parseCells(in:splitee:task:)`，用來在指定行範圍內掃描 space-separated cells、保留既有 `omittingEmptySubsequences` 語義。
- `lmCoreEX` 的 `replaceData()` 與 `unigramsFor()` 改為直接用 `parseCells()` 抽取前 2~3 個 cell，移除每行 `split(separator: " ")` 陣列配置。
- `lmAssociates` 的 `replaceData()`、`valuesFor(pair:)` 與 `dictRepresented` 改為以 `parseCells()` 掃描 cell；值端仍保存 `(lineRange, cellIndex)`，但改為需要時對目標行重掃一次，不再物化整行 token array。
- `lmReplacements` 的 `replaceData()` 與 `valuesFor(key:)` 同步改為 `parseCells()`。
- 上述變更已同步移植到 `vChewing-OSX-Legacy` 對應檔案。
- `task` 參數亦改回同步 non-escaping closure，使簽章更符合實際使用情境。

### 基準結果

以 `vanguardTextMap_test.txtMap` 重複展開為約 13.7 MB UTF-8 文本、使用 `swiftc -O` 測得：

- `String.parse(Character)`：平均約 158.9 ms
- `String.UTF8View scan`：平均約 14.9 ms
- `Data byte scan`：平均約 5.1 ms

結論：`UTF8View` 雖仍慢於純 `Data` byte scan，但已較原本 `Character` 掃描快約一個數量級，且避免全面 Data 化的相容性成本，適合作為本階段最小且有效的改善。

### 追加 profiling（replaceData 熱路徑）

另以 synthetic user-data fixture 模擬三個 SubLM 的 `replaceData()` 載入流程，測得 parser 換成 UTF8View 快路徑後的端到端差異如下：

| 模組 | 舊 Character parser | 新 UTF8View parser | 改善幅度 |
|------|---------------------|--------------------|----------|
| LMCoreEX parse-only | 46.9 ms | 3.9 ms | 約 12.1x |
| LMCoreEX replace-like | 116.7 ms | 112.7 ms | 約 3.4% |
| LMReplacements replace-like | 123.9 ms | 104.7 ms | 約 15.5% |
| LMAssociates replace-like | 184.4 ms | 159.9 ms | 約 13.3% |

這組數據的結論很明確：

- 對 parser 本身而言，UTF8View 掃描確實大幅快於 Character 掃描。
- 但對 LMCoreEX 整體載入而言，真正主要成本已轉移到每行 `split(separator: " ")`、字串物化與 Dictionary 寫入，而不是行分割本身。
- LMReplacements 與 LMAssociates 因每行 token 操作較單純，parser 優化能轉化成更可見的端到端收益。

因此，本輪 Phase 13 沒有停在 parser 最佳化，而是依 profiling 結論正式把下一刀落在「space-separated cells 的 tokenization 與字串物化」；同時仍維持 `String` 為主的外部 API，不推進全面 `Data` 化。

### 第二階段：cell tokenization 收斂

- `parseCells(in:splitee:task:)` 讓三個 SubLM 共享同一套「先取行 range、再取 cell range」的掃描邏輯，避免各自重建 `[Substring]`。
- `lmAssociates` 在這一輪最需要小心：它不只要載入資料，還要在聯想選字時依 `(lineRange, cellIndex)` 回推 value；現行作法是在查值時對該行重掃一次 cell，語義與舊版一致，但不再建立整行 token array。

對本題而言，語義來源不是 vChewing 的 factory lexicon inventory，而是 ButKo 自身的單字讀音順序規格。只要 runtime 手上已經有 `(字, 讀音)`，就能直接查 ButKo slot map；完全不需要反推「這個字音配對有沒有出現在 Vanguard 主辭典裡」。因此 `VanguardLexicon` 在本題不是必改項，`VanguardFactoryDict4Typing.txtMap` 不需要加欄位或混入 BPMFVS 資訊。

### 四、查表使用原始注音，不受拼音偏好干擾

ButKo BPMFVS lookup 的資料源是注音，因此查表時應使用原始 phona key 經正規化後的注音字串。「輸出拼音取代注音」偏好不應介入 lookup 過程。

另外，Typewriter 的 raw key 使用後置輕聲（如 `ㄉㄜ˙`），而 ButKo 資料使用教材式前置輕聲（`˙ㄉㄜ`），查表前需先做正規化。

### 五、組字區顯示投影的追加研判

當偏好設定已經指定 `CommitableMarkupType = bpmfvsAnnotationButKo` 時，使用者會預期「組字區的可見形態」應與最終遞交形態一致。對多音字而言，若組字區仍只顯示裸字、而字型預設採第一讀音的 glyph，會出現語義錯位（例如 backend 是 `ㄅㄚ˙` 的「罷」，畫面卻看起來像 `ㄅㄚˋ` 的「罷」）。因此，**當且僅當 ButKo BPMFVS 遞交模式啟用時，將組字區顯示同步切到 BPMFVS 投影，是合理且自洽的 UX 規格。**

但這個追加需求有非常重要的實作前提：

- **只能做 display-side projection，不能改寫 backend state data。** 原因是 `displayedText` 在多條一般遞交流程裡也被直接當 commit payload 使用（`generateStateOfInputting(sansReading: true).displayedText` 會被 Enter / Esc 某些分支直接取用）。若粗暴地全域換成 BPMFVS 版本，一般 Enter 遞交也會連帶變成 BPMFVS commit。
- **`generateStateOfInputting(sansReading: true)` 必須維持原始字串**，避免普通 Enter / Esc 路徑誤吃投影結果。
- **Display transform 必須和 commit-time transform 共用同一套 ButKo lookup / normalize 規則**，否則顯示與遞交會分岔。
- **Swift 對「漢字 + Variation Selector Supplement」視為單一 extended grapheme cluster**，cursor math 風險低。
- **`isReadingMismatched` 節點仍應維持 fallback，不硬拆。**
- **`generateStateOfCandidates()` 與候選預覽也直接吃 `assembler.assembledSentence.values`**，因此若要保證畫面一致，不能只改 Inputting state；至少還得一併處理候選窗開啟時的 composing buffer preview。

### 2026-04-14 效能追加修正（Phase 18 Follow-up by GPT-5.4）

使用者回報 `ChewingRustDatabaseGenerator.writeArtifacts(...)` 在真實語料上的耗時偏高；本輪重新檢查熱路徑後，確認主要瓶頸並不在 DER 封裝本身，而是在 trie 建構與序列化期間的重複線性掃描與重複排序：

1. **children 改為「插入時即維持有序」**：`findOrInsertInternal` / `resolveLeafID` 不再對 `childIDs` 做線性掃描，而是以 binary search 在 syllable-sorted `childIDs` 內查找與插入。這樣一來，每個 internal node 在 encode 階段也不必再臨時 `sorted()` 一次。
