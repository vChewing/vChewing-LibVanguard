## 手術範圍

本次手術同時牽涉到 `vChewing-macOS`、`vChewing-OSX-Legacy` 兩個倉庫。

## 問題現狀

拼音模式現允許免聲調輸入。然而，田所選字窗缺乏兩個配套的特性：

- 尚未對同字不同調（同字不同音）的 candidates 做出視覺上的 disambiguation。例：「和」的各種讀音。
- 亦尚未對「字詞相同、幅節長度不同」的 candidates 做出視覺上的 disambiguation。例：「🐝」同時對應「ㄇㄧˋ-ㄈㄥ」與「ㄈㄥ」。

> 幅節長度 就是 這個 candidates 的 reading array 的 cell count。

目前的設想的整改方案，對選字窗補充特性：

- 無論選字窗排版是 vertical / horizontal 且無論是否處於 expanded 狀態，都在視窗底部新開一行、專門顯示當前候選字詞的讀音資訊。這個區域的文字樣式繼承 reverse lookup / unicode metadata 區域的文字顯示樣式，但文字透明度降為原先的 80%。當且僅當當前高亮候選字詞具備這種 disambiguation 需求的時候，這一行才會顯示。而且，如果某個讀音 cell 以 `_` 開頭的話（也就是說這個內容不是漢字），則這個讀音 cell 的內容以 `??` 來顯示。

額外需注意：

- 計算讀音 disambiguation 顯示的邏輯需要安置在 CanddiatePool 內，而不是 ViewController 層面。
- 目前發現的故障：在啟用拼音模式（也就是允許省略聲調的模式）時，有些候選字詞（無論實際聲調如何）完全沒有顯示 disambiguated reading。不用在 Session Delegate 層面實作了，改在 TDKCandidate 內部實作。
  - 規則1：直接對所有 same-candidate-text 的 candidates 的 readingArray 是否全等，不要再考慮 `segLength > 1 or not` 了。
  - 規則2：檢查當前 segLength == 1 的 candidates 的 reading array 種類數量。如果超過一種，則對當前 pool 所有 segLength == 1 的 candidates 顯示 reading disambiguation。

## Phase 61 手術方案（實作規格）

### 一、現有架構分析

#### 1.1 資料流

```
CandidateInState (keyArray: [String], value: String)
  │  從 delegate.candidatePairs(conv:) 取得
  ▼
CandidateCellData4AppKit
  │  目前僅存 segLength (= keyArray.count)，未存實際 keyArray 內容
  ▼
CandidatePool4AppKit
  │  持有 candidateDataAll / candidateLines
  │  持有 reverseLookupResult: [String]（底部反查欄資料源）
  │  持有 tooltip: String（底部提示欄資料源）
  │  計算 attributedDescriptionBottomPanes（組合底欄 NSAttributedString）
  │  計算 updateMetrics()（決定 peripherals 的 CGRect 與 fittingSize）
  ▼
VwrCandidateTDK4AppKit
  │  draw(_:) 繪製：背景 → 高亮行/候選字 → 候選字詞 → attributedDescriptionBottomPanes
  ▼
CtlCandidateTDK4AppKit
  │  updateDisplay() 計算 reverseLookupResult / tooltip → 寫入 thePool
  │  然後 thePool.updateMetrics() → NSView.setNeedsDisplay()
```

#### 1.2 現有底欄（Peripherals）組成

`attributedDescriptionBottomPanes` 目前的串接順序：
1. `attributedDescriptionPositionCounter`（位置計數器，如 `3/18`）
2. `attributedDescriptionTooltip`（如「📼」；僅在 tooltip 非空時顯示）
3. `attributedDescriptionReverseLookup`（反查 / Unicode 碼位；僅在 reverseLookupResult 非空時顯示）

所有三個元件都以 `NSTextAttachment` + `RoundedBadgeTextAttachmentCell` 的 badge 形式呈現。

`attributedDescriptionReverseLookup` 的文字樣式基準：
- 字級：`max(ceil(unifiedSize * 0.6), 9)`
- 字色：`CandidateCellData4AppKit.absoluteTextColor`（無透明度衰減）
- 背景：`.clear`

#### 1.3 關鍵佈局差異：Horizontal vs Vertical 模式下的 Peripherals 位置

`updateMetrics()` 中的 `finalContainerOrientation` 決定 peripherals 的放置位置：

- **Horizontal 單行模式**（`maxLinesPerPage == 1 && layout == .horizontal`）：
  peripherals 放在候選字詞的**右側**，垂直居中對齊。
  佈局示意：
  ```
  [cell1] [cell2] [cell3] ... | [pos] [tooltip] [revlookup]
  ```

- **Vertical 模式 / Horizontal 展開模式**（其他所有情況）：
  peripherals 放在候選字詞的**下方**，左對齊。
  佈局示意：
  ```
  [cell1] [cell4] [cell7]
  [cell2] [cell5] [cell8]
  [cell3] [cell6] [cell9]
  [pos] [tooltip] [revlookup]
  ```

這意味著讀音 disambiguation 行**不能**簡單追加到 `attributedDescriptionBottomPanes`——否則在 horizontal 單行模式下它會錯誤地出現在右側而非底部。需要作為獨立元素、始終置於視窗最底部。

#### 1.4 關鍵發現：CandidateCellData4AppKit 缺少 keyArray

在 `CandidatePool4AppKit.construct()` 中：

```swift
var allCandidates = candidates.map {
    CandidateCellData4AppKit(key: " ", displayedText: $0.value, segLength: $0.keyArray.count)
}
```

原始的 `keyArray` 內容被拋棄，僅保留其 `count` 作為 `segLength`。要實現讀音 disambiguation，必須將 `keyArray` 保留在 cell 中。

**E. 新增 `attributedDescriptionReadingDisambiguation` 計算屬性**

樣式基準：繼承 reverse lookup 樣式，但文字透明度降為 80%（`withAlphaComponent(0.8)`）。

注意：此屬性需為 `internal`（非 `private`），因為 `VwrCandidateTDK4AppKit.draw(_:)` 需要直接讀取它以繪製在獨立位置。

```swift
/// 讀音 disambiguation 顯示用的 NSAttributedString。
/// 樣式繼承 reverse lookup 區域，但文字透明度降為 80%。
/// 當 readingDisambiguationResult 為 nil 時返回空字串。
var attributedDescriptionReadingDisambiguation: NSAttributedString {
    guard let readingText = readingDisambiguationResult, !readingText.isEmpty else {
        return NSAttributedString(string: "")
    }
    let readingTextSize = max(ceil(CandidateCellData4AppKit.unifiedSize * 0.6), 9)
    let badgeFont = Self.blankCell.phraseFont(size: readingTextSize)
    let attrReading: [NSAttributedString.Key: Any] = [
        .kern: 0,
        .font: badgeFont,
        // Phase 61: 繼承 reverse lookup 區域的文字顏色，但透明度降為 80%
        .foregroundColor: CandidateCellData4AppKit.absoluteTextColor.withAlphaComponent(0.8),
    ]
    let label = " \(readingText) "
    let fontSize = CGFloat(readingTextSize)
    let referenceHeight = positionCounterBadgeHeight()
    let contentHeight = badgeContentHeight(label: label, attributes: attrReading)
    let verticalInset = max((referenceHeight - contentHeight) / 2, 1)
    let horizontalInset = max((fontSize * 0.45).rounded(.up), 4)
    let attachment = NSTextAttachment()
    let badgeCell = RoundedBadgeTextAttachmentCell(
        label: label,
        attributes: attrReading,
        backgroundColor: .clear,
        padding: .init(
            top: verticalInset,
            left: horizontalInset,
            bottom: verticalInset,
            right: horizontalInset
        )
    )
    attachment.attachmentCell = badgeCell
    return .init(attachment: attachment)
}
```

讀音 disambiguation 行是**獨立於** peripherals 的視覺元素，**不**應追加到 `attributedDescriptionBottomPanes` 中——否則在 horizontal 單行模式下它會錯誤地出現在右側而非底部。`attributedDescriptionBottomPanes` 保持原有邏輯不變。

**G. 修改 `updateMetrics()`：新增 disambiguation 行佈局計算**

在 `updateMetrics()` 末尾、`metrics = .init(...)` 之前，插入以下邏輯：

```swift
// Phase 61: 計算讀音 disambiguation 行的位置（永遠在視窗最底部，獨立於 peripherals）
let strReadingDisambiguation = attributedDescriptionReadingDisambiguation
var rectReadingDisambiguation: CGRect = .zero
if !strReadingDisambiguation.string.isEmpty {
    var dimReading = strReadingDisambiguation.getBoundingDimension(forceFallback: true)
    dimReading.width = ceil(dimReading.width)
    dimReading.height = ceil(dimReading.height)
    let readingOrigin: CGPoint
    if finalContainerOrientation == .horizontal {
        // Horizontal 單行模式：peripherals 在右側，disambiguation 行置於底部
        // 需確保在 peripherals 和候選字行二者的下方
        let bottomOfCandidates = originDelta + totalAccuSize.height
        let bottomOfPeripherals = rectPeripherals.maxY
        let contentBottom = max(bottomOfCandidates, bottomOfPeripherals)
        readingOrigin = .init(x: originDelta, y: contentBottom + padding)
    } else {
        // Vertical / Expanded 模式：peripherals 已在下方，disambiguation 置於其下方
        readingOrigin = .init(x: originDelta, y: rectPeripherals.maxY + padding)
    }
    rectReadingDisambiguation = .init(origin: readingOrigin, size: dimReading)
    // 更新 totalAccuSize 以容納新行
    totalAccuSize.width = max(totalAccuSize.width, dimReading.width + originDelta)
    totalAccuSize.height = readingOrigin.y + dimReading.height - originDelta
}
```

#### 3.4 `CtlCandidateTDK4AppKit.swift`（兩個倉庫）

**A. 修改 `updateDisplay()`：新增 disambiguation 判斷邏輯**

在現有的 reverse lookup / tooltip 計算之後、寫入 `Self.thePool` 之前，插入：

```swift
// Phase 61: 計算讀音 disambiguation 顯示
Self.thePool.readingDisambiguationResult = nil
if let currentCandidate = Self.thePool.currentCandidate {
    let displayedText = currentCandidate.displayedText
    let currentKeyArray = currentCandidate.keyArray
    let currentSegLength = currentCandidate.segLength
    // 檢查是否存在「同字不同音」或「同字詞不同幅節長度」的其他候選字
    let needsDisambiguation = Self.thePool.candidateDataAll.contains(where: { otherCell in
        guard otherCell !== currentCandidate else { return false }
        guard otherCell.displayedText == displayedText else { return false }
        // 讀音不同 或 幅節長度不同
        return otherCell.keyArray != currentKeyArray || otherCell.segLength != currentSegLength
    })
    if needsDisambiguation {
        // 構建讀音顯示字串：以 "-" 連接各讀音 cell；"_" 開頭者顯示為 "??"
        let readingString = currentKeyArray.map { key in
            key.hasPrefix("_") ? "??" : key
        }.joined(separator: "-")
        Self.thePool.readingDisambiguationResult = readingString
    }
}
```

> **說明**：disambiguation 判斷邏輯放在 `CtlCandidateTDK4AppKit.updateDisplay()` 而非 `CandidatePool4AppKit` 內部，是因為該邏輯需要遍歷 `candidateDataAll` 並理解 `displayedText` / `keyArray` / `segLength` 間的語意關係。這屬於 controller 層的「何時顯示何種資訊」決策，而非 pool 層的純資料管理。

### 四、Disambiguation 觸發條件細則

| 場景 | 候選池內其他 cell 條件 | 應顯示的讀音 |
|------|----------------------|-------------|
| 同字不同調（同字不同音） | `displayedText` 相同，`keyArray` 不同（segLength 可能相同） | 當前 cell 的 keyArray，`-` 連接 |
| 字詞相同、幅節長度不同 | `displayedText` 相同，`segLength` 不同（keyArray 必然不同） | 同上 |
| 唯一候選字詞（無歧義） | 不存在其他同 `displayedText` 的 cell | **不顯示該行** |
| 讀音 cell 以 `_` 開頭 | — | 該 cell 顯示為 `??` |

### 五、影響範圍與注意事項

1. **`attributedDescriptionBottomPanes` 不受影響**：disambiguation 行是獨立的視覺元素，有自己的 `UIMetrics.readingDisambiguation` 欄位，不干擾 peripherals 的佈局與顯示。
2. **`updateMetrics()` 受影響**：需新增 disambiguation 行的尺寸與位置計算。horizontal 模式下的邏輯需特別處理——取 `max(bottomOfCandidates, bottomOfPeripherals)` 作為 disambiguation 行的 y 起點。
3. **`UIMetrics` 結構受影響**：新增 `readingDisambiguation: CGRect` 欄位，且 `allZeroed` 靜態屬性需同步更新。
4. **`draw(_:)` 受影響**：在 peripherals 繪製完成後，額外繪製 disambiguation 行。
5. **`attributedDescriptionHorizontal` / `attributedDescriptionVertical` 無需修改**：它們僅供 SwiftUI preview / 除錯用途，不影響實際 AppKit 繪製路徑。
6. **Preview 測試資料**（macOS 專屬 `_TDK4AppKit_Previews.swift`）：測試 pool 的建構若未使用 `construct()` 的 `CandidateInState` 路徑，則需手動為測試 cell 指定 `keyArray` 以便驗證 disambiguation 顯示效果。
7. **L10n 考量**：本 Phase 無新增使用者可見的靜態文字（`??` 為符號佔位，無需翻譯）。

### 六、佈局示意（校正後）

**Horizontal 單行模式**（disambiguation 永遠在底部）：

**Vertical / Expanded 模式**（disambiguation 在 peripherals 下方）：

### 七、測試驗證點

- [ ] 當所有候選字的 `displayedText` 各異時，不出現 disambiguation 行。
- [ ] 當「和」同時以 `["ㄏㄜˊ"]` 和 `["ㄏㄜ˙"]` 出現在候選池時，高亮任一「和」都會在視窗底部顯示對應讀音。
- [ ] 當「🐝」同時以 `["ㄇㄧˋ", "ㄈㄥ"]`（幅節長度 2）和 `["ㄈㄥ"]`（幅節長度 1）出現時，高亮不同 candidate 會顯示不同長度的讀音串。
- [ ] 當讀音 cell 以 `_` 開頭時，該 cell 顯示為 `??`。
- [ ] **Horizontal 單行模式**：disambiguation 行在底部，而非右側（peripherals 仍保持在右側）。
- [ ] **Vertical 模式**：disambiguation 行在 peripherals 下方。
- [ ] **Expanded 多行模式**：disambiguation 行在 peripherals 下方。
- [ ] 文字透明度為 peripheral 文字顏色的 80%（`withAlphaComponent(0.8)`）。


## Phase 61 實作結果（2026-05-08, Deepseek-v4-Pro）

### 最終規則（雙規則）

| Rule | 條件 | 效果 |
|------|------|------|
| **1** | 同 `displayedText` 的 cell 群組內，`keyArray` 不全等 | 該組所有 cell 顯示 disambiguation |
| **2** | `segLength == 1` 且池內所有單字 cell 的 `keyArray` 種類數 > 1 | **全部** `segLength == 1` 的 cell 顯示 disambiguation |

- `_` 開頭的 reading cell → 顯示為 `??`
- 各 reading cell 以 `-` 連接
- 讀音行樣式繼承 reverse lookup，文字透明度 80%
- 讀音行永遠置於視窗底部

### 核心邏輯（`CandidatePool4AppKit.updateReadingDisambiguation()`）

```swift
func updateReadingDisambiguation() {
    readingDisambiguationResult = nil
    guard let currentCandidate = currentCandidate else { return }
    let currentKeyArray = currentCandidate.keyArray
    guard !currentKeyArray.contains(where: { $0.hasPrefix("_") }) else { return }
    let displayedText = currentCandidate.displayedText
    let currentSegLength = currentCandidate.segLength

    // Rule 1: 同字但 readingArray 不全等 → 顯示
    let sameTextCells = candidateDataAll.filter { $0.displayedText == displayedText }
    let rule1 = !sameTextCells.allSatisfy { $0.keyArray == currentKeyArray }

    // Rule 2: segLength == 1 且池內單字 reading 種類 > 1 → 全部單字顯示
    var rule2 = false
    if currentSegLength == 1 {
      let singleSegCells = candidateDataAll.filter { $0.segLength == 1 }
      let distinctReadings = Set(singleSegCells.map { $0.keyArray })
      rule2 = distinctReadings.count > 1
    }

    if rule1 || rule2 {
      let readingString = currentKeyArray.map { $0.hasPrefix("_") ? "??" : $0 }
        .joined(separator: "-")
      readingDisambiguationResult = readingString
    }
}
```

## 手術範圍

本次手術同時牽涉到 `vChewing-macOS`、`vChewing-OSX-Legacy` 兩個倉庫。
（`vChewing-LibVanguard` 的 SharedCore 已為純跨平台協定，無 NSAttributedString 等 Darwin 依賴，本次無需更動。）

## 問題現狀

IMEState 目前的實作恐怕過於複雜化了、被 associatedType 綁死了。
理想狀態是保持 IMEState 的絕對跨平台可用性：任何位於 MainAssembly 的 Darwin-Specific Implementation 都應該弄成一個 **即用即拋** 的 struct wrapper（類似 Typewriter (Swift Package) 內部的各種各樣的 `???Typewriter` Structs。這個 struct wrapper 可以命名為 `IMEStateParsed4Darwin`。須知：Session 層面保存的仍是 IMEState。

> **設計方針**：InputHandler protocol 與 SessionCoreProtocol 應直接使用 concrete `IMEState` struct（而非透過 `associatedtype State: IMEStateProtocol` 間接約束）。`associatedtype` 關係僅應存在於 InputHandler ↔ InputSession 之間。`IMEStateProtocol` 不應出現在任何 `associatedtype` 宣告中。

## Phase 62 手術方案（實作規格）

### 一、現有架構分析

#### 1.1 現有 associatedtype 鏈（Phase 62 將予拆除）

目前 `SessionCoreProtocol` 與 `InputHandlerCoreProtocol` 各自宣告了 `associatedtype State: IMEStateProtocol`，並以 `where` 子句彼此約束：

```
SessionCoreProtocol
  ├── associatedtype State: IMEStateProtocol            ← 🚫 拆除
  ├── associatedtype Handler: InputHandlerCoreProtocol  ← ✅ 保留
  │     where Handler.State == Handler.Session.State, Handler.State == State  ← 🚫 拆除 State 相關部份
  └── var state: State { get set }                     ← 改為 var state: IMEState

InputHandlerCoreProtocol
  ├── associatedtype State: IMEStateProtocol            ← 🚫 拆除
  └── associatedtype Session: SessionCoreProtocol & CtlCandidateDelegate  ← ✅ 保留
        where Session.State == Session.Handler.State, Session.State == State  ← 🚫 拆除 State 相關部份
```

問題：這條 associatedtype 鏈迫使所有操作 `state` 的泛型程式碼都必須攜帶 `where State: IMEStateProtocol` 約束，且 `IMEStateProtocol` 的協定擴展與 `IMEStateData` 的資料擴展混雜了大量 Darwin-only 程式碼（`NSAttributedString`、`AppKit`、`ChineseConverter`、`LMMgr` 等），導致這些擴展無法跨平台複用。

#### 1.2 目標：拆掉 State associatedtype，改用 concrete IMEState

```
SessionCoreProtocol
  ├── associatedtype Handler: InputHandlerCoreProtocol  ← ✅ 保留
  │     where Handler.Session == Self                   ← ✅ Handler↔Session 唯一關聯
  └── var state: IMEState { get set }                   ← concrete type

InputHandlerCoreProtocol
  ├── (無 State associatedtype)                          ← 🚫 已拆除
  └── associatedtype Session: SessionCoreProtocol        ← ✅ 保留
        where Session.Handler == Self                    ← ✅ Handler↔Session 唯一關聯
```

為此，concrete `IMEState` struct 必須從 `MainAssembly4Darwin`（macOS）/ `vChewing/Modules/`（Legacy）**上移至 `Shared/Protocols/`**，因為 `SessionCoreProtocol` 定義於 `Shared`，而 `Shared` 不能反向依賴 `MainAssembly4Darwin`。

> **`IMEState` 本身不含 Darwin 依賴**：它僅有 `type: StateType` + `data: IMEStateData` + `node: CandidateNode` 三個 stored properties，全都是純 Swift 型別。Darwin 汙染全在其協定擴展與 `IMEStateData` 擴展中，這些將抽離至 `IMEStateParsed4Darwin`。

#### 1.3 Darwin 汙染點盤點

以下列出目前 macOS/Legacy 倉庫中所有 Darwin 相關的 IMEState 程式碼。這些將全數抽離至 `IMEStateParsed4Darwin`：

> **與 Typewriter Pattern 的對照**：`IMEStateParsed4Darwin` 之於 `IMEState` 就如同 `BPMFFullMatchTypewriter<Handler>` 之於 `InputHandler`。Typewriter 是 handler 的「即用即拋」包裝——你建立一個 Typewriter 實例、呼叫 `handle()`、然後丟棄。同樣地，`IMEStateParsed4Darwin` 是 `IMEState` 值的「即用即拋」包裝——你在需要 Darwin-specific 屬性時建立它、讀取屬性、然後丟棄。Session 層面永久儲存的仍是純 `IMEState`（concrete struct）。

#### 3.1 Step 1: 將 concrete `IMEState` struct 上移至 `Shared/Protocols/IMEStateProtocolAndData.swift`

concrete `IMEState` struct 目前位於 `MainAssembly4Darwin/IMEState.swift`。由於 `Shared` 中的 `SessionCoreProtocol` 將直接引用 concrete `IMEState`（而非 `associatedtype State`），必須將 struct 定義搬至 `Shared`。

**搬遷內容**（從 `IMEState.swift` → `IMEStateProtocolAndData.swift`）：

```swift
// MARK: - IMEState

/// 用以呈現輸入法控制器（SessionCtl）的各種狀態。
///
/// … (doc comments unchanged) …
public struct IMEState: IMEStateProtocol {
  // MARK: Lifecycle

  public init(
    _ data: IMEStateData = IMEStateData(),
    type: StateType = .ofEmpty
  ) {
    self.data = data
    self.type = type
  }

  /// 泛用初期化函式。
  public init(
    _ data: IMEStateData,
    type: StateType = .ofSymbolTable,
    node: CandidateNode
  ) {
    self.data = data
    self.type = type
    self.node = node
    self.data.candidates = node.members.map { ([""], $0.name) }
    if node.members.isEmpty {
      var newDisplayTextSegments = [node.name]
      // hardenVerticalPunctuationsIfNeeded 已移至 IMEStateParsed4Darwin
      // factory method 的 call site 會在構造 state 之後手動呼叫
      self.data.displayTextSegments = newDisplayTextSegments
      self.data.cursor = self.data.displayTextSegments.first?.count ?? node.name.count
      self.data.marker = self.data.cursor
    } else {
      self.data.displayTextSegments.removeAll()
      self.data.cursor = 0
      self.data.marker = 0
    }
  }

  // MARK: Public

  public var type: StateType = .ofEmpty
  public var data: IMEStateData = .init()
  public var node: CandidateNode = .init(name: "")
}
```

> **注意**：`init(displayTextSegments:cursor:)` 中的 `Self.hardenVerticalPunctuationsIfNeeded` 呼叫**暫且移除**。Factory method 的 call site 將在構造 state 之後手動呼叫 `IMEStateParsed4Darwin.hardenVerticalPunctuationsIfNeeded`。

#### 3.2 Step 2: 修改 `Shared/Protocols/SessionCoreProtocol.swift` — 拆除 State associatedtype

##### 3.2.1 macOS/Legacy 版 (`Shared`)

##### 3.2.2 修改 `InputHandlerCoreProtocol`

```swift
// 舊：
```

### 核心邏輯

```swift
// MARK: - 型別定義
@usableFromInline typealias KVHashMap = [String: [Bool: Set<Character>]]

// MARK: - 建構區分表
static func generateDistinctionHashTables(using dataMap: DataMap) -> KVHashMap {
    var resultMap = KVHashMap()
    dataMap.forEach { key, valueMap in
        guard let simpChars = valueMap["S"],
              let tradChars = valueMap["T"],
              simpChars != tradChars
        else { return }
        let simpSet = Set(simpChars)
        let tradSet = Set(tradChars)
        let simpOnly = simpSet.subtracting(tradSet)
        let tradOnly = tradSet.subtracting(simpSet)
        guard !simpOnly.isEmpty || !tradOnly.isEmpty else { return }
        resultMap[key] = [true: simpOnly, false: tradOnly]
    }
    return resultMap
}

// MARK: - 查詢
func isExclusive(isCHS: Bool, reading: String, target: Character?) -> Bool? {
    guard let target else { return nil }
    return distinctionTables[reading]?[isCHS]?.contains(target)
}
```

### 一、需求精確化

#### 1.1 過濾對象的邊界條件

| 條件 | 判斷 | 理由 |
|------|------|------|
| 多字詞（`segLength > 1`）中的字 | **永不過濾** | 「所有能打出的詞所用到的字都是必要的字」 |
| LMPlainBopomofo（EtenDOS）收錄的字 | **永不過濾** | 「所有受 EtenDOS 倚天中文系統支持輸入的漢字也都是必要的漢字」 |
| CNS11643 全字庫收錄的字 | **永不過濾** | 「CNS 語料也不是過濾對象」 |
| 原廠 CHS/CHT 表格中的單字（`segLength == 1`） | **唯一過濾候選** | 「只是原廠 CHS / CHT 表格當中的 single-segmentLength 漢字對象才可能是過濾對象」 |

#### 1.2 過濾行為定義

- **繁體模式（CHT）**：隱藏「簡體特有字」（S-only），即該漢字僅存在於簡體中文體系、不存在於繁體中文體系。
- **簡體模式（CHS）**：隱藏「繁體特有字」（T-only），即該漢字僅存在於繁體中文體系、不存在於簡體中文體系。

#### 1.3 過濾語義：完全移除（remove），非降權（demote）

**與 CNS 過濾（Phase 22）的關鍵差異**：

| | CNS 過濾 | Phase 64 逆輸入模式過濾 |
|------|---------|------------------------|
| 動機 | 該字不在 CNS 標準內，但可能仍是合法漢字 | 使用者**主動排斥**另一體系的漢字 |
| 單字處理 | demote to -9.5（仍可選取） | **完全移除**（不出現在候選窗） |
| 多字詞處理 | 完全移除 | 永不過濾 |

CNS 過濾採用 demote 是因為：不在 CNS 中的字仍可能是使用者需要的罕用字。但 Phase 64 的使用者群體是**刻意不想看到**另一體系的字——降權不足夠，必須直接消失。

### 三、方案比較

#### 方案 A：基於 Phase 63 `isExclusive` 的輕量過濾（推薦首步）

**運作原理**：
1. 在 `LMInstantiator.unigramsFor()` 中，對 `keyArray.count == 1` 的 `factoryCoreUnigramsResult` 逐筆檢查。
2. 呼叫 `lmPlainBopomofo.isExclusive(isCHS:reading:target:)` 判斷該單字是否為 contradictory mode 的特有字。
3. 若是，**完全移除**（`removeAll`）。
4. 若 `isExclusive` 返回 `nil`（讀音不在 EtenDOS 中），**保留該字**（不做過濾）。

**優點**：
- 實作極簡：~15 行新程式碼，插入點在 CNS 過濾區塊之後。
- 查詢效能優異：`isExclusive` 是兩層 Dictionary lookup（O(1)），無 Trie 查詢。
- 不需要任何新資料結構或跨模組 API。
- 不需要變更 TrieKit / VanguardLexicon。

**缺點**：
- 涵蓋範圍受限：僅對 EtenDOS 收錄的 ~277 個讀音有效。超出 EtenDOS 範圍的單字（如化學元素新簡化字 `𬬭`、`𫓧` 等）不會被過濾。
- 這是 Phase 63 初始設計者已指出的已知限制。

**效能影響**：
- 載入期：零額外成本（`distinctionTables` 已在 Phase 63 建構完畢）。
- 查詢期：每筆單字 Unigram 增加 1~2 次 Dictionary lookup + 1 次 Set.contains（合計 < 50ns）。
- 記憶體：零額外成本。
- 注意：移除操作本身是 `removeAll(where:)` 的 O(n) 遍歷，但 n 為單字 Unigram 數量（通常 < 50），成本可忽略。

**結論**：適合作為第一階段實作，涵蓋 >90% 的常用簡繁互異場景。

#### 方案 B：基於 TextMap Type B 行的完整區分表

**運作原理**：
1. 在 `LMInstantiator.connectFactoryDictionary()` 完成後，遍歷 TextMapTrie 的所有單字讀音。
2. 對每個讀音，分別收集所有 `.chs`（typeID=5）和 `.cht`（typeID=6）的單字條目。
3. 計算 CHS-only = CHS 字集 − CHT 字集、CHT-only = CHT 字集 − CHS 字集。
4. 建立 `[String: [Bool: Set<Character>]]`（與 Phase 63 同結構）。
5. 查詢時直接查表（無需再交叉比對 EtenDOS）。

**優點**：
- 涵蓋範圍完整：覆蓋原廠辭典中的所有單字（不僅限 EtenDOS）。
- 查詢效能與方案 A 相同（O(1) Dictionary lookup）。

**缺點**：

| 問題 | 說明 |
|------|------|
| **TextMapTrie 無公開 Key 列舉 API** | `keyEntries` 是 `internal`，外部無法遍歷所有讀音。需要修改 TrieKit 新增公開 API（影響 LibVanguard + macOS + Legacy 三個倉庫）。 |
| **Type B 行按機率分組、非按字義配對** | 同讀音下的 CHS 和 CHT 條目可能分散在多行（不同機率）。需要聚合所有行才能得到完整字集。這意味著必須 parse 每個讀音的**所有** entries，而非僅讀取個別行。 |
| **載入期成本** | 全量 parse 所有單字讀音的 entries（預估數百個讀音 × 數十個 entries）。雖然總量不大，但 TextMapTrie 的 lazy parse 設計被繞過。 |
| **跨倉庫漣漪效應** | 需要變更 TrieKit（LibVanguard）、LMInstantiator（macOS + Legacy）、可能需要變更 VanguardTrieProtocol。 |
| **TextMap 非 TYPING 模式的相容性** | 若未來 factory trie 改用非 TYPING 格式（如 `TRIE_TEXTMAP`），Type B 行不存在，完整區分表無法建構。 |

**效能影響**：
- 載入期：需遍歷所有單字讀音並 parse entries。預估 200~500 個讀音，每個 parse 5~50 行，總計約 1,000~25,000 行 parse。以每行 ~1μs 計，約 1~25ms。可接受但非零。
- 查詢期：與方案 A 相同。
- 記憶體：額外 ~100KB（Dictionary + Set\<Character\>）。

**結論**：涵蓋範圍完整，但實作複雜度高、跨模組漣漪效應大。建議作為方案 A 的後續升級路徑，待方案 A 落地後視使用者回饋決定是否推進。

#### 方案 C：VanguardLexicon 建置期預編譯靜態區分表

**運作原理**：
1. 在 VanguardLexicon 的 `VanguardTextMapDataBuilder` 建置過程中，於 Trie 寫入完畢後、TextMap 序列化前，計算單字層級的 CHS-only / CHT-only 字集。
2. 將結果序列化為 compact sidecar 檔案（如 `FactoryDistinctionTables.json`），與 `.txtMap` 一同發布。
3. 執行期由 `LMInstantiator` 載入該 sidecar，直接使用。

**優點**：
- 涵蓋範圍完整（同方案 B）。
- 執行期零建構成本，查詢 O(1)。
- 不需要變更 TrieKit 公開 API。
- 建置期可做更複雜的區分邏輯（如合併 CNS、EtenDOS 資料做交叉驗證）。

**缺點**：
- 需要變更 VanguardLexicon 建置管線。
- sidecar 檔案需要與 `.txtMap` 版本同步（若使用者手動更換辭典但未更新 sidecar，可能出現不一致）。
- 增加發布檔案數量。

**效能影響**：
- 載入期：sidecar JSON 解析（預估 < 5ms）。
- 查詢期：與方案 A 相同。
- 記憶體：約 100KB。

**結論**：若方案 A 的涵蓋範圍不足、且方案 B 的動態建構成本過高，這是最終的穩健方案。

### 四、推薦方案：方案 A（首步） + 方案 C 演進路徑

#### 4.1 決策理由

1. **方案 A 涵蓋 >90% 的實際使用場景**：常用簡繁互異字（如 `罢`/`罷`、`个`/`個`、`么`/`麼`）皆在 EtenDOS 收錄範圍內。
2. **零基礎設施變更**：完全在 `LMInstantiator.unigramsFor()` 內部實作，不觸碰 TrieKit、VanguardTrieProtocol、VanguardLexicon。
3. **效能影響可忽略**：Dictionary + Set lookup，每次查詢 < 50ns。
4. **降級路徑清晰**：若未來涵蓋範圍不足，可平滑升級至方案 B 或 C，且方案 A 的程式碼結構（插入點、Config 開關）可完全複用。

#### 4.2 過濾邏輯詳解

以 CHT 模式（`isCHS = false`）為例：

```
輸入：factoryCoreUnigramsResult（已通過 entryType=.cht 篩選，即 factory CHT 單字結果）
條件：config.filterFactoryKanjisOfNonCurrentInputMode == true
      && keyArray.count == 1（僅處理單字）

對每個 unigram：
  let char = unigram.current.first
  // 詢問 EtenDOS：該字是否為「簡體特有」？
  let result = lmPlainBopomofo.isExclusive(isCHS: true, reading: keyChain, target: char)

  if result == true {
    // EtenDOS 認定該字為簡體特有 → 在繁體模式下移除
    return nil  // ← removeAll 會過濾掉 nil
  }
  // result == false（共用字）或 nil（不在 EtenDOS 中） → 保留
  return unigram
```

CHS 模式（`isCHS = true`）則相反：詢問 `isExclusive(isCHS: false, ...)`（即「是否為繁體特有」），若為 `true` 則移除。

**關鍵語義**：`isExclusive` 的第一參數 `isCHS` 指定「查詢哪個體系的特有字」。過濾 contradictory mode 時，傳入 `!isCHS`（即當前模式的相反模式）。

> 與先前一樣：施術之後請恪守 `DevPlans` 目錄下的 KnowledgeMemo 裡面的 Response Pattern 完成 markdown 文件的記錄工作。

## 手術範圍

本次手術同時牽涉到 `vChewing-macOS`、`vChewing-OSX-Legacy` 兩個倉庫。

## 要解決的問題

Phase 59 在 `SessionCoreProtocol` 中新增了 `func commit(text: String, clearDisplayBeforeCommit: Bool, bypassAsync: Bool)` 協定需求，並在 `applyAutoSplitCandidate` 中改以 `bypassAsync: true` 同步遞交 ASCII prefix text。然而此改動在 **Ghostty 實機測試**中引發了嚴重 regression：**所有鍵入 Tekkon composer 的字元之 raw char 全部洩漏給客體應用**（即每個按鍵的原始字元都被意外 commit）。

## 根因分析

1. **最初錯誤假設（Phase 65 第一版）**：認為是 `clearInlineDisplay()` 在 `overflowText` 為空時未被呼叫、導致 inline display 為空而洩漏事件。在 `setInlineDisplayWithCursor()` 中加入 placeholder 防禦後，**問題依舊存在**。

2. **真正根因**：在 `SessionCoreProtocol` 中新增第二個 `commit` 協定需求（3-parameter overload）導致 **Swift runtime protocol witness table dispatch 異常**。當 protocol 同時擁有 `commit(text:clearDisplayBeforeCommit:)` 與 `commit(text:clearDisplayBeforeCommit:bypassAsync:)` 兩個 requirements 時，透過泛型 `Session` associated type 呼叫 `commit` 的路徑（如 `switchState` 內的 `.ofInputting` / `.ofCommitting` 分支、及所有 Typewriter 層的 `commit` 呼叫）出現 dispatch 不穩定，導致 compositor 狀態無法正確過渡、event 攔截失效。

3. **Ghostty 的特殊性**：Ghostty 作為 GPU-accelerated 終端模擬器，其 IMK 處理管線較長、對 marked text 的存在與否更敏感。其他應用可能因 IMK 實作差異而未被觸發。

## 解決方法

**三管齊下**：

### A. 回滾 Phase 59 的 `commit(bypassAsync:)` 協定變更

從 `SessionCoreProtocol` 移除 `func commit(text:clearDisplayBeforeCommit:bypassAsync:)` 協定需求與其預設 extension 實作。從 `InputSession_HandleStates.swift`（兩倉庫）移除對應的 concrete implementation。

### B. 恢復 `applyAutoSplitCandidate` 至 Phase 59 前行為

移除獨立的 `session.commit(text: prefixText, clearDisplayBeforeCommit: false, bypassAsync: true)` 呼叫，改回將 `prefixText + overflowText` 合併至 `.ofInputting` 狀態的 `textToCommit` 中一併遞交：

```swift
var inputting = handler.generateStateOfInputting()
inputting.textToCommit = prefixText + overflowText
session.switchState(inputting)
```

### C. 保留 Phase 65 placeholder 防禦（安全網）

## 要解決的問題

Izanami tests（`test_IH400_MixedAlnumKanjiInputTest_Izanami`）在 macOS 上的執行時間為 Linux 的 3.4×、Windows 的 4.8×。GitHub Actions CI 數據（tag 4.4.4）：
- macOS: 119s
- Linux: 35s
- Windows: 25s

Instruments Time Profiler 追蹤確認瓶頸在 Apple Swift runtime 的 `CFString` 橋接：
`__CFStringCreateImmutableFunnel3`、`CFEqual`、`CFStringFindWithOptionsAndLocale`、`_platform_memmove` 等函式佔據 CPU 時間首位。macOS 上 Swift `String` 的幾乎所有操作（`==`、`hash`、`hasPrefix`、`lowercased()`、`range(of:regex:)`）都透過 `#if _runtime(_ObjC)` 區塊橋接到 `NSString` / `CFString`，而 Linux/Windows 走 pure Swift 實作。

## 根因分析

### 初步假說（均被實測否定）

1. **鎖開銷**（`NSLock` → `pthread_mutex_t` / `DispatchQueue.sync`）：~10% 改善，不足。
2. **`Data` / `NSData` 橋接**（`TextMapTrie.rawData` 改為 `[UInt8]`）：無改善。`TextMapTrie` 內部大量 byte-level 操作本身已繞過 `Data` subscript 橋接。
3. **`String.range(of:regex:)` → UTF-8 byte helpers**：僅 ~4% 改善。

### 真正有效的優化

| 步驟 | 措施 | Intel MBP 實測 | 機制 |
|------|------|---------------|------|
| 1 | `[Character]` slicing → `String.prefix(n)` / `String.suffix(n)` | 99s → 80s (**19%**) | `Array(fullInput)` + `String(Character slice)` 改為 Substring 共享底層儲存，無字元拷貝 |
| 2 | `for char in` + `char.description` → `for scalar in text.unicodeScalars` + `receiveKey(fromScalar:)` | 80s → 77s (**4%**) | 繞過 `Character` grapheme cluster 分析與 `char.description` 的 String 分配，直接走 `UnicodeScalar` |

**三平台最終對比（GitHub CI，相同程式碼）：**

| Prefix | macOS | Linux | WinNT | macOS/Linux |
|--------|-------|-------|-------|-------------|
| NoPF | 10.6s | 3.4s | 1.5s | 3.1× |
| ai | 18.5s | 6.1s | 3.3s | 3.0× |
| hello | 28.7s | 10.1s | 5.8s | 2.8× |
| **總計** | **104.5s** | **36.2s** | **20.4s** | **2.89×** |

## 解決方法

### A. String 切片：`[Character]` → `String.prefix`/`suffix`

`bestAutoSplitCandidate(fullInputChars: [Character])` 改為 `bestAutoSplitCandidate(fullInput: String)`。呼叫方不再建立 `Array(fullInput)`。loop 內：
- `String(fullInputChars.prefix(n))` → `String(fullInput.prefix(n))`（Substring 共享儲存）
- `String(fullInputChars.suffix(n))` → `String(fullInput.suffix(n))`（同上）

### B. Character iteration → UnicodeScalar iteration

`buildAutoSplitCandidate` 內部：
- `effectiveSuffixText`（`String.dropLast()`）移除，不再分配新 String
- `for char in effectiveSuffixText` + `char.description` → `for scalar in suffixText.unicodeScalars` + `receiveKey(fromScalar:)`
- 首字元檢查：`effectiveSuffixText.first?.description` → `suffixText.unicodeScalars.first` + `receiveKey(fromScalar:)`

## 要解決的問題

InputSession 與 SessionCtl 目前使用 `NSMapTable<NSObject, InputSession>.weakToStrongObjects()` 來管理自身副本與 client() object 的映射關係。這個映射關係依賴於 client object 的記憶體位址。

然而，這招似乎對 Chrome 瀏覽器無效，導致 SessionCtl 對 Chrome 內部 client object memAddr 變更的情況無能為力、只能每次都新增 key 與 InputSession 副本。這個過程經常導致輸入法與整個桌面都卡死好幾秒。

P.S.: 需要評估是否將 key 換成 `client().uniqueClientIdentifierString`。

## Phase 67 手術方案（實作規格）

### 一、現有架構分析

#### 1.1 資料流

```
SessionCtl.init(server:delegate:client:)
  └─ callCoreAtLeastOnce(client:)
       ├─ client() as? NSObject → clientObj
       ├─ InputSession.cachedSession(for: memAddr)  ← [Int: InputSession] 查詢
       │    └─ hit: reassign() → 複用既有 InputSession ✅
       └─ miss: new InputSession(controller:clientProvider:)
            └─ init { construct(client:); registerInCache() }
```

- `SessionCtl` 是 IMKInputSessionController 的子類別，由 IMKServer 在每次輸入法被啟用時建立。
- `SessionCtl.core`（即 `InputSession`）透過靜態快取 `sessionsByClient` 嘗試複用，目的是緩解 CapsLock 中英切換場景下的 MainActor ARC 壓力（詳見 `DevLab/InputMethodKitPhuquingRetarded.txt`）。
- 快取鍵 = client NSObject 的記憶體位址，以 `Int(bitPattern:)` 表示。

#### 1.2 原快取機制與問題

```swift
// 舊版 (Phase 67 之前)
private static var sessionsByClient =
  NSMapTable<NSObject, InputSession>.weakToStrongObjects()
```

**雙重問題**：

1. **NSMapTable 自身內部設計導致 hang**：實測發現即使對於 native AppKit 應用（如 TextEdit），在以 CapsLock 高頻切換中英輸入法時也會 hang。根因推測為 NSMapTable 內部在頻繁 insert/remove 時的 synchronization 或 hash table rehash 開銷阻塞了 MainActor。

2. **`uniqueClientIdentifierString()` 完全不可信**：在 CotEditor 等應用中實測發現該值會持續 mutate，無法作為穩定的快取鍵。

#### 1.3 為何 `bundleIdentifier()` 不適合作為替代鍵

- 同一應用程式的多個視窗／分頁會共用相同的 Bundle ID（如 `"com.google.Chrome"`）。
- 若以此為鍵，多個文字輸入區會意外共用同一個 `InputSession`，造成狀態互相汙染。

### 二、方案：以 `Int`（memAddr）取代 NSMapTable 作為快取容器

#### 2.1 核心思路

不改變鍵的語義（仍以 client NSObject 的記憶體位址標識不同客體），但**將容器從 `NSMapTable` 換成普通的 `[Int: InputSession]` Swift Dictionary**，以繞過 NSMapTable 自身的內部設計缺陷。

- 鍵：`Int(bitPattern: Unmanaged.passUnretained(clientObj).toOpaque())` —— 記憶體位址的整數表示。
- 值：`InputSession` 強引用。
- 同步：`NSLock`。
- 清理：惰性孤本偵測 + LRU 淘汰（count > 5）。

#### 2.2 對 Chrome/Electron 的影響

Chrome/Electron 每次 CapsLock 切換可能回傳不同記憶體位址的 client proxy 物件 → 快取仍會 miss → 新建 `InputSession`。**但 NSMapTable 被替換為 plain Dictionary 後，新建流程不再觸發 NSMapTable 內部的 hang 邏輯，桌面不再凍結。**

### 三、具體變更規格

#### 3.1 `InputSession.swift`（兩個倉庫）

##### 3.1.1 變更 `sessionsByClient` 型別

```swift
// 變更前
private static var sessionsByClient =
  NSMapTable<NSObject, InputSession>.weakToStrongObjects()

// 變更後
private static var sessionsByClient = [Int: InputSession]()
private static let sessionsLock = NSLock()
private static var globalLRUTick: UInt = 0
```

##### 3.1.2 變更 `cachedSession(for:)`（含孤本偵測 + LRU tick）

```swift
// 變更前
static func cachedSession(for clientObj: NSObject) -> InputSession? {
  sessionsByClient.object(forKey: clientObj)
}

// 變更後
static func cachedSession(for memAddr: Int) -> InputSession? {
  sessionsLock.withLock {
    guard let session = sessionsByClient[memAddr] else { return nil }
    // 孤本清理
    if session.inputControllerAssigned == nil,
       Self.current?.id != session.id {
      sessionsByClient[memAddr] = nil
      return nil
    }
    // LRU tick
    session.lruTick = Self.globalLRUTick
    Self.globalLRUTick &+= 1
    return session
  }
}
```

##### 3.1.3 新增 `evictLRUIfNeeded()`

```swift
private static func evictLRUIfNeeded() {
  sessionsLock.withLock {
    guard sessionsByClient.count > 5 else { return }
    var oldestKey: Int?
    var oldestTick: UInt = .max
    for (key, session) in sessionsByClient {
      guard session.inputControllerAssigned == nil,
            Self.current?.id != session.id
      else { continue }
      if session.lruTick < oldestTick {
        oldestTick = session.lruTick
        oldestKey = key
      }
    }
    if let oldestKey { sessionsByClient[oldestKey] = nil }
  }
}
```

##### 3.1.4 變更 `registerInCache()`

```swift
func registerInCache() {
  guard let clientObj = theClient() else { return }
  let key = Int(bitPattern: Unmanaged.passUnretained(clientObj).toOpaque())
  Self.sessionsLock.withLock { Self.sessionsByClient[key] = self }
  Self.evictLRUIfNeeded()
}
```

##### 3.1.5 新增實例屬性

```swift
var lruTick: UInt = 0
```

##### 3.1.6 `deinit` — 無變更

`deinit` 內不做快取清理。條目由 `evictLRUIfNeeded()` 或 `cachedSession` 惰性偵測移除。若在 deinit 內嘗試取得 `sessionsLock` 會造成同執行緒 NSLock 重入死結。

#### 3.2 `SessionCtl.swift`（兩個倉庫）

```swift
// 變更前
if let clientObj, let cached = InputSession.cachedSession(for: clientObj) { ... }

// 變更後
if let clientObj {
  let key = Int(bitPattern: Unmanaged.passUnretained(clientObj).toOpaque())
  if let cached = InputSession.cachedSession(for: key) { ... }
}
```

### 四、孤本清理策略

| 層級 | 機制 | 觸發時機 |
|------|------|----------|
| 1 | `cachedSession` 惰性偵測 | 查詢命中時發現 `inputControllerAssigned == nil` 且非 `current` |
| 2 | LRU 淘汰 | `registerInCache` 登記後，僅在 `count > 5` 時淘汰最久未存取的孤本 |

### 五、影響範圍與注意事項

| 影響項目 | 說明 |

```
Type `macOS `，expect `macOS `, actual `ma<u>ㄏ</u>`。
```

## Phase 68 手術方案（實作規格）

### 一、現有架構分析

#### 1.1 資料流

MixedAlphanumericalTypewriter 的 `handle(_:)` 接收每個按鍵事件後：

1. 判斷按鍵是否為合法注音鍵（`inputValidityCheck`，大小寫不敏感）。
2. 若為大寫字母（Shift+letter），將 `isPhoneticKey` 設為 `false`（保留大寫語義），但字元仍以原始大小寫存入 `mixedAlphanumericalBuffer`。
3. 以 `fullInput = buffer + inputText` 進行三條路徑判斷：
   - **整段注音路徑**（`isFullyParserCovered`）：僅限無大寫字母、全部為合法注音鍵的輸入。
   - **auto-split 路徑**（`tryAutoSplitASCIIAndPhoneticSuffix`）：嘗試將 fullInput 切分為 ASCII 前綴 + 注音後綴。
   - **fallback**：清空 composer、buffer 保留 fullInput。

#### 1.2 auto-split 後綴候選建構流程（`buildAutoSplitCandidate`）

`buildAutoSplitCandidate` 負責對一個（前綴, 後綴）組合進行評估：

1. 將後綴（suffix）的 Unicode scalar 逐一餵入 `trialComposer.receiveKey(fromScalar:)`。
2. **Tekkon Composer 的 `receiveKey(fromScalar:)` 對大小寫不敏感**：大寫 'O' 會映射到與小寫 'o' 相同的注音符號。
3. 若 trialComposer 可發音（`isPronounceable`）且有語彙命中（`hasUnigramsForFast` 或 `unigramsFor`），則候選成立。

#### 1.3 問題機制：大寫字母後綴被誤判為注音

在 Dachen26 鍵盤佈局下，「macOS 」的按鍵序列：

| 按鍵 | Shift | 存入 buffer | isPhoneticKey |
|------|-------|-------------|---------------|
| `m`  | No    | `m`         | true          |
| `a`  | No    | `a`         | true          |
| `c`  | No    | `c`         | true          |
| `O`  | Yes   | `O`         | **false**     |
| `S`  | Yes   | `S`         | **false**     |
| `␣`  | No    | (Space handler) | —         |

Space 按下時，`buffer = "macOS"`，`fullInput = "macOS "`。auto-split 掃描：

- `suffixLength = 2`：`prefix = "ma"`，`suffix = "OS"`。
  - `prefix.count = 2`，含 ASCII letter → `requiredMinLength = 2` ✓
  - suffix `"OS"`（大寫）通過 `receiveKey(fromScalar:)` 的大小寫不敏感檢查
  - 若 `"os"` 在 Dachen26 下為合法且可發音的音節，且語彙庫有對應條目
  - → **候選被接受！** 前綴 `"ma"` 被提交為 ASCII、後綴 `"OS"` 被當作注音。

### 一、現有架構分析

#### 1.1 問題機制：buffer 被大寫字母「汙染」導致 auto-split 失效

以 `cOSq `（prefix `cOS` + reading `ㄆ` 的 Dachen26 鍵序 `q `）為例，完整鍵入序列：

| 步驟 | 按鍵 | buffer | composer | 說明 |
|------|------|--------|----------|------|
| 1 | `c` | `"c"` | `ㄘ` | 小寫注音鍵，正常吸收 |
| 2 | `O` (Shift) | `"cO"` | empty | 大寫→阻斷吸收，composer 清空 |
| 3 | `S` (Shift) | `"cOS"` | empty | 大寫→同上 |
| 4 | `q` | `"cOSq"` | empty | **關鍵點**：buffer 非空，`fullInput = "cOSq"` |
| 5 | ` ` (Space) | `"cOSq"` | empty | Space handler 嘗試處理 |

**步驟 4 的詳細機制（非 Space handler）：**

- `fullInput = "cOSq"`
- `bestAutoSplitCandidate` 嘗試拆分：
  - `suffixLength=1`：prefix=`"cOS"`, suffix=`"q"`。`buildAutoSplitCandidate`：
    - Phase 68 guard（`suffixHasUppercase`）：`"q"` 無大寫 ✓
    - `isPronounceable`：ㄆ（單聲母）→ `true`
    - `hasIntonation()`：無聲調 → `false`；`suffixEndsWithSpace`：`false`
    - → guard 失敗，return nil ❌
  - `suffixLength=2`：prefix=`"cO"`, suffix=`"Sq"`。Phase 68 guard → `"Sq"` 含大寫 → return nil ❌
  - `suffixLength=3`：prefix=`"c"`, suffix=`"OSq"`。Phase 68 guard → `"OSq"` 含大寫 → return nil ❌
- 所有候選失敗 → 進入 fallback：`buffer = "cOSq"`, composer 清空。
- `"q"` 本應啟動新的注音序列，卻被 append 到含大寫的 buffer 中，buffer 被「汙染」。

**步驟 5 的詳細機制（Space handler）：**

- `handler.mixedAlphanumericalBuffer = "cOSq"`
- `shouldPreferASCIIWordOnSpace = shouldPreferASCIIWordPath("cOSq", 1)` → `true`
  - `"cOSq"` 全為 ASCII 字母，destructive overwrite count = 2 ≥ 1
- `twoCharPrefixIsPhonetic`：buffer prefix `"cO"` → Dachen26 下為 `ㄘㄛ`
  - `occupiedSlots = 2` ✓
  - `hasUnigrams` for `"ㄘㄛ"`：`"ㄘㄛ"` 非標準華語音節（Pinyin 無 "co"），LM 中無對應 unigram → `false`
  - → `twoCharPrefixIsPhonetic = false`
- 進入 auto-split 的條件閘：
  - `!shouldPreferASCIIWordOnSpace || twoCharPrefixIsPhonetic || buffer.count >= 5`
  - = `false || false || false` → **auto-split 被跳過！**
- Composer 為空，最終 fallback：`"cOSq "` 直接作為 ASCII 提交。
- → Assembler 始終為空，reading 從未進入組字管線。

#### 1.2 根因總結

Phase 68 阻止了含大寫的 suffix 被誤判為注音後綴（正確）。但副作用是：當 prefix 本身含大寫字母（如 camelCase 縮寫 `cOS`），後續的小寫注音鍵 `q` 無法被 auto-split 單獨拆分（因缺乏聲調），而含大寫的較長 suffix 又被 Phase 68 阻斷。結果 `q` 被 append 到 buffer，buffer 被大寫字母「汙染」。到了 Space 階段，`shouldPreferASCIIWordPath` 因為 destructive overwrite 而返回 true，結合 `twoCharPrefixIsPhonetic = false`，auto-split 的條件閘完全關閉，最終整個 `"cOSq "` 被當作純 ASCII 提交。

**方案 A（採用）：Space handler 端強制 auto-split**
- 優點：僅在最終提交階段介入，不改變逐鍵行為，風險最低。

- 缺點：中間狀態（Space 之前）buffer 仍會被大寫「汙染」，但此為 cosmetic issue，不影響最終結果。

**方案 B（否決）：非 Space handler 端立即拆分**
- 在逐鍵處理階段，當 buffer 結尾為大寫且新鍵為小寫注音時立即拆分。

## 要解決的問題

使用者直接編輯使用者片語辭典目錄下的 `userdata-cht.txt`，以 **tab 字元** (`\t`) 作為欄位分隔符號，新增約九千多條片語記錄。重啟輸入法後，輸入法完全無法感知這些新增詞彙。將 delimiter 改回 ASCII space 後才恢復正常。

## 調查報告暨手術方案

### 調查過程

#### 1. LMCoreEX 的兩條載入路徑

##### 路徑 A：同步路徑 → `open()`

見 `lmCoreEX.swift`：

```swift
let consolidated = allowConsolidation  // lmUserPhrases 設為 true
do {
    let rawStrData: String = try LMAssembly.withFileHandleQueueSync {
        if consolidated {  // ✅ true → 進 consolidation
            LMConsolidator.fixEOF(path: path)
            LMConsolidator.consolidate(path: path, pragma: true)  // ❌ pragma 存在 → early return!
        }
        return try String(contentsOfFile: path, encoding: .utf8)
    }
    var processed = rawStrData
    if !consolidated {  // ❌ false → tab→space 替換被跳過！
        processed = processed.replacingOccurrences(of: "\t", with: " ")
        processed = processed.replacingOccurrences(of: "\r", with: "\n")
    }
    replaceData(textData: processed)  // ← 含 tab 的原味資料直接進入解析
}
```

##### 路徑 B：非同步路徑（生產環境預設）→ `readFileContentAsync` + `replaceData()`

見 `LMInstantiator.swift` (`loadUserPhrasesData`)：

```swift
LMAssembly.readFileContentAsync(
    path: path, shouldConsolidate: lmUserPhrases.allowConsolidation  // true
) { [weak self] content in
    self.lmUserPhrases.clear()
    self.lmUserPhrases.replaceData(textData: content)  // ← 直接進 replaceData，無任何 tab 處理！
}
```

見 `vChewingLMAssembly_Common.swift` (`readFileContentAsync`)：

```swift
if shouldConsolidate {  // true
    LMConsolidator.fixEOF(path: path)
    LMConsolidator.consolidate(path: path, pragma: true)  // ❌ pragma 存在 → early return!
}
let rawStrData = try String(contentsOfFile: path, encoding: .utf8)  // ← 讀回來的仍是 tab 分隔
```

#### 2. `LMConsolidator.consolidate(path:pragmma:)` 的 early return

```swift
let pragmaResult = checkPragma(path: path)
if shouldCheckPragma {
    if pragmaResult { return true }  // ← pragma 完好就跳過，不做任何格式整理！
}
```

**關鍵**：`lmUserPhrases` 初始化時 `consolidate: true`，表示期望走 LMConsolidator。但 LMConsolidator 在 pragma header 完好時會直接 early return，**完全不執行任何格式正規化**（包括 tab→space）。兩條路徑都因此無法處理 tab。

#### 3. 最終匯流點：`replaceData()` 完全不認 tab

```swift
// lmCoreEX.swift - replaceData()
strData.parse(splitee: "\n") { theRange in
    ...
    rawStrData.parseCells(in: theRange, splitee: " ") { ... }  // ← 只認 ASCII space！
    ...
    guard let firstCellRange, let secondCellRange else { return }  // ← tab 行只有一個 cell → 整行靜默丟棄！
}
```

`parseCells(in: splitee: " ")` 以 ASCII space 為分隔符。當資料用 tab 分隔時，整行被視為單一 cell，`secondCellRange` 永遠為 nil，`guard` 觸發 → **該行靜默丟棄，毫無錯誤訊息**。

#### 4. `lmUserPhrases` 初始化參數

```swift
// LMInstantiator.swift
var lmUserPhrases = LMCoreEX(
    reverse: true,
    consolidate: true,   // ← 設為 true
    defaultScore: { _ in 0 },
    forceDefaultScore: false
)
```

同樣受影響的還有 `lmFiltered`、`lmUserSymbols`（皆 `consolidate: true`）。

### 根因總結

```
使用者用 tab 編輯 userdata-cht.txt（pragma header 完好）
        │
        ▼
   open() 或 readFileContentAsync()
        │
        ▼
   LMConsolidator.consolidate(path: pragma: true)
        │  pragmaResult == true
        │  → early return，不做任何格式整理
        ▼
   replaceData(textData:)
        │  parseCells(in: splitee: " ")  ← 只認空格，不認 tab
        │  tab 分隔行 → 無 secondCell → guard 跳過
        ▼
   九千多條資料全部丟失，毫無錯誤訊息
```

### 手術方案

**修法**：在 `LMCoreEX.replaceData()` 內部最前端統一做 tab→空格正規化。這是所有載入路徑（`open()`、`readFileContentAsync`、`LMInstantiator.replaceData(textData:for:)`、`injectTestData`）最終都會經過的唯一匯流點。

只需在 `replaceData()` 開頭加一行：

```swift
mutating func replaceData(textData rawStrData: String) {
    let rawStrData = rawStrData.replacingOccurrences(of: "\t", with: " ")  // ← 新增
    if strData == rawStrData { return }
    ...
}
```

**理由**：
- `replaceData()` 是所有載入路徑的必經之路，在此處正規化可一勞永逸。
- 不依賴 `LMConsolidator` 的行為（其 pragma early-return 是設計意圖，不應改變）。
- `open()` 中原有的 `if !consolidated` 區塊內的 tab 替換邏輯可以保留作為冗餘防禦，但不再必要。
