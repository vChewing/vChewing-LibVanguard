**問題解決思路**：用一個 non-modal 的 NSOpenPanel 視窗向使用者請求對自身 bundle 所在目錄的存取權限。該視窗的觸發契機至少有「輸入法的 CLI 卸除命令」與「輸入法選單當中的卸除命令」。注意還需要額外實裝對於「NSOpenPanel 沒能成功獲取權限，或使用者取消了 NSOpenPanel」情形時的提示文字，且該提示需要誘導使用者前往 `https://github.com/vChewing/vChewing-macOS/wiki/HOW_TO_UNINSTALL` 。您或可設法設計更優的 UX 體驗。

### 技術決策

1. **`CandidatePairWeighted.weight` 只是語言模型 metadata，不是 explicit override score**：`overrideCandidate(_ candidate: CandidatePairWeighted, type: .withSpecified, ...)` 若直接重用 raw LM 權重，會把使用者主動選字退化成「再套一次語言模型排序」，於是出現節點找到卻沒有真正覆寫成功的情況。

### 真正元兇：`showContentForTab` 的 `docView.frame` 硬碼高度問題

`CtlSettingsCocoa.swift` 內的 `showContentForTab(_:)` 在 macOS 10.13+ 分支中，使用了如下邏輯：

```swift
let docView = FlippedClipContainerView()
let fittingSize = paneView.fittingSize        // ← (1) 在此時取值

docView.frame = NSRect(
    origin: .zero,
    size: CGSize(
        width: max(fittingSize.width, SettingsPanesCocoa.windowWidth),
        height: fittingSize.height             // ← (2) docView 高度被硬碼固定
    )
)

docView.addSubview(paneView)

paneView.translatesAutoresizingMaskIntoConstraints = false
docView.addConstraints([
    .top   pinned to docView.top,
    .leading pinned to docView.leading,
    // ← (3) 沒有 .trailing / .bottom / .width / .height 約束
])

scrollView.documentView = docView
```

### 完整的因果鏈

1. 第一次切換進某個設定頁時，`paneView.fittingSize` 以初始顯示的 Tab A 內容高度為準計算
2. `docView.frame.height` 被硬碼為這個值，且之後不再更新
3. `paneView` 對 `docView` 只有 `.top` 和 `.leading` 約束，**沒有四邊 pin**，所以 NSTabView 切換到 Tab B 後，`paneView` 本身確實會膨脹
4. 但 `docView.frame` 是 `NSScrollView` 的 `documentView`，決定了可滾動的高度範圍，此值不變
5. Tab B 的內容超出 `docView` 邊界，ScrollView 的可滾動範圍未更新 → 超出部分不可達 → 視覺上呈現「bottom border 切割」

### 為何版本行為有差異

- **macOS 10.9**：走 `#available(macOS 10.13, *)` 的 `else` 分支，不使用 ScrollView + docView 機制，直接 pin 到 `contentContainerView`，完全不存在這個問題。
- **macOS 26 Tahoe**：可能對 `documentView` frame 的裁切行為有不同實現（例如不再嚴格以 `frame` 為邊界），恰好不觸發截斷。
- **macOS 10.13–15.x**：嚴格依賴 `documentView.frame.height` 決定可滾動範圍，觸發截斷。

### 根本修復方向

問題的核心是：**不應靠孤兒狀態或初始狀態的 `fittingSize` 手動設定 `docView.frame`，應讓 AutoLayout 自行決定 `docView` 的大小。**

1. 移除 `docView.frame = NSRect(...)` 手動設定（或僅設定 width，不設定 height）
2. 設置 `docView.translatesAutoresizingMaskIntoConstraints = false`
3. 補齊 `paneView` 對 `docView` 的 `.trailing` 和 `.bottom` 約束（四邊 pin）
4. 讓 NSScrollView 透過 AutoLayout 自動得知 `documentView` 的實際高度

如此，NSTabView 切換內頁後的高度變化能自動傳播至 `docView`，ScrollView 的可滾動範圍隨之正確更新，截斷問題消除。

另需注意：每次 `paneView` 重新進入 hierarchy 前，應確保 `translatesAutoresizingMaskIntoConstraints` 明確設置為 `false`（避免殘留前次狀態）。

- `Packages/vChewing_MainAssembly4Darwin/Sources/MainAssembly4Darwin/Settings/SettingsCocoa/VwrSettingsPaneCocoaServices.swift`：同樣將 ServiceMgr table 的 `rowHeight` 改為依字型 metrics 計算，與 Clients pane 對齊。

### 技術決策

1. **根因不是 `NSTextFieldCell` 類型，而是固定 `rowHeight = 17` 與 15pt 窄體字型不相容**：這兩張 table 都使用 `NSFont.narrowedFont(size: 15)`，卻把 row height 硬寫成 17pt；對 condensed font 來說，descender 幾乎沒有任何下緣餘裕，因此會在某些 macOS 版本下被裁切。
2. **不必改成 `NSAttributedString` / view-based table**：現有 cell-based `NSTableView` 本身沒有證據顯示無法正確繪製 descender；真正出問題的是 row box 太矮。只要讓 `rowHeight` 跟著字型 metrics 走，既有資料來源與 cell 設定即可維持不變。
3. **row height 改採 font-metrics helper，而不是再硬編碼另一個常數**：這次以 `ascender - descender + leading + 4` 計算，並設下 20pt 下限，避免未來 condensed font 在不同 SDK / macOS 版本有細微 metrics 差異時又重演同類裁切問題。
