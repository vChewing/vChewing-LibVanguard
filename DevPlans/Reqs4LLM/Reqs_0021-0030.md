## 功能需求及其背景

**背景**：`vChewing-macOS` 與 `vChewing-OSX-Legacy` 自 4.1.4 版開始收緊了自身的 Sandbox 策略，不再盲目地將 home 資料夾塞入 `com.apple.security.temporary-exception.files.home-relative-path.read-write` 存取權能例外清單。這帶來的副作用就是自己無權卸除自己了：唯音輸入法的安裝位置固定於 `~/Library/Input Methods/vChewing` 資料夾。除非把 `~/Library/Input Methods/` 整個塞入存取權能例外清單，否則該輸入法的自我卸除功能將不會生效。但這樣一來就又多了「這個輸入法可能會擅自存取其他輸入法的 Bundle」的嫌疑。

**問題解決思路**：用一個 non-modal 的 NSOpenPanel 視窗向使用者請求對自身 bundle 所在目錄的存取權限。該視窗的觸發契機至少有「輸入法的 CLI 卸除命令」與「輸入法選單當中的卸除命令」。注意還需要額外實裝對於「NSOpenPanel 沒能成功獲取權限，或使用者取消了 NSOpenPanel」情形時的提示文字，且該提示需要誘導使用者前往 `https://github.com/vChewing/vChewing-macOS/wiki/HOW_TO_UNINSTALL` 。您或可設法設計更優的 UX 體驗。

### 技術決策

1. **`CandidatePairWeighted.weight` 只是語言模型 metadata，不是 explicit override score**：`overrideCandidate(_ candidate: CandidatePairWeighted, type: .withSpecified, ...)` 若直接重用 raw LM 權重，會把使用者主動選字退化成「再套一次語言模型排序」，於是出現節點找到卻沒有真正覆寫成功的情況。
2. **Overlap consolidation 必須以真實 node range 為錨**：`consolidateCandidateCursorContext(...)` 的掃描位置可能落在多鍵節點中段；若直接把該位置拿來當 `nodeStart`，會把逐鍵覆寫錯落到錯誤的 key 上，導致 rare-case 永遠卡在 `nothingOverriddenAtNode`。
3. **首輪 consolidation 不能無條件，也不能完全省略**：這個 rare-case 的第一步是從 `幾乎 / 氣 / 坑` 切到跨節點候選 `呼氣`。若當前節點尚未 explicit 且目標 span 改變，就必須先做一次 context consolidation；但若後續狀態已經 explicit，再每輪都先 consolidate 反而會過度固定結構。
4. **rare-case 的成功條件不是「沒有 throw」而已**：真正的成功必須同時滿足「assembled sentence 改變」「目標候選被套用到邏輯游標」「該節點鎖成 explicit」。原先 debug log 一直出現 `applied=false, explicit=false, changed=false`，本質上就是覆寫語義根本沒有成立。
